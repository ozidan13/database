🚀 ما هو الـ Migration؟

Database Migration — Version Control for Your Schema

لو عندك تطبيق على Production وأردت تضيف عمود جديد — كيف تطبّق التغيير بشكل آمن على السيرفر؟ الـ Migration هو الجواب: ملف SQL يصف التغيير، يُحفظ في git، ويُطبَّق بترتيب.

⌨️ أوامر Migration الكاملة

Prisma Migrate Commands Reference

migrate dev

npx prisma migrate dev --name add_slug

للـ Development فقط: يقارن schema.prisma بالـ DB الحالية، ينشئ migration SQL، ويطبّقه. يُشغّل الـ seed تلقائياً إذا حدث reset.

migrate deploy

npx prisma migrate deploy

للـ Production فقط: يطبّق الـ migrations المعلّقة فقط — لا يُنشئ migrations جديدة ولا يُعيد تعيين الـ DB. آمن تماماً.

migrate reset

npx prisma migrate reset

للـ Development فقط: يحذف الـ DB ويُعيد تطبيق كل الـ migrations من الصفر ثم يُشغّل الـ seed. خطير في Production!

migrate status

npx prisma migrate status

يعرض حالة كل migration (applied/pending) — مفيد قبل الـ Deploy لمعرفة ما سيتغيّر.

db push

npx prisma db push

يُطبّق الـ schema مباشرة بدون إنشاء migration — للـ Prototyping السريع فقط. لا تستخدمه مع بيانات مهمة.

db pull (introspect)

npx prisma db pull

يقرأ الـ DB الموجودة ويولّد schema.prisma منها — مفيد لـ Legacy DBs.

🌱 Seeding — ملء البيانات الأولية

Seed Scripts — Populating Database with Initial and Test Data

prisma/seed.ts — Complete E-Commerce Seed
import { PrismaClient } from '@prisma/client'; import { faker } from '@faker-js/faker'; import * as bcrypt from 'bcryptjs'; const prisma = new PrismaClient(); async function main() { console.log('🌱 Seeding database...'); // ① Clear existing data (حذف بالترتيب لتجنب FK violations) await prisma.review.deleteMany(); await prisma.orderItem.deleteMany(); await prisma.order.deleteMany(); await prisma.product.deleteMany(); await prisma.customer.deleteMany(); await prisma.category.deleteMany(); // ② Categories const categories = await Promise.all([ prisma.category.create({ data: { name: 'Electronics', slug: 'electronics' } }), prisma.category.create({ data: { name: 'Clothing', slug: 'clothing' } }), prisma.category.create({ data: { name: 'Books', slug: 'books' } }), ]); // ③ Products (10 per category) const products = []; for (const cat of categories) { for (let i = 0; i < 10; i++) { const name = faker.commerce.productName(); const p = await prisma.product.create({ data: { categoryId: cat.id, name, slug: faker.helpers.slugify(name).toLowerCase() + '-' + faker.string.alphanumeric(4), price: Number(faker.commerce.price({ min: 50, max: 5000 })), } }); products.push(p); } } // ④ Customers + Orders const hashedPwd = await bcrypt.hash('password123', 10); for (let i = 0; i < 20; i++) { const customer = await prisma.customer.create({ data: { email: faker.internet.email(), fullName: faker.person.fullName(), passwordHash: hashedPwd, addresses: { create: { line1: faker.location.streetAddress(), city: faker.location.city(), isDefault: true } } } }); // 1-3 orders per customer const orderCount = faker.number.int({ min: 1, max: 3 }); for (let j = 0; j < orderCount; j++) { const items = faker.helpers.arrayElements(products, { min: 1, max: 4 }); const total = items.reduce((s, p) => s + Number(p.price), 0); await prisma.order.create({ data: { customerId: customer.id, totalAmount: total, status: faker.helpers.arrayElement(['PENDING','DELIVERED','SHIPPED']), items: { create: items.map(p => ({ productId: p.id, quantity: faker.number.int({min:1,max:5}), unitPrice: p.price })) } } }); } } console.log('✅ Seed complete!'); console.log(` Categories: ${categories.length}, Products: ${products.length}`); } main().catch(console.error).finally(() => prisma.$disconnect());

🌍 Environment Strategies

Dev, Staging, Production — Different DBs, Different Strategies

📋 استراتيجية البيئات الثلاث

Development: migrate dev + seed — يمكن reset بحرية
Staging: migrate deploy + Seed بيانات Test حقيقية
Production: migrate deploy فقط — لا reset أبداً، backup قبل كل تغيير

.env files + CI/CD Pipeline
# .env (Development) DATABASE_URL="postgresql://user:pass@localhost:5432/ecommerce_dev" # .env.staging DATABASE_URL="postgresql://user:pass@staging-host:5432/ecommerce_staging" # .env.production (نستخدم Secrets Manager) DATABASE_URL="postgresql://user:pass@prod-host:5432/ecommerce_prod" ## CI/CD Pipeline (GitHub Actions example) # deploy.yml - name: Run Migrations (Production Safe) run: npx prisma migrate deploy env: DATABASE_URL: ${{ secrets.DATABASE_URL }} # لا تستخدم migrate dev أو migrate reset في CI/CD Production!
✅ Production Migration Checklist

Before Every Production Migration — Things to Verify

  • ✅خذ backup كامل لقاعدة البيانات قبل كل migration
  • ✅طبّق الـ migration على Staging أولاً وتحقق من النتائج
  • ✅تحقق من prisma migrate status — كل الـ migrations في قائمة التوقع
  • ✅استخدم npx prisma migrate deploy ليس dev
  • ✅تجنب الـ Blocking DDL (مثل NOT NULL بدون DEFAULT على جدول كبير) — فكّر في Multi-Step Migration
  • ✅لا تحذف عموداً مباشرةً — أضف @ignore أولاً ثم احذف في migration لاحق (Backward Compatible)
  • ✅يفضّل تشغيل الـ migration خارج ساعات الذروة

🧠 اختبار الفهم
١. ما الفرق بين prisma migrate dev و prisma migrate deploy؟
لا فرق — كلاهما متطابق
migrate dev: للـ Development — ينشئ migration files ويمكنه reset. migrate deploy: للـ Production — يطبّق migrations معلّقة فقط، لا ينشئ جديدة ولا يعيد تعيين الـ DB.
deploy أبطأ لأنه يُنشئ migrations
٢. ما خطر استخدام prisma db push في Production؟
أبطأ من migrate
db push يُطبّق التغييرات مباشرة بدون حفظ migration file — لا يمكن التراجع ولا التتبع في git. في Production تضيع القدرة على audit التغييرات.
لا يدعم PostgreSQL
٣. لماذا نحذف البيانات بترتيب محدد في الـ Seed Script؟
لتسريع الحذف
لأن الـ Foreign Key Constraints تمنع حذف record مرتبط به records أخرى. يجب حذف الـ children قبل الـ parents (مثل: order_items قبل orders قبل customers).
لأن Prisma يتطلب ذلك
٤. ما أفضل ممارسة قبل كل migration في Production؟
تشغيل migrate reset أولاً
خذ backup كامل + طبّق على Staging أولاً + تحقق من migrate status + استخدم migrate deploy فقط + فضّل ساعات الذروة المنخفضة.
تشغيل migrate dev على Production مباشرة
