One-to-One — Employee and Profile, User and Settings

schema.prisma — One-to-One (Employee ↔ Profile)
model Employee { id Int @id @default(autoincrement()) email String @unique name String profile Profile? // optional — موظف بدون profile مسموح @@map("employee") } model Profile { id Int @id @default(autoincrement()) bio String? avatarUrl String? @map("avatar_url") employeeId Int @unique @map("employee_id") // @unique يضمن 1:1 employee Employee @relation(fields: [employeeId], references: [id], onDelete: Cascade) @@map("profile") }
One-to-One Queries
// Create Employee + Profile together const emp = await prisma.employee.create({ data: { email: 'ahmed@co.com', name: 'أحمد', profile: { create: { bio: 'Senior DB Engineer' } } }, include: { profile: true } }); // Get Employee with Profile (eager loading) const withProfile = await prisma.employee.findUnique({ where: { id: 1 }, include: { profile: true } }); // Update Profile via Employee (nested update) await prisma.employee.update({ where: { id: 1 }, data: { profile: { update: { bio: 'Updated Bio' } } } });

One-to-Many — Customer → Orders, Category → Products

One-to-Many — Customer and Orders
// Schema: Customer (1) → Order (N) — موجود في درس Prisma Intro // ══ Create Customer with Multiple Orders ══ const customer = await prisma.customer.create({ data: { email: 'fatima@shop.com', fullName: 'فاطمة', orders: { create: [ { totalAmount: 1500, status: 'DELIVERED' }, { totalAmount: 2800, status: 'PENDING' }, ] } }, include: { orders: true } }); // ══ connect: ربط order موجود بـ customer ══ await prisma.order.update({ where: { id: 5 }, data: { customer: { connect: { id: 2 } } } }); // ══ Get Customer with Orders + Total Count ══ const cust = await prisma.customer.findUnique({ where: { id: 1 }, include: { orders: { where: { status: 'DELIVERED' }, orderBy: { orderDate: 'desc' }, take: 5, }, _count: { select: { orders: true } } // إجمالي عدد الأوردرات } }); // ══ N+1 Problem — المشكلة الأخطر في الـ ORM ══ // ❌ خطأ: N+1 — يعمل query لكل customer const customers = await prisma.customer.findMany(); for (const c of customers) { const orders = await prisma.order.findMany({ where: { customerId: c.id } }); // يُنفّذ SELECT query لكل customer! إذا 100 customer = 101 query } // ✅ صحيح: Eager Loading — query واحدة فقط const custsWithOrders = await prisma.customer.findMany({ include: { orders: true } // JOIN واحدة — مش 100 query });

Explicit M:N — Product ↔ Tag (مع بيانات إضافية)
// Explicit Join Table — عندما تريد بيانات في العلاقة نفسها model Product { id Int @id @default(autoincrement()) name String tags ProductTag[] } model Tag { id Int @id @default(autoincrement()) name String @unique products ProductTag[] } model ProductTag { productId Int @map("product_id") tagId Int @map("tag_id") addedAt DateTime @default(now()) @map("added_at") // بيانات إضافية في الـ junction product Product @relation(fields: [productId], references: [id]) tag Tag @relation(fields: [tagId], references: [id]) @@id([productId, tagId]) @@map("product_tag") } // ══ Adding Tags to a Product ══ await prisma.product.update({ where: { id: 1 }, data: { tags: { create: [ { tag: { connect: { id: 1 } } }, // connect existing tag { tag: { create: { name: 'new-tag' } } }, // create & connect new tag ] } } }); // ══ Get Product with Tags ══ const product = await prisma.product.findUnique({ where: { id: 1 }, include: { tags: { include: { tag: { select: { name: true } } }, orderBy: { addedAt: 'desc' } } } }); // ══ Implicit M:N (بدون بيانات إضافية) — Prisma يُنشئ الـ Junction تلقائياً ══ model Post { id Int @id categories Category[] // Prisma ينشئ _PostToCategory تلقائياً } model Category { id Int @id posts Post[] }

⚠️ Prisma لا يدعم Recursive Include تلقائياً

لجلب شجرة هرمية بعمق غير محدد، استخدم $queryRaw مع Recursive CTE في SQL — أو اجلب على مستويات محدودة (2-3 مستويات) يدوياً.

✍️ Nested Writes — create, connect, disconnect, delete

All Nested Write Operations in Prisma Relations

All Nested Operations in One Place
// ══ create: ينشئ related record جديد ══ await prisma.customer.update({ where: { id: 1 }, data: { addresses: { create: { line1: 'شارع جديد', city: 'Alex' } } } }); // ══ connect: يربط record موجود ══ await prisma.product.update({ where: { id: 1 }, data: { category: { connect: { id: 3 } } } }); // ══ disconnect: يفك الرابط (مش يحذف) ══ await prisma.customer.update({ where: { id: 1 }, data: { orders: { disconnect: { id: 5 } } } // يجعل order.customerId = null (يجب أن يكون nullable) }); // ══ update: يُعدّل related record ══ await prisma.customer.update({ where: { id: 1 }, data: { profile: { update: { bio: 'New bio' } } } }); // ══ upsert: ينشئ أو يُعدّل الـ relation ══ await prisma.employee.update({ where: { id: 1 }, data: { profile: { upsert: { create: { bio: 'New Profile' }, update: { bio: 'Updated Profile' } } } } }); // ══ deleteMany: حذف كل related records ══ await prisma.customer.update({ where: { id: 1 }, data: { addresses: { deleteMany: { city: 'Alex' } } } }); // ══ set: يستبدل الـ M:N relationships (Implicit) ══ await prisma.post.update({ where: { id: 1 }, data: { categories: { set: [{ id: 1 }, { id: 3 }] } } // يمسح كل القديمة ويضع الجديدة فقط });
🧠 اختبار الفهم
١. ما الفرق بين connect و create في Nested Writes؟
لا فرق
create: ينشئ record جديد ويربطه. connect: يربط record موجود مسبقاً في قاعدة البيانات. استخدم connect لو الـ record موجود.
connect أسرع دائماً
٢. ما الـ N+1 Problem ولماذا هو خطير؟
مشكلة في الـ Schema
جلب قائمة ثم لكل عنصر نُنفّذ query منفردة للعلاقة — 100 customer = 101 query. الحل: العواد Eager Loading (include) في query واحدة.
تحدث فقط في Many-to-Many
٣. ما الفرق بين Implicit وExplicit Many-to-Many في Prisma؟
Implicit أكثر قوة دائماً
Implicit: Prisma يُنشئ Junction Table تلقائياً — لكن لا تستطيع إضافة بيانات للعلاقة. Explicit: تُعرّف Junction Model بنفسك — تدعم بيانات إضافية (addedAt, role...).
Explicit يحتاج SQL مباشر
٤. ما وظيفة set في Nested Writes؟
مثل connect لكن أسرع
set: يستبدل كل العلاقات الحالية بالقائمة الجديدة فقط — يمسح القديمة ويضع الجديدة. مثالي لـ M:N Implicit. مختلف عن connect الذي يُضيف فقط.
set يضيف للقائمة الموجودة
