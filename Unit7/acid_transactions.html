⚛️ Atomicity — كل أو لا شيء

All-or-Nothing — BEGIN, COMMIT, ROLLBACK

كلمة "Atom" (ذرة) بالمعنى القديم = لا يمكن تقسيمها. Transaction الـ Atomic لا يمكن أن تكون "نصف مكتملة" — إما تكتمل بالكامل أو لا يحدث شيء.

PostgreSQL — Bank Transfer Transaction
-- ══ الطريقة الصحيحة: Transaction كاملة ══ BEGIN; -- الخطوة 1: اخصم من المُرسِل UPDATE accounts SET balance = balance - 1000 WHERE account_id = 1 -- حساب المُرسِل AND balance >= 1000; -- تحقق من الرصيد الكافي -- تحقق من نجاح الخصم (rowCount = 1 تعني نجاح) DO $$ DECLARE affected_rows INT; BEGIN GET DIAGNOSTICS affected_rows = ROW_COUNT; IF affected_rows = 0 THEN RAISE EXCEPTION 'Insufficient balance or account not found'; END IF; END $$; -- الخطوة 2: أضف للمُستقبِل UPDATE accounts SET balance = balance + 1000 WHERE account_id = 2; -- حساب المُستقبِل -- الخطوة 3: سجّل العملية في سجل التدقيق INSERT INTO transaction_log (from_account, to_account, amount, status) VALUES (1, 2, 1000, 'completed'); -- كل الخطوات نجحت → COMMIT COMMIT; -- ══ إذا حدث خطأ في أي خطوة → ROLLBACK تلقائي بالـ EXCEPTION ══ -- أو يدوياً: ROLLBACK;
💡 Transaction في PostgreSQL — الضمانات

إذا حدث خطأ بعد BEGIN وقبل COMMIT، PostgreSQL يُنفّذ ROLLBACK تلقائياً عند انتهاء الاتصال — حتى لو انقطع الكهرباء مع فترة وجيزة بعد COMMIT، الـ WAL يضمن اكتمال الكتابة.

📏 Consistency — القواعد لا تُكسر

Consistency — Constraints, Triggers, and Business Rules Always Hold

Consistency تعني أن قاعدة البيانات تنتقل من حالة صحيحة إلى حالة صحيحة أخرى — لا يمكن لـ Transaction أن تكسر أي Constraint أو Business Rule.

Consistency — Constraints Enforcement PostgreSQL
-- تعريف الـ Constraints التي تضمن الـ Consistency CREATE TABLE accounts ( account_id BIGSERIAL PRIMARY KEY, owner_name TEXT NOT NULL, balance NUMERIC(15,2) NOT NULL DEFAULT 0 CHECK (balance >= 0), -- لا رصيد سالب! account_type TEXT CHECK (account_type IN ('savings','checking','investment')) ); -- هذا ROLLBACK تلقائياً لأن balance سيصبح سالباً (CHECK constraint) BEGIN; UPDATE accounts SET balance = balance - 9999999 WHERE account_id = 1; -- ERROR: new row for relation "accounts" violates check constraint "accounts_balance_check" ROLLBACK; -- أو يحدث تلقائياً -- Trigger لـ Consistency إضافية (مثلاً: منع التحويل من حساب مجمّد) CREATE OR REPLACE FUNCTION check_transfer_allowed() RETURNS TRIGGER AS $$ BEGIN IF EXISTS (SELECT 1 FROM accounts WHERE account_id = OLD.account_id AND is_frozen = true) THEN RAISE EXCEPTION 'Account % is frozen', OLD.account_id; END IF; RETURN NEW; END; $$ LANGUAGE plpgsql;
🏝️ Isolation — العزل بين المعاملات

Isolation — Concurrent Transactions See Consistent Snapshots

تخيّل 1000 مستخدم يحجزون نفس مقعد الطيارة في نفس اللحظة. كيف تمنع أن يحصل اثنان منهم على نفس المقعد؟ الـ Isolation هو الجواب.

Isolation — Seat Booking با SELECT FOR UPDATE
-- المشكلة: بدون Isolation -- Lost Update -- T1: SELECT balance → 500 -- T2: SELECT balance → 500 -- T1: UPDATE balance = 500 - 200 = 300 -- T2: UPDATE balance = 500 - 300 = 200 ← يتجاهل خصم T1! -- النتيجة: 200 بدلاً من 0 — نقود من الهواء! -- ✅ الحل: SELECT FOR UPDATE (Pessimistic Locking) BEGIN; SELECT seat_number, is_available FROM flight_seats WHERE seat_number = '12A' AND flight_id = 100 FOR UPDATE; -- يُقفّل الصف — T2 ستنتظر حتى COMMIT/ROLLBACK -- لو الكرسي متاح: UPDATE flight_seats SET is_available = false, passenger_id = 42 WHERE seat_number = '12A' AND flight_id = 100; INSERT INTO bookings (passenger_id, seat_number, flight_id) VALUES (42, '12A', 100); COMMIT; -- T2 تُفرَج وتجد is_available = false → تُخفق أو تختار مقعدًا آخر -- ✅ Optimistic Locking — لـ Low Contention Scenarios UPDATE products SET stock = stock - 1, version = version + 1 WHERE product_id = 1 AND version = 5 -- يفشل لو T2 غيّرت الـ version AND stock > 0; -- إذا rowCount = 0 → conflict → أعد المحاولة
🪨 Durability — البقاء بعد الكارثة

Durability — WAL Ensures Data Survives Crashes

بعد COMMIT، البيانات يجب أن تبقى حتى لو انقطع الكهرباء فوراً بعدها. PostgreSQL يحقق هذا بالـ Write-Ahead Log (WAL).

① COMMIT يُنفَّذ

PostgreSQL يكتب التغيير في الـ WAL على الـ Disk أولاً قبل أي شيء آخر.

② WAL فقط يضمن الـ Durability

حتى لو البيانات لم تُكتب بعد في الـ Data Pages (في الـ shared_buffers)، الـ WAL كافٍ للاسترداد.

③ Crash Recovery

عند إعادة التشغيل بعد crash، PostgreSQL يقرأ الـ WAL ويُطبّق كل الـ Committed transactions تلقائياً.

④ Checkpoint

بشكل دوري (كل 5 دقائق افتراضياً)، PostgreSQL يكتب الـ Buffer Pool على الـ Disk — ينظّف الـ WAL القديم.

📌 SAVEPOINT — نقاط الحفظ

SAVEPOINT — Partial Rollback within a Transaction

SAVEPOINT يُتيح لك التراجع جزئياً داخل نفس Transaction بدون إلغاء كل شيء — مفيد في العمليات الطويلة.

SAVEPOINT — Partial Rollback Pattern
BEGIN; -- الخطوة 1: إنشاء الطلب INSERT INTO orders (customer_id, total) VALUES (1, 5000); SAVEPOINT after_order; -- حفظ نقطة هنا -- الخطوة 2: محاولة تطبيق الكوبون UPDATE coupons SET usage_count = usage_count + 1 WHERE code = 'SUMMER20' AND usage_count < usage_limit AND expires_at > NOW(); -- إذا الكوبون غير صالح → تراجع جزئي فقط عن الكوبون DO $$ BEGIN IF (SELECT ROW_COUNT()) = 0 THEN ROLLBACK TO SAVEPOINT after_order; -- يُلغي فقط تحديث الكوبون -- يواصل: الطلب لا يزال موجوداً END IF; END $$; -- الخطوة 3: خصم المخزون UPDATE inventory SET quantity_in_stock = quantity_in_stock - 1 WHERE product_id = 101; RELEASE SAVEPOINT after_order; -- حذف الـ savepoint (اختياري) COMMIT;
✍️ تمرين عملي

🏦 تمرين: Banking System Transactions

  • Aاكتب Transaction لتحويل مبلغ بين حسابين مع التحقق من: الرصيد الكافي، وجود الحسابين، عدم تجاوز الحد اليومي.
  • Bأي الخصائص الأربعة تضمن أن رصيد الحساب لا يصبح سالباً؟ فسّر.
  • Cمتى تستخدم SAVEPOINT؟ اذكر مثالاً من الواقع.

B: Consistency — الـ CHECK constraint هو جزء من تعريف الـ Consistency. يمنع أي Transaction من ترك قاعدة البيانات في حالة تنتهك القواعد المعرّفة (مثل balance ≥ 0). كذلك Atomicity تدعم هذا — إذا كسرت Constraint، الـ Transaction كلها تُلغى.

C: مثال عملي — في نظام E-Commerce: البدء بـ Transaction لإتمام الطلب، ثم SAVEPOINT. نحاول تطبيق الكوبون — إذا انتهت صلاحيته، ROLLBACK TO SAVEPOINT ونكمل بدون خصم. هكذا لا نُلغي الطلب كله بسبب كوبون خاطئ.
