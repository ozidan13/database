Lock Types — Row Locks, Table Locks, Advisory Locks

Row-Level Locks

الأكثر شيوعاً. تنشأ تلقائياً مع UPDATE/DELETE/SELECT FOR UPDATE. تُقفّل صفاً واحداً دون التأثير على باقي الصفوف.

Table-Level Locks

تنشأ مع DDL (ALTER TABLE, TRUNCATE) أو LOCK TABLE. ثقيلة — تمنع كل العمليات. تجنّبها في ساعات الذروة.

Advisory Locks

تتحكم فيها أنت بشكل صريح. مثالية لـ Application-Level locking (مثل منع تشغيل job مرتين).

Page Locks

داخلية في PostgreSQL — تُقفّل صفحة الـ Storage أثناء عمليات Sequential Scan. نادراً ما تؤثر على الـ Application.

Lock Modes — من الأضعف للأقوى
-- ══ Row Locks ══ SELECT ... FOR SHARE; -- Shared: يسمح FOR SHARE آخر لكنه يمنع FOR UPDATE SELECT ... FOR NO KEY UPDATE; -- يسمح FOR SHARE & FOR KEY SHARE SELECT ... FOR UPDATE; -- Exclusive: يمنع كل الـ Locks الأخرى SELECT ... FOR KEY SHARE; -- أضعف — يُستخدم للـ FK checks -- NOWAIT: لا تنتظر — أُخفق فوراً إذا كان هناك Lock SELECT * FROM accounts WHERE id = 1 FOR UPDATE NOWAIT; -- ERROR: could not obtain lock on row in relation "accounts" -- SKIP LOCKED: تجاهل الصفوف المُقفّلة (مثالي للـ Job Queues) SELECT * FROM job_queue WHERE status = 'pending' LIMIT 10 FOR UPDATE SKIP LOCKED; -- Worker يأخذ 10 jobs بدون انتظار -- ══ Table Locks ══ LOCK TABLE orders IN SHARE ROW EXCLUSIVE MODE; LOCK TABLE orders IN EXCLUSIVE MODE; -- يمنع كل القراءات والكتابات -- ══ Advisory Locks ══ SELECT pg_advisory_xact_lock(1234); -- يُحرَّر مع Transaction SELECT pg_advisory_lock(1234); -- يُحرَّر يدوياً SELECT pg_advisory_unlock(1234); -- تحرير يدوي SELECT pg_try_advisory_lock(1234); -- يُرجع false إذا مُقفّل (لا ينتظر)
💀 Deadlock — الطريق المسدود

Deadlock — When Two Transactions Wait for Each Other Forever

Deadlock يحدث عندما كل Transaction تنتظر الـ Lock الذي تُمسكه Transaction أخرى — دوامة لا تنتهي. PostgreSQL يكتشفها ويُلغي واحدة.

سيناريو Deadlock

T1 يُقفّل
→ Account A ←
T2 تنتظر
T1 تنتظر
→ Account B ←
T2 يُقفّل

PostgreSQL يكتشف الدائرة → يُلغي T1 → T2 تكمل → T1 يُعيد المحاولة

Deadlock — Cause and Prevention
-- ❌ سيناريو Deadlock (T1 وT2 معاً) -- T1: BEGIN; UPDATE accounts SET balance = balance - 100 WHERE id = 1; -- يُقفّل Row 1 -- T2: BEGIN; UPDATE accounts SET balance = balance - 50 WHERE id = 2; -- يُقفّل Row 2 -- T1 يحاول قفل Row 2 → ينتظر T2 -- T2 يحاول قفل Row 1 → ينتظر T1 → DEADLOCK! -- ✅ الحل: دائماً أُقفّل الصفوف بنفس الترتيب -- T1 وT2 كلاهما يُقفّل Row 1 أولاً ثم Row 2 BEGIN; SELECT * FROM accounts WHERE id IN (1, 2) ORDER BY id -- ORDER BY يضمن الترتيب الثابت! FOR UPDATE; -- ✅ Deadlock Detection Settings SHOW deadlock_timeout; -- الافتراضي: 1s (PostgreSQL ينتظر ثانية قبل الكشف) -- إذا حدث Deadlock → ERROR 40P01 في الـ Application → يجب Retry -- ✅ تقليل Deadlock: استخدم Transactions قصيرة وسريعة -- لا تضع عمليات بطيئة (HTTP calls, file I/O) داخل Transaction
👁️ مراقبة الـ Locks — pg_locks

Monitoring Active Locks and Blocking Queries

pg_locks — Find and Kill Blocking Queries
-- ══ إيجاد كل الـ Locks الحالية ══ SELECT pid, relation::regclass AS table_name, mode, granted, query_start, state, query FROM pg_locks l JOIN pg_stat_activity a USING (pid) WHERE relation IS NOT NULL ORDER BY query_start; -- ══ إيجاد الـ Blocking Queries (مَن يمنع مَن) ══ SELECT blocked_locks.pid AS blocked_pid, blocked_activity.query AS blocked_query, blocking_locks.pid AS blocking_pid, blocking_activity.query AS blocking_query, blocked_activity.wait_event_type, blocked_activity.wait_event FROM pg_catalog.pg_locks blocked_locks JOIN pg_catalog.pg_stat_activity blocked_activity ON blocked_activity.pid = blocked_locks.pid JOIN pg_catalog.pg_locks blocking_locks ON blocking_locks.locktype = blocked_locks.locktype AND blocking_locks.relation = blocked_locks.relation AND blocking_locks.pid != blocked_locks.pid JOIN pg_catalog.pg_stat_activity blocking_activity ON blocking_activity.pid = blocking_locks.pid WHERE NOT blocked_locks.granted; -- ══ إنهاء Query تعيق العمل ══ SELECT pg_cancel_backend(12345); -- يُرسل SIGINT → يُلغي Query فقط SELECT pg_terminate_backend(12345); -- يُنهي الـ Connection كاملاً -- ══ lock_timeout: لا تنتظر أكثر من N ثانية ══ SET lock_timeout = '5s'; -- إذا لم يُحصل على الـ Lock في 5 ثوانٍ → ERROR: canceling statement due to lock timeout
🔄 Recovery — الاسترداد بعد الكارثة

WAL-Based Crash Recovery and Point-in-Time Recovery (PITR)

🛡️ Crash Recovery — يعمل تلقائياً

عند إعادة تشغيل PostgreSQL بعد Crash، يقرأ الـ WAL (Write-Ahead Log) تلقائياً ويُطبق كل الـ Committed transactions. لا حاجة لتدخل يدوي. هذا يضمن الـ Durability من ACID.

Backup & PITR — Point-in-Time Recovery
-- ══ pg_dump: منطقي (الأسرع للـ Small-Medium DBs) ══ # Backup pg_dump -U postgres -d ecommerce_db -F c -f backup.dump # Restore pg_restore -U postgres -d ecommerce_db_new backup.dump # جميع الـ Databases pg_dumpall -U postgres > all_databases.sql -- ══ pg_basebackup: فيزيائي (للـ PITR) ══ pg_basebackup -D /backup/base -Ft -z -P -U postgres # -D: destination, -Ft: tar format, -z: gzip, -P: progress -- ══ PostgreSQL PITR Setup (postgresql.conf) ══ wal_level = replica # enable WAL archiving archive_mode = on archive_command = 'cp %p /wal_archive/%f' -- أو S3 -- ══ Recovery (recovery.conf أو postgresql.conf في PG v12+) ══ restore_command = 'cp /wal_archive/%f %p' recovery_target_time = '2024-03-15 14:30:00' -- PITR: استعادة قبل الحادثة recovery_target_action = 'promote' -- تصبح Primary بعد الاسترداد -- ══ Continuous Archiving Check ══ SELECT pg_switch_wal(); -- يُجبر archiving لملف WAL الحالي SELECT * FROM pg_stat_archiver; -- حالة الـ archiver وآخر ملف مُرشَفَ

🎯 RPO وRTO — المقياسان الأساسيان

RPO (Recovery Point Objective): ما أكبر قدر من البيانات يمكنك تحمّل خسارته؟ (مثلاً: 5 دقائق من البيانات)
RTO (Recovery Time Objective): ما أطول وقت مقبول للتعطل قبل الاستعادة؟ (مثلاً: 30 دقيقة)
كلما كانت RPO وRTO أقل, احتجت استثماراً أكبر في البنية التحتية.

🧠 اختبار الفهم
١. ما الفرق بين SELECT FOR UPDATE وSELECT FOR UPDATE SKIP LOCKED؟
لا فرق
FOR UPDATE: ينتظر حتى يُحرَّر الـ Lock. FOR UPDATE SKIP LOCKED: يتخطى الصفوف المُقفّلة ويأخذ فقط المتاحة — مثالي للـ Job Queues لمنع الـ Workers من الانتظار.
SKIP LOCKED أبطأ
٢. كيف تمنع Deadlock في عمليات التحويل المصرفي؟
استخدام SERIALIZABLE
قفل الصفوف بنفس الترتيب دائماً (SELECT FOR UPDATE ORDER BY account_id) — إذا كانت T1 وT2 يُقفّلان نفس الصفوف بنفس الترتيب، لا دوامة انتظار ممكنة.
استخدام Advisory Locks
٣. ما الفرق بين pg_cancel_backend وpg_terminate_backend؟
لا فرق
pg_cancel_backend: يُرسل SIGINT — يُلغي الـ Query الحالية لكن الـ Connection يبقى مفتوحاً. pg_terminate_backend: يُرسل SIGTERM — يُنهي الـ Connection كاملاً. استخدم cancel أولاً.
cancel يُنهي الـ DB
٤. ما الفرق بين RPO وRTO في استراتيجية النسخ الاحتياطي؟
مترادفان
RPO (Recovery Point Objective): أقصى قدر مقبول من البيانات المُفقودة (Time كم). RTO (Recovery Time Objective): أقصى وقت مقبول للتعطل قبل الاستعادة. كلاهما يحدد مستوى الاستثمار في البنية التحتية.
RTO هو RPO بالثواني
