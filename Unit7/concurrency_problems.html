🦠 Dirty Read — قراءة البيانات الملوّثة

Dirty Read — Reading Uncommitted Data from Another Transaction

Dirty Read
Severity: خطير

T1 تقرأ بيانات كتبتها T2 لكن T2 لم تُعتمد بعد (لم تُنفّذ COMMIT). إذا أجرت T2 ROLLBACK، T1 بنت قراراتها على بيانات وهمية!

الوقتT1 (Report)T2 (Transfer)
t1BEGINBEGIN
t2UPDATE balance = 5000 WHERE id=1 (لم COMMIT بعد)
t3SELECT balance → 5000 ← Dirty!
t4ROLLBACK (العملية فشلت)
t5يبني تقريره على 5000 ← كاذب!
الحل: READ COMMITTED (الافتراضي في PostgreSQL)
-- PostgreSQL الافتراضي = READ COMMITTED → لا Dirty Reads أبداً -- رؤية T1 دائماً للبيانات المُعتمدة فقط -- تغيير الـ Isolation Level لـ Transaction محددة: BEGIN; SET TRANSACTION ISOLATION LEVEL READ COMMITTED; -- أو إذا حدث Dirty Read في نظام آخر (MySQL مثلاً بـ READ UNCOMMITTED): SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED; -- يسمح Dirty Reads! -- Good news: PostgreSQL لا يدعم READ UNCOMMITTED فعلياً -- حتى لو حددته، يتصرف كـ READ COMMITTED
🔄 Non-Repeatable Read — القراءة غير المتسقة

Non-Repeatable Read — Same Query, Different Results in Same Transaction

Non-Repeatable Read
Severity: متوسط

T1 تقرأ صفاً، ثم T2 تُعدّله وتُعتمد، ثم T1 تقرأ نفس الصف مرة أخرى — تجد قيمة مختلفة! التقرير لم يأتِ من "لحظة واحدة متسقة".

الوقتT1 (Audit Report)T2 (Price Update)
t1BEGIN
t2SELECT price WHERE id=1 → 100 LE
t3UPDATE price = 150 WHERE id=1; COMMIT;
t4SELECT price WHERE id=1 → 150 LE ← مختلف!
الحل: REPEATABLE READ في PostgreSQL
BEGIN; SET TRANSACTION ISOLATION LEVEL REPEATABLE READ; -- الآن: T1 ترى Snapshot ثابت من لحظة BEGIN -- حتى لو T2 عدّلت وCOMMIT، T1 لن ترى التغيير SELECT price FROM products WHERE id = 1; -- → 100 -- [T2 تُعدّل price=150 وتُعتمد] SELECT price FROM products WHERE id = 1; -- → 100 (نفس الـ snapshot) COMMIT; -- الآن يرى T1 الـ value المحدّثة إذا قرأ مرة أخرى -- PostgreSQL يُحقق هذا بـ MVCC (Multi-Version Concurrency Control) -- يحتفظ بـ snapshots متعددة للبيانات — لا locking على القراءة
👻 Phantom Read — الصفوف الشبحية

Phantom Read — New Rows Appear Within the Same Transaction

Phantom Read
Severity: متوسط-عالي

T1 تُنفّذ نفس الـ Query مرتين — في المرة الثانية تجد صفوفاً جديدة (أضافتها T2). المشكلة ليست في صف موجود بل في صفوف إضافية ظهرت!

الوقتT1 (Capacity Check)T2 (New Booking)
t1SELECT COUNT(*) WHERE flight=100 → 149 (متاح مقعد واحد)
t2INSERT booking (flight=100); COMMIT;
t3يحجز المقعد الأخير
t4SELECT COUNT(*) WHERE flight=100 → 151! ← Phantom!
الحل: SERIALIZABLE أو SELECT FOR UPDATE مع Range Lock
-- الحل 1: SERIALIZABLE Isolation Level BEGIN; SET TRANSACTION ISOLATION LEVEL SERIALIZABLE; SELECT COUNT(*) FROM bookings WHERE flight_id = 100; -- إذا T2 أضافت booking جديد بعد هذا → Serialization Failure عند COMMIT -- PostgreSQL يُعيد ERROR: could not serialize access due to concurrent update -- يجب إعادة المحاولة (retry) -- الحل 2: Advisory Lock للـ Flight Capacity SELECT pg_advisory_xact_lock(100); -- يُقفّل الـ flight_id=100 SELECT COUNT(*) FROM bookings WHERE flight_id = 100; -- الآن T2 ستنتظر حتى يُفرَج القفل
💨 Lost Update — التعديل المفقود

Lost Update — One Transaction's Write Overwrites Another's

Lost Update
Severity: خطير جداً

T1 وT2 يقرآن نفس الرصيد، كل منهما يُعدّله، لكن T2 تكتب فوق تعديل T1 — التعديل الأول يُفقد تماماً!

الوقتT1 (User A — يضيف 500)T2 (User B — يضيف 1000)
t1SELECT stock → 10
t2SELECT stock → 10
t3UPDATE stock = 10 + 5 = 15
t4UPDATE stock = 10 + 3 = 13 ← يمسح تعديل T1!
t5COMMITCOMMIT → النتيجة 13 بدلاً من 18!
الحلول: Atomic UPDATE و SELECT FOR UPDATE
-- ❌ الطريقة الخاطئة (Read-Modify-Write في الـ Application) -- stock = SELECT stock → تعديل في الكود → UPDATE stock = new_value -- ✅ الحل 1: Atomic UPDATE — لا قراءة في الـ Application UPDATE inventory SET quantity_in_stock = quantity_in_stock + 5 WHERE product_id = 101; -- PostgreSQL ينفّذ هذا كـ Atomic operation بدون Lost Update -- ✅ الحل 2: SELECT FOR UPDATE (Pessimistic Lock) BEGIN; SELECT quantity_in_stock FROM inventory WHERE product_id = 101 FOR UPDATE; -- يُقفّل الصف — T2 تنتظر -- الآن آمن للـ Read-Modify-Write UPDATE inventory SET quantity_in_stock = quantity_in_stock - 3 WHERE product_id = 101; COMMIT; -- T2 تُفرَج وترى القيمة المُحدَّثة -- ✅ الحل 3: Optimistic Locking (version column) UPDATE inventory SET quantity_in_stock = quantity_in_stock - 3, version = version + 1 WHERE product_id = 101 AND version = 7; -- يفشل إذا T2 غيّر version قبلنا → rowCount=0 → retry
🌍 Write Skew — مشكلة أعمق

Write Skew = نوع متقدم من Lost Update. مثال: مستشفى — قاعدة "يجب أن يكون هناك طبيب على الأقل في كل وقت". T1 تقرأ: 2 أطباء موجودون → الطبيب أ يأخذ إجازة. T2 تقرأ: 2 موجودون → الطبيب ب يأخذ إجازة. النتيجة: 0 أطباء! الحل: SERIALIZABLE Isolation.
