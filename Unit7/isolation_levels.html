Four Isolation Levels — Lowest to Highest Protection

① READ UNCOMMITTED — لا يُستخدم عملياً

يسمح بقراءة بيانات لم تُعتمد بعد. في PostgreSQL لا يُطبَّق فعلياً — يتصرف كـ READ COMMITTED.

✗ Dirty Read ✗ Non-Rep ✗ Phantom ← ممكن

② READ COMMITTED — الافتراضي في PostgreSQL

كل Statement ترى Snapshot من بداية تنفيذها (ليس من بداية الـ Transaction). يمنع Dirty Read. الأداء الجيد مع الحماية الأساسية.

✓ Dirty Read محمي ✗ Non-Rep ممكن ✗ Phantom ممكن

③ REPEATABLE READ — Snapshot من بداية الـ Transaction

كل الـ Queries في الـ Transaction ترى نفس الـ Snapshot من لحظة BEGIN. يمنع Dirty وNon-Repeatable. PostgreSQL يمنع Phantoms أيضاً بسبب MVCC.

✓ Dirty محمي ✓ Non-Rep محمي ✓ Phantom محمي (PostgreSQL)

④ SERIALIZABLE — التنفيذ المتسلسل الكامل

يضمن أن النتيجة هي كما لو كانت كل الـ Transactions تنفّذت واحدة تلو الأخرى. أعلى حماية، أعلى صعوبة للأداء. يستخدم SSI (Serializable Snapshot Isolation).

✓ كل الأنواع محمية ✓ Write Skew محمي ← قد يُرجع Serialization Error

PostgreSQL — Setting Isolation Levels
-- لـ Transaction محددة (الأشيع) BEGIN; SET TRANSACTION ISOLATION LEVEL REPEATABLE READ; -- ... queries ... COMMIT; -- مختصر (في نفس السطر) BEGIN ISOLATION LEVEL SERIALIZABLE; -- لـ Session كاملة SET default_transaction_isolation = 'repeatable read'; -- لـ Database كاملة (يحتاج ALTER DATABASE) ALTER DATABASE ecommerce_db SET default_transaction_isolation = 'read committed'; -- الاستعلام عن الـ Level الحالي SHOW transaction_isolation; -- → read committed

MVCC — Multi-Version Concurrency Control — No Read Locks

سرّ قوة PostgreSQL في الـ Concurrency هو MVCC (Multi-Version Concurrency Control) — بدلاً من قفل الصفوف عند القراءة، يحتفظ بنسخ متعددة من كل صف.

💡 فكرة MVCC

كل صف في PostgreSQL لديه:
• xmin: الـ Transaction ID التي أنشأته
• xmax: الـ Transaction ID التي حذفته (أو 0 لو لا يزال حياً)
عند القراءة، PostgreSQL يُفلتر الصفوف المرئية بناءً على الـ Snapshot — بدون locks على القراءة!

MVCC — رؤية الـ Transaction Metadata
-- رؤية الـ MVCC metadata SELECT product_id, name, price, xmin, -- Transaction ID التي أنشأت هذه النسخة xmax, -- Transaction ID التي حذفتها (0 = لا يزال حياً) ctid -- Physical location (page, row) FROM product; -- MVCC يعني: Writers لا يُحجبون Readers والعكس -- T1 تقرأ بيانات قديمة (Old Snapshot) -- T2 تكتب نسخة جديدة من نفس الصف -- كلاهما لا يتعطل بسبب الآخر! -- VACUUM: ينظّف النسخ القديمة (Dead Tuples) VACUUM product; -- تنظيف يدوي VACUUM ANALYZE product; -- تنظيف + تحديث الإحصائيات VACUUM FULL product; -- إعادة كتابة كاملة (يُقفّل الجدول!) -- Autovacuum يعمل تلقائياً في الـ Background -- يتحكم فيه: autovacuum_vacuum_threshold, autovacuum_vacuum_scale_factor

Decision Guide — Which Isolation Level for Your Use Case

🏦 READ COMMITTED — الاختيار الافتراضي (90% من الحالات)

✅ رؤية الإعلانات والمنتجات
✅ قراءة قائمة الطلبات
✅ عمليات CRUD اليومية في E-Commerce
✅ API endpoints عامة
لأن كل Statement ترى أحدث بيانات مُعتمدة — مقبول لمعظم Reads.

📊 REPEATABLE READ — للتقارير والإحصائيات

✅ إنشاء تقرير مالي شهري
✅ حساب إحصائيات المبيعات لفترة محددة
✅ Export كبير للبيانات
لأن التقرير يريد لحظة متسقة واحدة — لا يريد أن يرى بيانات تتغيّر أثناء معالجته.

🔐 SERIALIZABLE — للعمليات الحساسة

✅ تحويلات مالية مع قواعد معقدة
✅ حجز المقاعد في الطيران (Overbooking Prevention)
✅ أنظمة الرعاية الصحية (لا يجوز خطأ!)
✅ عندما تخشى الـ Write Skew
⚠️ تذكّر: يمكن أن يُرجع Serialization Failure — يجب Retry Logic.

SERIALIZABLE — Retry Pattern
// Node.js — Retry Pattern for SERIALIZABLE Transactions async function withSerializableRetry(fn: () => Promise<any>, maxRetries = 5) { for (let attempt = 1; attempt <= maxRetries; attempt++) { const client = await pool.connect(); try { await client.query('BEGIN ISOLATION LEVEL SERIALIZABLE'); const result = await fn(); await client.query('COMMIT'); return result; } catch (err: any) { await client.query('ROLLBACK'); // Serialization Failure عند إعادة المحاولة if (err.code === '40001' && attempt < maxRetries) { const delay = 50 * Math.pow(2, attempt); // Exponential backoff await new Promise(r => setTimeout(r, delay)); continue; } throw err; } finally { client.release(); } } } // PG Error 40001 = serialization_failure // PG Error 40P01 = deadlock_detected
🧠 اختبار الفهم
١. ما الـ Isolation Level الافتراضي في PostgreSQL وما الذي يمنعه؟
REPEATABLE READ — يمنع Non-Repeatable وPhantom
READ COMMITTED — يمنع Dirty Read فقط. Non-Repeatable وPhantom ممكنان. مثالي لـ 90% من التطبيقات لأوازن بين الأمان والأداء.
SERIALIZABLE — يمنع كل الأنواع
٢. ما MVCC وكيف يختلف عن Traditional Locking؟
MVCC أبطأ من Locking
MVCC يحتفظ بنسخ متعددة من كل صف — القراءة ترى snapshot مناسبة بدون Lock. Writers لا يمنعون Readers والعكس. Traditional Locking: القراءة تُقفّل — Writers يمنعون Readers.
MVCC لا يدعم Isolation
٣. تنفّذ تقريراً مالياً يحسب إجمالي المبيعات لشهر كامل — أي مستوى عزل تختار؟
READ COMMITTED
REPEATABLE READ — يضمن رؤية Snapshot ثابت من بداية التقرير. بدونه، صفقات جديدة مُسجَّلة أثناء معالجة التقرير قد تفسد الأرقام.
READ UNCOMMITTED
٤. ماذا يحدث عند Serialization Failure في PostgreSQL (Error 40001)؟
يُدمر قاعدة البيانات
PostgreSQL يُلغي Transaction واحدة ويُرجع Error 40001 — يجب الـ Application يُعيد المحاولة (Retry). هذا التصميم طبيعي ومتوقع مع SERIALIZABLE.
