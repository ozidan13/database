🤔 متى تـ Denormalize؟

When Normalization Becomes a Performance Bottleneck

الـ Denormalization هو إضافة تكرار مقصود في البيانات لتحسين أداء القراءة — على حساب زيادة تعقيد الكتابة. مش هروب من التصميم السليم — هو قرار هندسي محسوب.

📊 OLTP vs OLAP — النقطة المحورية

OLTP (Online Transaction Processing): كتابة + قراءة متزامنة، transactions صغيرة ومتكررة → التطبيع الكامل مثالي

OLAP (Online Analytical Processing): قراءات تحليلية ضخمة، JOINs معقدة على ملايين صفوف، تقارير دورية → الـ Denormalization يحسّن الأداء جذرياً

القاعدة: طبّع أولاً، ثم Denormalize بناءً على قياسات أداء حقيقية — لا تـ Denormalize "احتياطاً".

✅ متى يكون الـ Denormalization مبرّراً؟

① Query سريعة جداً ولا يمكن تحسينها بـ Index أو Materialized View
② الـ JOIN يجمع أكثر من 5-6 جداول في كل request
③ البيانات نادراً ما تتغيّر (أو التغييرات يمكن إدارتها)
④ الـ Analytics/Reporting هو الاستخدام الأساسي — وليس الـ Transactions
⑤ Data Warehouse / BI Dashboard يحتاج استجابة في ثوانٍ

🛠️ تقنيات الـ Denormalization

Denormalization Techniques — Practical Patterns

① Redundant Columns

نسخ عمود من جدول لجدول آخر لتجنب الـ JOIN — مثالي للأعمدة اللي ما بتتغيّرش كتير.

orders(order_id, customer_name) -- نسخ customer_name من customer -- بدل JOIN في كل مرة

② Pre-aggregation

حفظ نتائج الـ COUNT/SUM الثقيلة في عمود محسوب مسبقاً — يُحدَّث بـ Trigger.

customer(customer_id, orders_count) -- Trigger يزيد orders_count عند INSERT

③ Derived/Computed Columns

تخزين نتيجة العمليات الحسابية الثقيلة (مثل الـ TAX، الـ Discount) مباشرة في الجدول.

order_item(subtotal GENERATED ALWAYS AS (quantity * unit_price) STORED)

④ Flattened/Wide Tables

دمج جداول مرتبطة في جدول واحد عريض للـ Analytics — الـ Data Warehouse pattern الأساسي.

fact_sales(date, category_name, product_name, customer_city, revenue, units_sold)
Denormalization Patterns — SQL Implementation
-- ══ Pattern 1: Redundant Column + Trigger Sync ══ ALTER TABLE orders ADD COLUMN customer_name TEXT; -- ملء البيانات التاريخية UPDATE orders o SET customer_name = (SELECT full_name FROM customer WHERE customer_id = o.customer_id); -- Trigger للتزامن عند التحديث CREATE OR REPLACE FUNCTION sync_customer_name() RETURNS TRIGGER AS $$ BEGIN UPDATE orders SET customer_name = NEW.full_name WHERE customer_id = NEW.customer_id; RETURN NEW; END; $$ LANGUAGE plpgsql; CREATE TRIGGER trig_sync_name AFTER UPDATE OF full_name ON customer FOR EACH ROW EXECUTE FUNCTION sync_customer_name(); -- ══ Pattern 2: Pre-aggregation Counter ══ ALTER TABLE product ADD COLUMN reviews_count INT DEFAULT 0; ALTER TABLE product ADD COLUMN avg_rating NUMERIC(3,2) DEFAULT 0; CREATE OR REPLACE FUNCTION update_product_stats() RETURNS TRIGGER AS $$ BEGIN UPDATE product SET reviews_count = (SELECT COUNT(*) FROM review WHERE product_id = NEW.product_id), avg_rating = (SELECT AVG(rating) FROM review WHERE product_id = NEW.product_id) WHERE product_id = NEW.product_id; RETURN NEW; END; $$ LANGUAGE plpgsql; CREATE TRIGGER trig_review_stats AFTER INSERT OR UPDATE OR DELETE ON review FOR EACH ROW EXECUTE FUNCTION update_product_stats(); -- ══ Pattern 3: GENERATED ALWAYS (PostgreSQL 12+) ══ CREATE TABLE order_item ( item_id BIGSERIAL PRIMARY KEY, order_id BIGINT, product_id BIGINT, quantity INT, unit_price NUMERIC(10,2), subtotal NUMERIC(12,2) GENERATED ALWAYS AS (quantity * unit_price) STORED ); -- subtotal تُحسب تلقائياً — لا تحتاج UPDATE أبداً
⭐ Star Schema — قلب الـ Data Warehouse

Star Schema — Denormalized Design for Business Intelligence

الـ Star Schema هو Pattern أساسي للـ Data Warehousing: جدول Fact مركزي بيحتوي الأرقام والقياسات، وحوله جداول Dimension تحتوي السياق. الجداول مطبّعة جزئياً فقط (أحياناً مش للـ 3NF).

Star Schema — E-Commerce DWH
dim_date
date_id, year, month, quarter, day_name
↓
dim_customer
customer_id, name, city, segment
→
fact_sales
date_id, customer_id, product_id, store_id
revenue, units_sold, discount
←
dim_product
product_id, name, category, brand
↑
dim_store
store_id, city, region, manager
Star Schema Query — BI Report
-- تقرير: إجمالي المبيعات لكل فئة بكل منطقة في 2024 SELECT d.quarter, p.category, s.region, SUM(f.revenue) AS total_revenue, SUM(f.units_sold) AS total_units, COUNT(DISTINCT f.customer_id) AS unique_customers FROM fact_sales f JOIN dim_date d USING (date_id) JOIN dim_product p USING (product_id) JOIN dim_store s USING (store_id) WHERE d.year = 2024 GROUP BY d.quarter, p.category, s.region ORDER BY total_revenue DESC; -- ✅ الـ Dimension tables صغيرة وسريعة القراءة -- ✅ الـ Fact table به column-level indexes -- ✅ Columnar storage (Parquet/Redshift) يعمل بشكل رائع مع هذا النمط
⚖️ الفوائد والتكاليف

Denormalization Trade-offs — Read vs Write Performance

✅ الفوائد

  • قراءات أسرع بكثير — أقل JOINs
  • Queries Analytics بسيطة ومقروءة
  • ملائم للـ Caching والـ BI Tools
  • أفضل مع الـ Columnar Storage
  • أقل CPU في كل Query

⚠️ التكاليف

  • كتابة أبطأ وأعقد (UPDATE أكثر)
  • خطر التناقض إذا لم تُزامَن البيانات جيداً
  • حجم Storage أكبر
  • Triggers وMaintenance overhead
  • أصعب في الصيانة والتطوير
⚠️ القاعدة الذهبية: Measure First — Denormalize Second

لا تـ Denormalize بدون قياس. خطوات صحيحة:
① طبّع أولاً (3NF/BCNF)
② أضف الـ Indexes المناسبة
③ جرّب Materialized Views
④ قيس الأداء بـ EXPLAIN ANALYZE
⑤ فقط إذا لم يكن كافياً → Denormalize بعناية

🏢 أمثلة واقعية من Production Systems

Amazon/Shopify: orders تحتفظ بـ customer_name وaddress وقت الأوردر — لأن بيانات العميل قد تتغيّر لاحقاً
Twitter/Reddit: posts_count وfollowers_count في جدول المستخدم — Trigger يُحدّثهم
Banks: account_balance مخزونة (مش محسوبة من transactions) — Pre-aggregation للأداء والـ Consistency
Google Analytics: Star Schema كامل — Fact table بـ billions of rows، Dimensions صغيرة

🧠 اختبار الفهم
١. ما الـ Pattern المناسب لتخزين اسم العميل في الأوردر حتى لو غيّر اسمه لاحقاً؟
JOIN في كل Query
Redundant Column — نسخ قيمة customer_name وقت الأوردر في جدول orders مباشرة
Materialized View
٢. ما الـ GENERATED ALWAYS AS ... STORED في PostgreSQL؟
عمود يُحسب في كل Query (Virtual)
عمود يُحسب تلقائياً عند INSERT/UPDATE ويُخزَّن على disk — لا تحتاج Trigger لتحديثه
عمود ثابت لا يتغيّر
٣. في Star Schema، ما الفرق بين Fact Table وDimension Table؟
لا فرق — كلاهما نفس الشيء
Fact Table: يحتوي القياسات (revenue, units) والـ Foreign Keys. Dimension Table: يحتوي السياق (name, category, region) وأصغر حجماً.
Fact Table أصغر دايماً
٤. ما الخطوة الصحيحة قبل الـ Denormalization؟
Denormalize أولاً لتجنب المشاكل
طبّع أولاً → أضف Indexes → جرّب Materialized Views → قيس الأداء بـ EXPLAIN ANALYZE → ثم Denormalize إذا لزم
Denormalize بناءً على الخبرة بدون قياس
