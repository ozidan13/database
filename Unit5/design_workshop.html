📋 المتطلبات (Requirements)

✅ العملاء يسجّلون حسابات ولديهم عناوين متعددة
✅ المنتجات تنتمي لفئات ويمكن أن يكون لها عدة صور
✅ العملاء يضعون أوردرات تحتوي عدة منتجات بكميات وأسعار وقت البيع
✅ لكل أوردر حالة (pending/confirmed/shipped/delivered/cancelled)
✅ العملاء يكتبون مراجعات للمنتجات (تقييم + تعليق)
✅ نظام كوبونات خصم (نسبة مئوية أو قيمة ثابتة) مع تواريخ صلاحية
✅ تتبع المخزون (stock) لكل منتج في كل مستودع

💾 DDL الكامل — Production-Ready Schema

Complete DDL — Types, Tables, Constraints, Indexes

E-Commerce — Complete Schema DDL
-- ══ Extensions ══ CREATE EXTENSION IF NOT EXISTS "pgcrypto"; -- gen_random_uuid() -- ══ ENUM Types ══ CREATE TYPE order_status AS ENUM ('pending','confirmed','shipped','delivered','cancelled'); CREATE TYPE discount_type AS ENUM ('percentage', 'fixed'); -- ══ customer ══ CREATE TABLE customer ( customer_id BIGSERIAL PRIMARY KEY, email TEXT NOT NULL UNIQUE, full_name TEXT NOT NULL, phone TEXT, password_hash TEXT NOT NULL, is_active BOOLEAN DEFAULT TRUE, created_at TIMESTAMPTZ DEFAULT NOW(), updated_at TIMESTAMPTZ DEFAULT NOW(), CONSTRAINT ck_email_format CHECK (email ~* '^[^@]+@[^@]+\.[^@]+$') ); -- ══ address ══ CREATE TABLE address ( address_id BIGSERIAL PRIMARY KEY, customer_id BIGINT NOT NULL REFERENCES customer(customer_id) ON DELETE CASCADE, line1 TEXT NOT NULL, line2 TEXT, city TEXT NOT NULL, country TEXT NOT NULL DEFAULT 'Egypt', postal_code TEXT, is_default BOOLEAN DEFAULT FALSE ); -- ══ category (self-referencing للـ subcategories) ══ CREATE TABLE category ( category_id BIGSERIAL PRIMARY KEY, name TEXT NOT NULL UNIQUE, slug TEXT NOT NULL UNIQUE, parent_id BIGINT REFERENCES category(category_id) ON DELETE RESTRICT, description TEXT ); -- ══ product ══ CREATE TABLE product ( product_id BIGSERIAL PRIMARY KEY, category_id BIGINT NOT NULL REFERENCES category(category_id), name TEXT NOT NULL, slug TEXT NOT NULL UNIQUE, description TEXT, price NUMERIC(12,2) NOT NULL CHECK (price >= 0), cost_price NUMERIC(12,2), attributes JSONB DEFAULT '{}', is_active BOOLEAN DEFAULT TRUE, -- Denormalized aggregates (maintained by triggers) reviews_count INT DEFAULT 0, avg_rating NUMERIC(3,2) DEFAULT 0, created_at TIMESTAMPTZ DEFAULT NOW() ); -- ══ coupon ══ CREATE TABLE coupon ( coupon_id BIGSERIAL PRIMARY KEY, code TEXT NOT NULL UNIQUE, discount_type discount_type NOT NULL, discount_value NUMERIC(10,2) NOT NULL CHECK (discount_value > 0), min_order_amount NUMERIC(10,2) DEFAULT 0, usage_limit INT, usage_count INT DEFAULT 0, starts_at TIMESTAMPTZ DEFAULT NOW(), expires_at TIMESTAMPTZ, CONSTRAINT ck_pct_range CHECK ( discount_type != 'percentage' OR discount_value <= 100 ) ); -- ══ orders ══ CREATE TABLE orders ( order_id BIGSERIAL PRIMARY KEY, customer_id BIGINT NOT NULL REFERENCES customer(customer_id), address_id BIGINT REFERENCES address(address_id), coupon_id BIGINT REFERENCES coupon(coupon_id), status order_status NOT NULL DEFAULT 'pending', total_amount NUMERIC(12,2) NOT NULL CHECK (total_amount >= 0), -- Snapshot of customer_name at order time (Denormalization) customer_name TEXT, notes TEXT, order_date TIMESTAMPTZ DEFAULT NOW(), updated_at TIMESTAMPTZ DEFAULT NOW() ); -- ══ order_item ══ CREATE TABLE order_item ( item_id BIGSERIAL PRIMARY KEY, order_id BIGINT NOT NULL REFERENCES orders(order_id) ON DELETE CASCADE, product_id BIGINT NOT NULL REFERENCES product(product_id), quantity INT NOT NULL CHECK (quantity > 0), unit_price NUMERIC(12,2) NOT NULL, -- snapshot at order time discount_pct NUMERIC(5,2) DEFAULT 0, subtotal NUMERIC(12,2) GENERATED ALWAYS AS (quantity * unit_price * (1 - discount_pct/100)) STORED ); -- ══ warehouse & inventory ══ CREATE TABLE warehouse ( warehouse_id BIGSERIAL PRIMARY KEY, name TEXT NOT NULL UNIQUE, city TEXT NOT NULL, is_active BOOLEAN DEFAULT TRUE ); CREATE TABLE inventory ( inventory_id BIGSERIAL PRIMARY KEY, product_id BIGINT NOT NULL REFERENCES product, warehouse_id BIGINT NOT NULL REFERENCES warehouse, quantity_in_stock INT NOT NULL DEFAULT 0 CHECK (quantity_in_stock >= 0), reorder_level INT DEFAULT 10, UNIQUE (product_id, warehouse_id) ); -- ══ review ══ CREATE TABLE review ( review_id BIGSERIAL PRIMARY KEY, product_id BIGINT NOT NULL REFERENCES product, customer_id BIGINT NOT NULL REFERENCES customer, rating SMALLINT NOT NULL CHECK (rating BETWEEN 1 AND 5), body TEXT, is_verified BOOLEAN DEFAULT FALSE, created_at TIMESTAMPTZ DEFAULT NOW(), UNIQUE (product_id, customer_id) -- مراجعة واحدة لكل منتج لكل عميل ); -- ══ Indexes ══ CREATE INDEX idx_orders_customer ON orders (customer_id); CREATE INDEX idx_orders_status ON orders (status); CREATE INDEX idx_orders_date ON orders (order_date DESC); CREATE INDEX idx_product_category ON product (category_id); CREATE INDEX idx_product_attrs ON product USING GIN (attributes); CREATE INDEX idx_order_item_order ON order_item (order_id); CREATE INDEX idx_review_product ON review (product_id);
🚫 Design Anti-Patterns — أكثر الأخطاء شيوعاً

Common Database Design Mistakes and How to Avoid Them

❌ Anti-Pattern 1: God Table

جدول واحد ضخم بكل البيانات — Customers وOrders والمنتجات كلها في جدول entity مع عمود type.
الحل: فصل الـ Entities في جداول مستقلة وربطها بـ Foreign Keys.

❌ Anti-Pattern 2: Magic Numbers والـ Status بـ INT

status = 1 (ماذا تعني 1؟ pending؟ approved؟) — يصعّب القراءة والصيانة.
الحل: ENUM types أو جدول Lookup مستقل بالتوصيف الكامل.

❌ Anti-Pattern 3: Repeating Groups / EAV

جداول مثل attribute(entity_id, attr_name, attr_value) — صعبة الـ Indexing وضعيفة الـ Type Safety.
الحل: JSONB للخصائص المرنة أو جداول Typed منفصلة.

❌ Anti-Pattern 4: بدون Soft Delete

حذف الصفوف مباشرة يكسر الـ History وقد يكسر الـ Foreign Keys.
الحل: إضافة عمود deleted_at وفلترة WHERE deleted_at IS NULL.

❌ Anti-Pattern 5: أسعار بدون snapshot

order_item يحتفظ بـ product_id فقط — لو تغيّر السعر، الأوردر القديم يظهر بسعر خاطئ!
الحل: حفظ unit_price وقت الأوردر في order_item مباشرة.

✅ Checklist التصميم الاحترافي

Professional Database Design Checklist

  • 🔑كل جدول له Primary Key واضح (BIGSERIAL أو UUID)
  • 🔗كل العلاقات مُعرَّفة بـ Foreign Key Constraints مع ON DELETE المناسب
  • ⚡ENUM types لكل الـ Status والحالات المحدودة القيم
  • 📅created_at وupdated_at في كل جدول مهم
  • 🔒Unique Constraints على emails وslugs وcodes
  • ✅CHECK Constraints للقيم المنطقية (salary >= 0, rating BETWEEN 1 AND 5)
  • 🗑️Soft Delete (deleted_at) للجداول الرئيسية
  • 📷Snapshot prices/names في الـ Transaction tables
  • 📇Indexes على الـ Foreign Keys والـ Filter columns المتكررة
  • 📊الوصول للـ 3NF كحد أدنى — مع Denormalization مبرّر بقياس
  • 🔍EXPLAIN ANALYZE لاختبار الـ Queries قبل Production
  • 📝Comments على الـ Tables والأعمدة غير الواضحة

🎯 التحدي النهائي: نظام إدارة مكتبة

المتطلبات

✅ الكتب لها عدة نسخ فيزيائية (copies) وعدة مؤلفين
✅ الأعضاء يستعيرون نسخاً (لكل عضو حد أقصى للاستعارة)
✅ تتبع تاريخ الاستعارة والإعادة والغرامات
✅ فئات الكتب (Fiction, Science...) مع subcategories
✅ نظام حجز (reservation) للكتب غير المتاحة

  • Aارسم Entity Diagram يغطي كل المتطلبات.
  • Bاكتب DDL كامل مع ENUMs، Constraints، وIndexes.
  • Cهل تحتاج أي Denormalization؟ وضّح مع سبب.

A — Entities الرئيسية:

book(book_id, isbn UNIQUE, title, pub_year, category_id, description, search_vector)
author(author_id, name, bio)
book_author(book_id, author_id) — Junction Table (N:N)
book_copy(copy_id, book_id, barcode UNIQUE, condition ENUM, is_available)
category(category_id, name UNIQUE, parent_id?)
member(member_id, email UNIQUE, full_name, max_borrows DEFAULT 3, is_active)
borrow(borrow_id, copy_id, member_id, borrowed_at, due_at, returned_at, fine_amount)
reservation(reservation_id, book_id, member_id, reserved_at, status ENUM)

B — ENUMs المقترحة:
copy_condition: ('new','good','fair','worn')
reservation_status: ('active','fulfilled','cancelled','expired')

C — Denormalization مبرّر:
① book.copies_available INT — عدد النسخ المتاحة (Pre-aggregated، يُحدَّث بـ Trigger عند الاستعارة والإعادة) — لتجنب COUNT query في كل طلب
② book.avg_rating — لو في نظام تقييم (Pre-aggregated)
③ member.active_borrows_count — للتحقق من الحد الأقصى بدون COUNT query
