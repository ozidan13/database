الـ Functional Dependencies

Functional Dependencies — The Mathematical Foundation of Database Design

قبل ما تعرف تطبّع قاعدة البيانات، لازم تفهم الاعتمادات الوظيفية — هي اللي بتقولك: "العمود ده بيحدد العمود ده". ده الأساس الأكاديمي لكل تصميم سليم.

🔗 ما هو الـ Functional Dependency؟

Functional Dependency — When One Attribute Determines Another

لو عندنا علاقة R، وقلنا إن X → Y ، ده معناه: "لكل قيمتين من صفوف R، لو اتفقوا في قيمة X، لازم يتفقوا في قيمة Y".
بالعربي: X بيحدد Y — لما تعرف X مش ممكن تبقى مش عارف Y.

🏢 أمثلة واقعية

emp_id → full_name — رقم الموظف يحدد اسمه
emp_id → dept_id — رقم الموظف يحدد قسمه
dept_id → dept_manager — كل قسم عنده مدير واحد
order_id, product_id → quantity — تحتاج الأوردر والمنتج معاً عشان تعرف الكمية
zip_code → city — الرمز البريدي يحدد المدينة (في معظم الحالات)

emp_id→full_name, salary, dept_id-- رقم الموظف يحدد كل بياناته
{order_id, product_id}→quantity, unit_price-- مفتاح مركّب
dept_id→dept_name, manager_id-- رقم القسم يحدد اسمه ومديره

الـ FD مش مجرد observation من البيانات الحالية — هو قيد دلالي (semantic constraint) بتحدده أنت كمصمّم بناءً على طبيعة الأعمال. لو في الجدول الحالي مش في تعارض، ده مش معناه إن الـ FD صحيحة — الـ FD بتنطبق على كل حالة محتملة.

⚠️ خطأ شائع: استنتاج الـ FD من البيانات فقط

مش كل تطابق في البيانات = FD حقيقية. مثلاً: لو كل موظفين في نفس القسم عندهم نفس الراتب بالصدفة، ده مش معناه
dept_id → salary. الـ FD لازم تكون صحيحة دايماً بحسب قواعد الأعمال.

🏷️ أنواع الـ Functional Dependencies

Trivial, Non-trivial, Full, Partial, Transitive

١. Trivial vs Non-trivial

Trivial FD

Y ⊆ X (Y جزء من X). دايماً صحيحة، لكن ما بتقول حاجة مفيدة.
مثال: {A, B} → A

Non-trivial FD

Y ⊄ X. هذه المفيدة في التصميم.
مثال: emp_id → salary

٢. Full vs Partial Dependency (مهمان للـ 2NF)

Order_Item(order_id, product_id, quantity, product_name)
{order_id, product_id}→quantity-- FULL: تحتاج المفتاح كاملاً
product_id→product_name-- PARTIAL: جزء من المفتاح يكفي ❌

٣. Transitive Dependency (مهم للـ 3NF)

Employee(emp_id, dept_id, dept_name)
emp_id→dept_id-- مباشرة
dept_id→dept_name-- مباشرة
emp_id→dept_name-- Transitive (عن طريق dept_id) ❌
⚗️ Armstrong's Axioms — قواعد الاستنتاج

Armstrong's Axioms — Sound and Complete Rules for FD Inference

قواعد Armstrong هي مجموعة قواعد كاملة ومتسقة لاستنتاج كل الـ FDs الممكنة من مجموعة معطاة. 3 قواعد أساسية + 3 مشتقة.

① Reflexivity

If Y ⊆ X, then X → Y

لو Y جزء من X، فـ X يحدد Y تلقائياً (Trivial FD). {A,B} → A دايماً صحيحة.

② Augmentation

If X → Y, then XZ → YZ

إضافة نفس الأعمدة للطرفين لا تغيّر صحة الـ FD. إذا emp→dept إذن emp,name→dept,name.

③ Transitivity

If X→Y and Y→Z, then X→Z

لو X يحدد Y ،وY يحدد Z، إذن X يحدد Z. أساس الـ Transitive Dependency.

القواعد المشتقة (Derived Rules)

④ Union

X→Y and X→Z → X→YZ

لو X يحدد Y وZ كل على حدة، فهو يحددهم معاً.

⑤ Decomposition

X→YZ → X→Y and X→Z

العكس: لو X يحدد YZ، ممكن تفصلهم.

⑥ Pseudo-transitivity

X→Y and WY→Z → WX→Z

تعميم للـ Transitivity مع إضافة عوامل.

🔒 Attribute Closure — X⁺

Closure of Attribute Set — What X Can Determine

إغلاق مجموعة الأعمدة X⁺ هو مجموعة كل الأعمدة اللي ممكن تُستَنتج من X باستخدام مجموعة الـ FDs. بتستخدمه لـ: اختبار FDs، إيجاد الـ Candidate Keys، تحقق من الـ Normal Forms.

خوارزمية حساب X⁺

①

ابدأ بـ X⁺ = X (نفس المجموعة)

②

لكل FD في المجموعة: إذا كان LHS ⊆ X⁺، أضف RHS لـ X⁺

③

كرر الخطوة ② حتى تتوقف X⁺ عن التغيّر (Fixed Point)

📐 مثال محلول: حساب {emp_id}⁺

FDs: emp_id → dept_id, salary, full_name
       dept_id → dept_name, manager_id

① X⁺ = {emp_id}
② emp_id → dept_id, salary, full_name → X⁺ = {emp_id, dept_id, salary, full_name}
② dept_id ⊆ X⁺ → أضف dept_name, manager_id → X⁺ = {emp_id, dept_id, salary, full_name, dept_name, manager_id}
③ لا جديد → انتهى

النتيجة: {emp_id}⁺ = كل الأعمدة → emp_id يصلح Candidate Key!

💡 متى X يكون Candidate Key؟

X هو Candidate Key إذا وفقط إذا: X⁺ = كل أعمدة العلاقة (يحدد كل شيء) — وليس هناك مجموعة فرعية من X تحقق نفس الشرط (Minimality).

🗝️ إيجاد Candidate Keys خطوة بخطوة

Finding All Candidate Keys Using FD Closure

📋 مثال: جدول Order_Invoice

الأعمدة: A, B, C, D, E
FDs: A → B,C
     C → D
     B,C → A
     E → A

①

حدد الأعمدة اللي ما تظهرش على اليمين (RHS) أبداً — هذه لازم تكون في كل Key.
E لا تظهر في RHS أبداً → E لازم في كل Candidate Key.

②

احسب {E}⁺: E→A, A→{B,C}, C→D → {E}⁺ = {E,A,B,C,D} = كل الأعمدة ✅
→ E هو Candidate Key!

③

جرب E مع إضافات: {E,A}⁺ = كل الأعمدة لكن {E} كافي → ليست Minimal → مش Candidate Key.

④

جرب A: {A}⁺ = {A,B,C,D} ≠ كل الأعمدة (E ناقصة) → A وحده مش كافي.

⑤

جرب {B,C}: {B,C}⁺ = {B,C,A,D} ≠ كل الأعمدة (E ناقصة) → ليس Key.
جرب {B,C,E}: {B,C,E}⁺ = كل الأعمدة لكن {E} كافي → ليست Minimal.

✅

Candidate Keys الوحيدة: {E} — لأن E دايماً لازمة وبتقفل كل الأعمدة.

✂️ Minimal Cover — الغطاء الأدنى

Minimal Cover (Canonical Cover) — The Smallest Equivalent Set of FDs

الـ Minimal Cover هو أصغر مجموعة FDs مكافئة للمجموعة الأصلية — بدون تكرار أو FDs زائدة. بتحتاجه لـ 3NF synthesis.

①

Simplify RHS: حوّل كل FD لـ single attribute على اليمين.
A → {B,C} بيبقى: A→B و A→C

②

Simplify LHS: لكل FD، اختبر إزالة عمود من LHS — لو الـ Closure ما تغيّرش، احذفه.
مثال: {A,B}→C وكان {A}→C → A كافي، احذف B

③

Remove Redundant FDs: احذف أي FD ممكن استنتاجها من البقية.
إذا A→B, B→C, وعندنا A→C → احذف A→C (مستنتجة بالـ Transitivity)

📐 مثال: Minimal Cover

المجموعة الأصلية:
A → BC, B → C, AB → C

① Split RHS: A→B, A→C, B→C, AB→C
② Simplify LHS: AB→C: هل A→C بدون B؟ نعم (A→C موجودة) → احذف B: A→C مكرر → ابقى معها
③ Remove Redundant: A→C: منتجة من A→B, B→C → احذفها
        AB→C: مع إزالة B من LHS → A→C لكن هي مكررة من A→B, B→C أيضاً → احذف

Minimal Cover: A→B, B→C

✍️ تمرين عملي

📦 تمرين: جدول Shipment

الجدول والـ FDs

Shipment(S, P, W, Q) — Supplier, Part, Warehouse, Quantity
FDs: S,P → Q  |  S,W → P  |  P,W → S

  • Aاحسب {S,P}⁺ و{S,W}⁺ و{P,W}⁺.
  • Bاذكر كل الـ Candidate Keys وبرّر.
  • Cهل S,P,W,Q → Q صحيحة؟ لماذا؟

A — حساب الـ Closures:

{S,P}⁺: S,P→Q → {S,P,Q}. لا جديد → {S,P,Q} ≠ كل الأعمدة
{S,W}⁺: S,W→P → {S,W,P}. الآن S,P→Q → {S,W,P,Q} = كل الأعمدة ✅
{P,W}⁺: P,W→S → {P,W,S}. الآن S,W→P موجودة, S,P→Q → {P,W,S,Q} = كل الأعمدة ✅
كمان تجرب {S,P,W}⁺ = كل الأعمدة لكن {S,W} و{P,W} كافيين.

B — Candidate Keys:

{S,W} ✅ و {P,W} ✅ — كلاهما Candidate Keys لأن كل منهما يحدد كل الأعمدة وهو Minimal (لا توجد مجموعة فرعية تحقق نفس الشرط).
{S,P,W} ليس Candidate Key لأنها غير Minimal — {S,W} وحده يكفي.

C — S,P,W,Q → Q:

نعم، صحيحة — لأنها Trivial FD: Q ⊆ {S,P,W,Q} بحكم قاعدة Reflexivity في Armstrong's Axioms. لكنها لا تضيف معلومة مفيدة.
