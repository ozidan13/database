خطوات التحويل السبعة

The 7-Step ER-to-Relational Mapping Algorithm

Strong Entities → Tables

كل Strong Entity بيبقى جدول. الصفات البسيطة بتبقى أعمدة. الـ Primary Key بتاع الكيان يبقى PK بتاع الجدول.
-- Entity: DEPARTMENT (Dept_ID, Name, Location) CREATE TABLE department ( dept_id INT PRIMARY KEY, name VARCHAR(100) NOT NULL, location VARCHAR(100) );

Weak Entities → Tables + FK

كل Weak Entity بيبقى جدول — بس الـ Primary Key بتاعه بيكون مركب: الـ Partial Key + الـ FK من الـ Owner Entity.
-- Weak Entity: DEPENDENT (depends on EMPLOYEE) CREATE TABLE dependent ( emp_id INT REFERENCES employee(emp_id), dep_name VARCHAR(100), birth_date DATE, PRIMARY KEY (emp_id, dep_name) );

1:1 Relationships → FK in One Side

في علاقة 1:1، بنضيف الـ FK في جانب الـ Total Participation (اللي لازم يشارك). لو الاتنين Total — بنختار أي جانب.
-- 1:1 → EMPLOYEE manages DEPARTMENT ALTER TABLE department ADD manager_id INT UNIQUE REFERENCES employee(emp_id);

1:N Relationships → FK in the N Side

أسهل خطوة! في علاقة 1:N، بنضيف الـ FK في جدول الجانب الـ N (الكتير). مثال: كل موظف في قسم واحد → نضيف dept_id في جدول الموظف.
-- 1:N → DEPARTMENT (1) has EMPLOYEE (N) ALTER TABLE employee ADD dept_id INT REFERENCES department(dept_id);

M:N Relationships → Junction Table

مهم جداً! علاقة M:N بتحتاج جدول وسيط جديد. الـ PK بتاعه = مفتاحين أجنبيين من الجدولين. ولو العلاقة عليها صفات — بتضاف كأعمدة في الجدول الوسيط.
-- M:N → STUDENT enrolls COURSE (with grade attribute) CREATE TABLE enrollment ( student_id INT REFERENCES student(student_id), course_id INT REFERENCES course(course_id), grade CHAR(2), -- relationship attribute semester VARCHAR(20), PRIMARY KEY (student_id, course_id) );

Multivalued Attributes → Separate Table

كل صفة متعددة القيم بتبقى جدول منفصل. الـ PK = الـ FK من الكيان الأصلي + القيمة نفسها.
-- Multivalued: Employee has multiple phone numbers CREATE TABLE emp_phone ( emp_id INT REFERENCES employee(emp_id), phone VARCHAR(20), PRIMARY KEY (emp_id, phone) );

Specialization/Generalization → Multiple Options

عندنا كذا طريقة:
Option A: جدول للـ Superclass + جدول لكل Subclass (بيشاركوا نفس الـ PK). ✅ الأفضل في أغلب الحالات.
Option B: جدول لكل Subclass بس — كل واحد فيه كل أعمدة الـ Superclass. ⚠️ بيسبب تكرار.
Option C: جدول واحد بس فيه كل الأعمدة + Type column. ⚠️ هيبقى في NULLs كتير.
