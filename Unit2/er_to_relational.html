ER → SQL Mapping

The 7-Step Algorithm: From ER Diagram to SQL Schema

الـ ER Diagram جميل على الورق — لكن الـ database أو بيفهم SQL. هنتعلم كيف نحوّل كل خطوة بخطوة.

لماذا المهارة دي حاسمة 7 خطوات كاملة 3 حالات الـ 1:1 Multivalued Attribute → Table Specialization → 3 approaches University Schema كامل
🎯 ليه التحويل مهارة؟

Why Mapping is a Core Skill

الـ ER Diagram هو التصميم والـ SQL Schema هو التنفيذ. التحويل الغلط بيسبّب:

💥 عواقب التحويل الغلط

◆ Data Redundancy: نفس البيانات متخزّنة أكتر من مرة → مشاكل عند التحديث.
◆ Update Anomalies: تعدّل في مكان وتنسى التاني → بيانات متناقضة.
◆ Delete Anomalies: حذف صف بتحذف معاه بيانات تانية محتاجها.
◆ Insert Anomalies: مش قادر تضيف بيانات إلا لو جبت بيانات تانية مش موجودة.

ده كله بيؤدي لـ Database Design Error وبيحتاج Refactoring مؤلم بعد كده.

📐 الخطوات 1 إلى 3
1

Strong Entities → Tables

Each Strong Entity → One Table

كل Strong Entity بتتحوّل لجدول. اسم الجدول = اسم الـ Entity. الصفات البسيطة بتتحوّل لأعمدة. Key Attribute بيبقى PRIMARY KEY.

  • كل صفة من صفات الـ Entity = عمود في الجدول
  • Key Attribute = PRIMARY KEY
  • الصفات المركّبة (Composite): اختار - إما افردها لأعمدة منفصلة أو خزّنها كعمود واحد حسب احتياج البحث
  • الصفات المشتقة (Derived): لا تُخزَّن — تُحسب في الاستعلام
Step 1 — STUDENT Entity
-- ER Entity: STUDENT(student_id, first_name, last_name, birth_date, gpa, phone*) -- *phone = Multivalued → هيتطاق في Step 6 -- *age = Derived → مش هنخزّنها CREATE TABLE student ( student_id SERIAL PRIMARY KEY, first_name VARCHAR(50) NOT NULL, -- Composite name مفردة last_name VARCHAR(50) NOT NULL, birth_date DATE NOT NULL, -- خزّن هذا -- age لا تُخزَّن: EXTRACT(year FROM AGE(birth_date)) gpa DECIMAL(3,2) CHECK (gpa BETWEEN 0 AND 4) );
2

Weak Entities → Tables with Composite PK

Partial Key + Owner FK = Composite PK

الـ Weak Entity بتتحوّل لجدول. الـ Primary Key بيكون مركّب = Partial Key + FK من الـ Owner. العلاقة Identifying بتتحوّل لـ FK بـ ON DELETE CASCADE.

  • أضف FK للـ Owner Entity (NOT NULL)
  • Primary Key = (FK + Partial Key)
  • ON DELETE CASCADE — لو الـ Owner اتحذف، الـ Weak Entity بتحذف معاه
Step 2 — DEPENDENT (Weak Entity of EMPLOYEE)
-- ER: DEPENDENT (dep_name [partial], relationship) — Owner: EMPLOYEE CREATE TABLE dependent ( emp_id INT NOT NULL REFERENCES employee(emp_id) ON DELETE CASCADE, -- الموظف اتحذف → التبعيين بيتحذفوا dep_name VARCHAR(100) NOT NULL, -- Partial Key relationship VARCHAR(30) NOT NULL, -- e.g. 'son','wife' birth_date DATE, PRIMARY KEY (emp_id, dep_name) -- Composite PK );
3

1:1 Relationships — 3 Cases

Which side gets the FK?

الـ 1:1 عندها 3 حالات مختلفة بناءً على الـ Participation:

Case A — Total/Total

الكيانين مرتبطين إلزامياً. ممكن تدمجهم في جدول واحد. أو ضع FK في أي من الجانبين (UNIQUE + NOT NULL).

COUNTRY — CAPITAL_CITY

Case B — Total/Partial

ضع الـ FK في جانب الـ Total (NOT NULL). الجانب Partial ممكن مش يشتغلش.

DEPARTMENT (Total) — MANAGER (Partial)

Case C — Partial/Partial

ضع الـ FK في الجانب الأكثر احتمالاً للمشاركة. الـ FK nullable.

EMPLOYEE — COMPANY_CAR

Step 3 — 1:1 Mapping Cases
-- Case A: COUNTRY (1:1, Total-Total) CAPITAL_CITY CREATE TABLE country ( country_id SERIAL PRIMARY KEY, country_name VARCHAR(100) NOT NULL, capital_id INT NOT NULL UNIQUE REFERENCES capital_city(city_id) -- Total+Total → ممكن FK في أي جانب. UNIQUE يضمن 1:1 ); -- Case B: DEPARTMENT (Total) HEADS (1:1) EMPLOYEE (Partial) -- الـ FK بيروح في DEPARTMENT لأنه Total (كل قسم لازم له رئيس) ALTER TABLE department ADD COLUMN head_emp_id INT NOT NULL UNIQUE REFERENCES employee(emp_id) ON DELETE RESTRICT; -- Case C: EMPLOYEE (Partial) DRIVES (1:1) COMPANY_CAR (Partial) ALTER TABLE company_car ADD COLUMN assigned_emp_id INT UNIQUE -- NULL = سيارة غير معيّنة REFERENCES employee(emp_id) ON DELETE SET NULL;

4

1:N Relationships → FK in N side

Foreign Key always goes to the "Many" side

الـ FK بيروح دايماً في جانب الـ N (الأبناء). الـ Total Participation من جانب N = NOT NULL على الـ FK. صفات العلاقة (لو موجودة) بتروح في جانب الـ N أيضاً.

Step 4 — 1:N Examples
-- DEPARTMENT (1) EMPLOYS (N) EMPLOYEE -- كل موظف في قسم واحد (Total) → NOT NULL ALTER TABLE employee ADD COLUMN dept_id INT NOT NULL -- Total Participation REFERENCES department(dept_id) ON DELETE RESTRICT; -- لا تحذف القسم لو فيه موظفين -- EMPLOYEE (1) SUPERVISES (N) EMPLOYEE — Unary 1:N ALTER TABLE employee ADD COLUMN supervisor_id INT -- Nullable: بعض الموظفين مش عندهم مشرف REFERENCES employee(emp_id) ON DELETE SET NULL; -- CUSTOMER (1) PLACES (N) ORDER — مع صفة العلاقة -- لو العلاقة عندها صفات، بتروح في جانب الـ N CREATE TABLE orders ( order_id SERIAL PRIMARY KEY, customer_id INT NOT NULL REFERENCES customer(customer_id), order_date TIMESTAMPTZ NOT NULL DEFAULT NOW(), status VARCHAR(20) NOT NULL DEFAULT 'pending' );
5

M:N Relationships → Junction Table

New Table with FKs from both sides

أي M:N بتحتاج جدول جديد (وسيط). الجدول الجديد بيحتوي على FK من كل جانب + أي صفات للعلاقة. الـ PK = مركّب من الـ FK-in.

Step 5 — M:N Mapping with Relationship Attributes
-- STUDENT (M) ENROLLS (N) COURSE -- صفات العلاقة: grade, semester, enrollment_date CREATE TABLE enrollment ( student_id INT NOT NULL REFERENCES student(student_id) ON DELETE CASCADE, course_id INT NOT NULL REFERENCES course(course_id) ON DELETE RESTRICT, grade CHAR(2) CHECK (grade IN ('A+','A','B+','B','C+','C','D','F')), semester VARCHAR(20) NOT NULL, -- '2024-Spring' enrollment_date DATE NOT NULL DEFAULT CURRENT_DATE, PRIMARY KEY (student_id, course_id, semester) -- طالب ممكن يكرر كورس ); -- DOCTOR (M) TREATS (N) PATIENT — Appointment CREATE TABLE appointment ( appt_id SERIAL PRIMARY KEY, -- surrogate PK أبسط هنا doctor_id INT NOT NULL REFERENCES doctor(doctor_id), patient_id INT NOT NULL REFERENCES patient(patient_id), appt_dt TIMESTAMPTZ NOT NULL, diagnosis TEXT, notes TEXT, UNIQUE (doctor_id, patient_id, appt_dt) -- نفس الطبيب والمريض مش في نفس الوقت );
🔧 الخطوات 6 و 7
6

Multivalued Attributes → Separate Table

Never store multivalued in same table

كل Multivalued Attribute بيتحوّل لجدول جديد. الجدول الجديد بيحتوي على FK من الـ Entity الأصلية + قيمة الـ Attribute. الـ PK = مركّب من الـ FK + القيمة.

Step 6 — Multivalued Attributes
-- STUDENT.phone (Multivalued) CREATE TABLE student_phone ( student_id INT NOT NULL REFERENCES student(student_id) ON DELETE CASCADE, phone VARCHAR(20) NOT NULL, phone_type VARCHAR(20) DEFAULT 'mobile', -- mobile / home / work PRIMARY KEY (student_id, phone) -- نفس الطالب مش عنده نفس الرقم مرتين ); -- EMPLOYEE.skills (Multivalued) CREATE TABLE employee_skill ( emp_id INT NOT NULL REFERENCES employee(emp_id) ON DELETE CASCADE, skill_name VARCHAR(50) NOT NULL, level VARCHAR(20) CHECK (level IN ('beginner','intermediate','expert')), PRIMARY KEY (emp_id, skill_name) ); -- DEPARTMENT.locations (Multivalued Complex — city + address) CREATE TABLE dept_location ( dept_id INT NOT NULL REFERENCES department(dept_id) ON DELETE CASCADE, city VARCHAR(50) NOT NULL, address VARCHAR(200), PRIMARY KEY (dept_id, city) );
7

Specialization / Generalization → 3 Approaches

Choose based on access patterns

فيه 3 طرق لتمثيل التخصص في SQL — اخترت في الدرس السابق. البرسنة: Approach A للـ Disjoint Total، Approach B للـ Overlapping أو Partial.

Step 7 — Specialization Approaches
-- Approach A: Single Table (Disjoint Total — e.g. EMPLOYEE types) CREATE TABLE employee ( emp_id SERIAL PRIMARY KEY, full_name VARCHAR(100) NOT NULL, emp_type VARCHAR(10) NOT NULL CHECK (emp_type IN ('salaried','hourly')), -- salaried attrs: monthly_sal DECIMAL(10,2) CHECK (monthly_sal > 0), -- hourly attrs: hourly_rate DECIMAL(8,2) CHECK (hourly_rate > 0) ); -- Approach B: Class Table Inheritance (recommended for Partial/Overlapping) CREATE TABLE person ( person_id SERIAL PRIMARY KEY, national_id CHAR(14) UNIQUE NOT NULL, full_name VARCHAR(100) NOT NULL ); CREATE TABLE employee ( person_id INT PRIMARY KEY REFERENCES person(person_id), hire_date DATE NOT NULL, salary DECIMAL(10,2) NOT NULL ); CREATE TABLE student ( person_id INT PRIMARY KEY REFERENCES person(person_id), student_id VARCHAR(20) UNIQUE NOT NULL, gpa DECIMAL(3,2) ); -- لو Overlapping: شخص ممكن يبقى في employee AND student

⚠️ أخطاء التحويل الشائعة
1. وضع FK في الجانب الغلط (1:N)

❌ بيضيفوا order_ids كعمود في CUSTOMER. ✅ الـ FK دايماً في جانب N: customer_id في ORDER.

2. نسيان جدول وسيط لـ M:N

❌ بيضيفوا course_ids array في STUDENT. ✅ ENROLLMENT جدول منفصل مع Composite PK.

3. تخزين Derived Attributes

❌ بيضيفوا age و total_price وorder_count كأعمدة ثابتة. ✅ احسبها في الاستعلام.

4. Multivalued في نفس الجدول

❌ phone1, phone2, phone3. ✅ جدول STUDENT_PHONE منفصل بـ FK + قيمة.

5. نسيان ON DELETE المناسب

❌ كل الـ FKs بدون ON DELETE (بيستخدم RESTRICT افتراضياً). ✅ فكّر في كل FK: لو اتحذف الـ parent — إيه اللي المفروض يحصل؟

🚗 تمرين: نظام إدارة سيارات

الـ ER المعطى

OWNER (owner_id PK, name, phones*) - VEHICLE (vin PK, brand, color, year) - INSURANCE (policy_no PK, Company, start_date, end_date)

العلاقات:
• OWNER OWNS VEHICLE: 1:N (مالك عنده سيارات كتير)
• VEHICLE HAS INSURANCE: 1:1 (سيارة عندها تأمين واحد — Total من ناحية السيارة، Partial من ناحية الـ Insurance — سيارة لازم مؤمّنة)
• VEHICLE HAS ACCIDENT: 1:N — ACCIDENT هو Weak Entity (accident_seq partial key)

  • Aطبّق الـ 7 خطوات واكتب كل الجداول بـ SQL.
  • Bفين بيروح الـ FK في العلاقة 1:1 بين VEHICLE وINSURANCE؟
B — فين الـ FK في 1:1؟

السيارة Total Participation (لازم عندها تأمين). الـ FK بيروح في جدول VEHICLE لأن Total Participation من ناحيتها. نضيف policy_no FK NOT NULL UNIQUE في VEHICLE.

Car Management System — Solution
-- Step 1: Strong Entities CREATE TABLE owner ( owner_id SERIAL PRIMARY KEY, full_name VARCHAR(100) NOT NULL ); CREATE TABLE insurance ( policy_no VARCHAR(30) PRIMARY KEY, company VARCHAR(100) NOT NULL, start_date DATE NOT NULL, end_date DATE NOT NULL CHECK (end_date > start_date) ); CREATE TABLE vehicle ( vin CHAR(17) PRIMARY KEY, -- Vehicle Identification Number brand VARCHAR(50) NOT NULL, color VARCHAR(30), year INT CHECK (year BETWEEN 1990 AND 2030), -- Step 3: 1:1 Total from vehicle side policy_no VARCHAR(30) NOT NULL UNIQUE REFERENCES insurance(policy_no), -- Step 4: 1:N Owner → Vehicle owner_id INT NOT NULL REFERENCES owner(owner_id) ON DELETE RESTRICT ); -- Step 2: Weak Entity ACCIDENT CREATE TABLE accident ( vin CHAR(17) NOT NULL REFERENCES vehicle(vin) ON DELETE CASCADE, accident_seq INT NOT NULL, -- Partial Key accident_dt TIMESTAMPTZ NOT NULL, description TEXT, damage_cost DECIMAL(10,2), PRIMARY KEY (vin, accident_seq) ); -- Step 6: Multivalued phones CREATE TABLE owner_phone ( owner_id INT NOT NULL REFERENCES owner(owner_id) ON DELETE CASCADE, phone VARCHAR(20) NOT NULL, PRIMARY KEY (owner_id, phone) );
