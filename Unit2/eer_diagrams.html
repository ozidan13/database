الـ EER Diagrams

Enhanced Entity-Relationship — Specialization, Generalization & Beyond

الـ ER العادي مش بيعبّر عن كل المواقف — الـ EER جاء عشان يغطّي الـ inheritance والتخصص اللي بنشوفه في الواقع يومياً.

🤔 ليه نحتاج الـ EER؟

Limitations of Basic ER — Why EER Exists

في الـ ER العادي، كل الـ Entities بتتعامل كأنها "متساوية". لكن في الواقع، عندنا مفاهيم زي:

⚡ مثال على قصور الـ ER العادي

في نظام موارد بشرية — عندنا entity EMPLOYEE. بعض الموظفين hourly والبعض salaried. الـ hourly بيتحسب مرتبهم بـ hours × rate والـ salaried بيبقى ليهم monthly_salary. في الـ ER العادي، هتعمل أعمدة nullable زي: hourly_rate (null لو salaried)، monthly_salary (null لو hourly). ده قبيح وبيسمح بـ inconsistencies.

الـ EER (Enhanced ER) جاء بـ 3 مفاهيم جديدة:

  1. Specialization / Generalization — التسلسل الهرمي (Superclass / Subclass)
  2. Category (Union Types) — Entity من عدة أنواع مختلفة
  3. Aggregation — علاقة بتشارك في علاقة تانية
🌳 التخصص والتعميم

Specialization (Top-Down) vs Generalization (Bottom-Up)

Top-Down

Specialization — التخصص

بنبدأ من Entity عامة (Superclass) ونقسّمها لـ Subclasses أكثر تخصصاً. مثال: PERSON → STUDENT, EMPLOYEE.
متى نستخدمها؟
لما نلاحظ إن مجموعة فرعية من الـ instances عندها صفات أو علاقات مختلفة عن البقية.
Bottom-Up

Generalization — التعميم

بنبدأ من Entities مستقلة ونوجد Superclass تجمعهم. مثال: CAR + TRUCK + MOTORCYCLE → VEHICLE.
متى نستخدمها؟
لما بنلاقي entities مختلفة عندها صفات مشتركة كتير — بنضمها في Superclass.
💡 النتيجة واحدة — الفرق في الاتجاه

Specialization وGeneralization بينتجوا نفس الهيكل الهرمي — الفرق هو إنت بتبدأ من فوق أو من تحت. في الـ ER Diagram، الرمز هو دائرة بيطلع منها سهم للـ Subclasses. الدائرة بتحتوي على:
• حرف "d" = Disjoint (كل subclass منفصلة)
• حرف "o" = Overlapping (الـ instance ممكن ينتمي لأكتر من subclass)

🧬 وراثة الصفات

Attribute Inheritance — What Subclasses Inherit

الـ Subclass بترث كل صفات وعلاقات الـ Superclass تلقائياً. وبالإضافة لكده، بيبقى ليها صفاتها الخاصة.

🏢 في الشغل الحقيقي — متى تغري التخصص؟

◆ لو الـ Subclasses عندها صفات مختلفة (سؤال: هل صح إني أضيف عمود nullable؟)
◆ لو الـ Subclasses عندها علاقات مختلفة مع entities تانية.
◆ لو بتعمل polymorphic behavior — الـ Superclass بتتعامل بطريقة موحدة.

◆ مش مهم لو الـ Subclasses عندهم نفس الصفات والعلاقات تماماً — ساعتها خليها entity واحدة بـ type column.

🔒 القيود الأربعة للـ Specialization

The Four Constraint Combinations

في كل Specialization، بيبقى فيه بعدين نحدّد فيهم القيود:

  • Disjoint (d) vs Overlapping (o) — هل الـ Instance ينتمي لـ Subclass واحدة بس أم أكتر؟
  • Total vs Partial — هل كل Superclass instance لازم ينتمي لـ Subclass؟
DisjointTotal

التخصص الكامل المنفصل

كل instance من الـ Superclass لازم ينتمي لـ Subclass واحدة بالظبط — مش أكتر ومش أقل.
مثال: EMPLOYEE
EMPLOYEE → {FULL_TIME_EMP, PART_TIME_EMP}
كل موظف إما full-time أو part-time — مش الاتنين ومش ولا واحدة. المبنى على كده إن لو EMPLOYEE مش موجود في أي منهم = البيانات غلط.

مثال ثاني: VEHICLE
VEHICLE → {CAR, TRUCK, MOTORCYCLE} — كل عربية لها نوع واحد بس.
DisjointPartial

التخصص الجزئي المنفصل

الـ instance ممكن ينتمي لـ Subclass واحدة — أو مش ينتمي لأي حاجة.
مثال: PERSON
PERSON → {EMPLOYEE, STUDENT}
البنك عنده بيانات أشخاص. منهم موظفين ومنهم طلاب — لكن ممكن يكون عنده person مش موظف ولا طالب (زي مقدّم طلب توظيف). كمان مش ممكن يبقى الاتنين (في هذا النظام).
OverlappingTotal

التخصص الكامل المتداخل

كل instance لازم ينتمي لـ Subclass واحدة على الأقل — وممكن يبقى في أكتر من واحدة.
مثال: STAFF في الجامعة
STAFF → {INSTRUCTOR, RESEARCHER, ADMINISTRATOR}
كل موظف بالجامعة لازم له دور واحد على الأقل. بعضهم بيدرّس ويعمل بحث في نفس الوقت. كمان بعضهم مدرّس وإداري.

مثال ثاني: DOCTOR specialties
DOCTOR → {SURGEON, INTERNIST, PEDIATRICIAN} — دكتور ممكن يبقى surgeon وinternist في نفس الوقت.
OverlappingPartial

التخصص الجزئي المتداخل

ممكن يبقى في أي عدد من الـ Subclasses — أو برّاهم كلهم. الأكثر مرونة والأقل تقييداً.
مثال: PERSON في نظام عام
PERSON → {CUSTOMER, EMPLOYEE, SUPPLIER}
في بعض الشركات، الشخص ممكن يكون عميل وموظف في نفس الوقت. وممكن يكون PERSON مش أي منهم (مجرد contact).

🗄️ 3 طرق لتمثيل Specialization في الـ SQL

①

Single Table (STI)

كل الـ Superclass والـ Subclasses في جدول واحد. عمود "type" بيحدّد النوع. الأعمدة الخاصة بكل subclass بتبقى nullable.

  • بسيط — query واحدة
  • JOIN-free performance
  • أعمدة كتير nullable
  • مش ممكن NOT NULL على subclass attrs
②

Class Table (CTI)

جدول للـ Superclass وجدول لكل Subclass. جداول الـ Subclasses بتاخد نفس PK الـ Superclass كـ FK.

  • Type safety — كل column meaningful
  • NOT NULL ممكن على subclass attrs
  • JOINs عند كل query
  • أكتر complex
③

Concrete Table

جدول مستقل لكل Subclass بيضم كلها صفات الـ Superclass + صفاتها الخاصة. مش بيبقاش جدول للـ Superclass.

  • كل subclass مستقل تماماً
  • تكرار الأعمدة المشتركة
  • Polymorphic queries صعبة
  • مش مناسب لـ Overlapping
Approach ② CTI — Class Table Inheritance
-- Superclass: PERSON CREATE TABLE person ( person_id SERIAL PRIMARY KEY, national_id CHAR(14) UNIQUE NOT NULL, full_name VARCHAR(100) NOT NULL, birth_date DATE NOT NULL, phone VARCHAR(20) ); -- Subclass: EMPLOYEE — بتاخد نفس الـ PK CREATE TABLE employee ( person_id INT PRIMARY KEY REFERENCES person(person_id) ON DELETE CASCADE, emp_number VARCHAR(20) UNIQUE NOT NULL, hire_date DATE NOT NULL, salary DECIMAL(10,2) NOT NULL CHECK (salary > 0), position VARCHAR(50) ); -- Subclass: STUDENT CREATE TABLE student ( person_id INT PRIMARY KEY REFERENCES person(person_id) ON DELETE CASCADE, student_code VARCHAR(20) UNIQUE NOT NULL, enroll_year INT NOT NULL, gpa DECIMAL(3,2) CHECK (gpa BETWEEN 0 AND 4), program VARCHAR(50) ); -- Query: اجيب بيانات employee مع بياناته كـ person SELECT p.full_name, p.national_id, e.emp_number, e.salary, e.hire_date FROM person p INNER JOIN employee e USING (person_id);
🔗 Category و Aggregation

Union Types (Category) and Aggregation

Category — Union Type

Entity بتكون subclass من أكتر من Superclass من أنواع مختلفة. مثال: مالك السيارة ممكن يكون شخص طبيعي أو شركة.
مثال: VEHICLE_OWNER
VEHICLE_OWNER هو union من {PERSON, COMPANY}
المالك ممكت يكون person أو company — مش الاتنين.

VEHICLE → BELONGS_TO → VEHICLE_OWNER ⊃ PERSON
                                                                 ⊃ COMPANY

في SQL: بنستخدم discriminator column: owner_type ENUM('person','company') مع nullable FKs.

Aggregation — علاقة فوق علاقة

لما علاقة M:N نفسها بتشارك في علاقة تانية. الـ ER العادي مش بيدعم ده مباشرة.
مثال: نظام بحثي
EMPLOYEE works_on PROJECT
العلاقة "WORKS_ON" دي بتتربط بـ GRANT (المنحة البحثية).
يعني: "منحة X بتمّول عمل الموظف Y في المشروع Z".

الحل: نعمل WORKS_ON كـ Entity (Junction Table) وبعدين نربطها بـ GRANT.
FUNDING (works_on_id FK, grant_id FK, funded_amount)

Complete EER — HR System with Employee Specialization

📋 المتطلبات
الشركة فيها موظفين. كل موظف عنده ID ووطني واسم وبيانات تواصل. الموظفين ينقسموا لنوعين مستحيل يتداخلوا: موظفين دوام كامل وموظفين جزئي. الدوام الكامل عنده monthly_salary وleave_entitlement. الجزئي عنده hourly_rate وmax_hours_per_week.

من الدوام الكامل، بعضهم مدراء (MANAGER) — عندهم bonus وmanagement_level. بعضهم مهندسين (ENGINEER) عندهم technical_specialization.

بعض الموظفين ممكن يكونوا في نفس الوقت مدراء ومهندسين (مدير تقني). والحالة دي Overlapping.

📖 قراءة الـ Diagram

• EMPLOYEE → {FULL_TIME, PART_TIME}: d (Disjoint) + Total — كل موظف لازم في واحدة بس.

SQL — Class Table Inheritance
HR System — EER to SQL
-- Level 1: Superclass CREATE TABLE employee ( emp_id SERIAL PRIMARY KEY, national_id CHAR(14) UNIQUE NOT NULL, full_name VARCHAR(100) NOT NULL, hire_date DATE NOT NULL, emp_type VARCHAR(10) NOT NULL CHECK (emp_type IN ('full','part')) ); -- Level 2: Subclass FULL_TIME {d, Total coverage via emp_type} CREATE TABLE full_time_emp ( emp_id INT PRIMARY KEY REFERENCES employee(emp_id), monthly_salary DECIMAL(10,2) NOT NULL, leave_days INT NOT NULL DEFAULT 21 ); -- Level 2: Subclass PART_TIME CREATE TABLE part_time_emp ( emp_id INT PRIMARY KEY REFERENCES employee(emp_id), hourly_rate DECIMAL(8,2) NOT NULL, max_hours_week INT NOT NULL DEFAULT 20 ); -- Level 3: MANAGER (overlapping — ممكن يكون engineer في نفس الوقت) CREATE TABLE manager ( emp_id INT PRIMARY KEY REFERENCES full_time_emp(emp_id), bonus DECIMAL(10,2), mgmt_level INT CHECK (mgmt_level BETWEEN 1 AND 5) ); -- Level 3: ENGINEER CREATE TABLE engineer ( emp_id INT PRIMARY KEY REFERENCES full_time_emp(emp_id), specialization VARCHAR(50) NOT NULL -- software, civil, etc. ); -- لو حبيت تعرف مين "مدير تقني" (كلها مع بعض) SELECT e.full_name, m.mgmt_level, eng.specialization FROM employee e JOIN full_time_emp ft USING(emp_id) JOIN manager m USING(emp_id) JOIN engineer eng USING(emp_id); -- INNER JOIN = فقط اللي فيهم الاتنين
✍️ تمرين عملي

🏦 تمرين: نظام بنك

البنك فيه حسابات بنكية. كل حساب عنده IBAN وتاريخ فتح ورصيد. الحسابات نوعين لا يتداخلوا: حساب جاري (checking) وحساب توفير (savings). الجاري عنده overdraft_limit. التوفير عنده interest_rate وmin_balance. من الحسابات الجارية، بعضها حسابات شركات (BUSINESS) عندها tax_number وcredit_limit. والبعض حسابات أفراد (PERSONAL) عندها linked_national_id.
  • Aارسم الـ EER Hierarchy مع كل القيود.
  • Bما هو نوع القيود؟ (Disjoint/Overlapping + Total/Partial)
  • Cاكتب جداول SQL باستخدام Class Table Inheritance.
A + B — الـ EER

ACCOUNT → {CHECKING, SAVINGS}: Disjoint + Total — كل حساب نوع واحد بالظبط.
CHECKING → {BUSINESS, PERSONAL}: Disjoint + Partial — حساب جاري ممكن يكون personal أو business، أو لا شيء (تجاري عادي). لكن مستحيل يكون الاتنين في آن واحد.

Bank Account Hierarchy — CTI
CREATE TABLE account ( account_id SERIAL PRIMARY KEY, iban VARCHAR(34) UNIQUE NOT NULL, open_date DATE NOT NULL DEFAULT CURRENT_DATE, balance DECIMAL(15,2) NOT NULL DEFAULT 0, acc_type VARCHAR(10) NOT NULL CHECK (acc_type IN ('checking','savings')) ); CREATE TABLE checking_account ( account_id INT PRIMARY KEY REFERENCES account(account_id), overdraft_limit DECIMAL(12,2) NOT NULL DEFAULT 0 ); CREATE TABLE savings_account ( account_id INT PRIMARY KEY REFERENCES account(account_id), interest_rate DECIMAL(5,4) NOT NULL, min_balance DECIMAL(12,2) NOT NULL DEFAULT 0 ); CREATE TABLE business_checking ( account_id INT PRIMARY KEY REFERENCES checking_account(account_id), tax_number VARCHAR(20) UNIQUE NOT NULL, credit_limit DECIMAL(15,2) ); CREATE TABLE personal_checking ( account_id INT PRIMARY KEY REFERENCES checking_account(account_id), national_id CHAR(14) NOT NULL );
