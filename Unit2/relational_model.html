📐 التعريف الرسمي للـ Relation

Mathematical Definition of a Relation

اقترح Edgar F. Codd سنة 1970 نموذجاً رياضياً لتمثيل البيانات قائماً على مفهوم Set Theory. الـ Relation (الجدول) مش مجرد "صفوف وأعمدة" — له تعريف رياضي دقيق.

📐 التعريف الرياضي

لو عندنا مجالات (Domains): D₁, D₂, ..., Dₙ
الـ Relation R هي مجموعة فرعية (subset) من الناتج الديكارتي: D₁ × D₂ × ... × Dₙ

بالكلام البسيط: الجدول هو مجموعة من الـ tuples (الصفوف) — كل tuple بتختار قيمة واحدة من كل domain. المجالات هي الأنواع المسموح بها لكل عمود.

🔑 المصطلحات الرياضية vs المصطلحات العملية

Relation = جدول (Table) | Tuple = صف (Row) | Attribute = عمود (Column)
Domain = نوع البيانات المسموح به | Degree = عدد الأعمدة | Cardinality = عدد الصفوف

📋 خصائص الـ Relation

Properties That Define a Valid Relation

مش كل "جدول" هو Relation بالمعنى الرياضي. لازم تتحقق الخصائص دي:

① الصفوف مش مرتّبة (Unordered Tuples)

ترتيب الصفوف في الجدول مش له معنى رياضي. النتيجة من SELECT مش لها ترتيب مضمون ما لم تحدّد ORDER BY. مبتركنش على ترتيب الإدراج!

② الأعمدة مش مرتّبة (Unordered Attributes)

ترتيب الأعمدة في الجدول مش مهم رياضياً. في SQL ممكن تعمل SELECT * وتاخد أعمدة بترتيب تعريفها — لكن الصح إنك تحدّد الأعمدة بالترتيب اللي تحتاجه.

③ كل قيمة ذرية (Atomic Values)

كل خلية في الجدول لازم تحتوي على قيمة واحدة بسيطة — مش list أو مجموعة. ده أساس الـ First Normal Form (1NF). مش ممكن تحط "أحمد, فاطمة" في خانة الاسم.

④ لا توجد صفوف مكرّرة (No Duplicate Tuples)

الـ Relation هي مجموعة (Set) — والمجموعة لا تحتوي على عناصر مكرّرة. الـ PK هو الضامن لعدم التكرار. في SQL الفعلي، بيسمح بتكرار لو مفيش PK — لكن ده يخلّ بالنموذج العلائقي.

⑤ الـ NULL معناه "غير معروف"

NULL مش صفر ومش string فاضية. هو يمثّل قيمة غير موجودة أو غير معروفة وقت الإدخال. عنده منطق ثلاثي (3-valued logic): TRUE, FALSE, UNKNOWN.

🗝️ أنواع المفاتيح

Key Types — From Super Key to Foreign Key

المفاتيح هي آلية التعرّف الفريد على الصفوف وربط الجداول ببعض. فيه تدرّج من الأعم للأخص:

Super Key

المفتاح الفائق

أي مجموعة من الأعمدة بتحدّد كل صف بشكل فريد — حتى لو فيها أعمدة زيادة مش ضروروية للتمييز.

{student_id}
{student_id, name}
{national_id}
{name, birth_date, phone}
Candidate Key

المفتاح المرشّح

Super Key "ممكّن" — فريد وبدون أعمدة زيادة. يعني لو حذفت أي عمود منه، مش هيكون فريد بعد كده.

student_id → فريد وحده ✅
national_id → فريد وحده ✅
email → فريد وحده ✅
Primary Key

المفتاح الأساسي

Candidate Key بختاروه كـ "الممثّل الرسمي" للـ Relation. لازم NOT NULL ولازم UNIQUE دايماً.

student_id SERIAL PRIMARY KEY
order_id UUID PRIMARY KEY
-- Composite PK:
(student_id, course_id)
Alternate Key

المفتاح البديل

Candidate Keys التانية اللي مختارناهاش كـ PK. بنحطها كـ UNIQUE constraint في الـ SQL.

national_id UNIQUE NOT NULL
email UNIQUE NOT NULL
username UNIQUE NOT NULL
Composite Key

المفتاح المركّب

PK مكوّن من أكتر من عمود. كل عمود لوحده مش فريد، بس الاتنين مع بعض فريد.

ENROLLMENT:
(student_id, course_id)
NURSE_DEPT:
(nurse_id, dept_id, start_date)
Foreign Key

المفتاح الأجنبي

عمود في جدول B بيشير لـ PK في جدول A. بيربط الجداول ويضمن Referential Integrity.

EMPLOYEE.dept_id
  REFERENCES DEPARTMENT(dept_id)
ORDER.customer_id
  REFERENCES CUSTOMER(id)
💡 كيف تختار الـ PK؟

المواصفات المثالية:
✅ Stable — مش بيتغيّر مع الوقت (مش الاسم ومش الإيميل)
✅ Short — للـ Performance في الـ JOINs والـ Indexes
✅ Meaningless — لا يحمل بيانات عمل (لأن البيانات بتتغيّر)
✅ Simple — عمود واحد أسهل من مركّب

الخيارات: SERIAL (auto-increment INT) أو UUID (للـ distributed systems) — كلاهما Surrogate Keys.

🛡️ قيود التكامل الأربعة

The 4 Integrity Constraints

الـ DBMS بيطبّق هذه القيود تلقائياً — هدفها إن البيانات تبقى صحيحة ومنطقية في أي وقت.

①

Domain Integrity — تكامل المجال

كل قيمة في عمود لازم تكون من النوع والنطاق المسموح به لهذا العمود.

الموقفالانتهاكالـ DBMS يعمل إيه
إدخال 'abc' في عمود salary DECIMALانتهاك DomainERROR: invalid input syntax for type numeric
إدخال gpa = 5.0 وعندنا CHECK (gpa BETWEEN 0 AND 4)انتهاك CHECKERROR: new row violates check constraint
إدخال '-1' في عمود age INT CHECK (age > 0)انتهاك DomainERROR: check constraint violation
إدخال 'ahmed@uni.edu' في عمود email VARCHAR(100)سليم ✅تمام
Domain Integrity in SQL
CREATE TABLE employee ( emp_id SERIAL PRIMARY KEY, full_name VARCHAR(100) NOT NULL, salary DECIMAL(10,2) NOT NULL CHECK (salary > 0), age INT CHECK (age BETWEEN 18 AND 65), email VARCHAR(150) CHECK (email LIKE '%@%'), emp_type VARCHAR(10) CHECK (emp_type IN ('full','part','contract')) );
②

Key Integrity — تكامل المفتاح

الـ Primary Key لازم يكون فريداً (UNIQUE) وغير فارغ (NOT NULL) في أي وقت.

الموقفالانتهاكالـ DBMS يعمل إيه
إضافة طالب بـ student_id = 20230001 وده موجود قبل كدهDuplicate PKERROR: duplicate key value violates unique constraint
إضافة طالب بـ student_id = NULLNULL PKERROR: null value in column "student_id" violates not-null constraint
تحديث PK لطالب موجود إلى قيمة موجودة لطالب آخرDuplicate PKERROR: duplicate key value
🏢 لو بتستخدم SERIAL / AUTO_INCREMENT

الـ DBMS بيتولى الأمر وحده — ما فيش تكرار. لو بتحدّد الـ ID يدوياً (زي في data imports) — خطر انتهاك Key Integrity أعلى. استخدم ON CONFLICT DO NOTHING/UPDATE في PostgreSQL.

③

Entity Integrity — تكامل الكيان

لا يوجد جدول أساسي (Base Table) بدون Primary Key. والـ PK لازم NOT NULL دايماً.

الموقفالنتيجة
جدول بدون PK معرّفmutable anonymity — ممكن تيجي صفوف مكرّرة تماماً
PK يقبل NULLEntity لا يمكن التعرّف عليها بشكل موثوق
كل جدول عنده PKEntity Integrity محققة ✅
⚠️ الخطأ الشائع — جداول بدون PK

بعض الـ developers بيعملوا junction tables بدون PK صريح. مثال: ENROLLMENT(student_id, course_id) بدون PRIMARY KEY. النتيجة: ممكن يتسجّل الطالب في نفس الكورس مرتين! الحل: PRIMARY KEY(student_id, course_id) أو إضافة enrollment_id SERIAL PK.

④

Referential Integrity — التكامل المرجعي

أي قيمة في عمود Foreign Key لازم تكون موجودة في الـ Primary Key بتاع الجدول المُشار إليه — أو تكون NULL (لو الـ FK nullable).

الانتهاكالمثالالـ DBMS Response
INSERT FK غير موجودإضافة موظف بـ dept_id = 999 وده مش موجودERROR: insert or update on table "employee" violates foreign key constraint
DELETE PK مُشار إليهحذف قسم وفيه موظفين مرتبطين بيهحسب ON DELETE action (ERROR / CASCADE / SET NULL)
UPDATE PK مُشار إليهتغيير dept_id في DEPARTMENT وفيه employees بيشيروا إليهحسب ON UPDATE action
ON DELETE / ON UPDATE Options
-- ON DELETE CASCADE: لو حذفت القسم، احذف الموظفين dept_id INT REFERENCES department(dept_id) ON DELETE CASCADE ON UPDATE CASCADE; -- ON DELETE SET NULL: لو حذفت القسم، اجعل dept_id = NULL dept_id INT REFERENCES department(dept_id) ON DELETE SET NULL; -- ON DELETE RESTRICT (افتراضي): امنع حذف الـ parent لو فيه children dept_id INT REFERENCES department(dept_id) ON DELETE RESTRICT; -- ON DELETE SET DEFAULT: ضع القيمة الافتراضية dept_id INT DEFAULT 1 REFERENCES department(dept_id) ON DELETE SET DEFAULT;
🏢 متى تستخدم كل ON DELETE؟

CASCADE: ORDER → ORDER_ITEMS (البنود لا معنى لها بدون الأوردر)
SET NULL: EMPLOYEE.dept_id (الموظف ما زال موجوداً بعد حذف القسم)
RESTRICT: CATEGORY → PRODUCTS (امنع حذف فئة فيها منتجات — البزنس rule)
SET DEFAULT: نادر — لما يكون فيه قيمة "غير محدد" بشكل رسمي

❓ الـ NULL وما يسبّبه

NULL — The Three-Valued Logic

الـ NULL مفهوم غريب لو اتعاملت معاه زي قيمة عادية. هو يعني "غير معروف" أو "غير منطبق". النتيجة هي منطق ثلاثي القيم:

NULL Handling in Practice
-- ❌ غلط: WHERE dept_id = NULL لن يرجع شيئاً! SELECT * FROM employee WHERE dept_id = NULL; -- دايماً 0 rows -- ✅ الصح: استخدم IS NULL SELECT * FROM employee WHERE dept_id IS NULL; -- COALESCE: ارجع قيمة بديلة لو NULL SELECT full_name, COALESCE(dept_id::TEXT, 'غير مُعيَّن') AS department, COALESCE(salary, 0) AS salary FROM employee; -- NULLIF: ارجع NULL لو القيمتين متساويتين SELECT NULLIF(quantity, 0) FROM order_items; -- يمنع قسمة على صفر -- NOT IN مع NULLs — الفخ الشهير SELECT * FROM student WHERE dept_id NOT IN (1, 2, NULL); -- يرجع 0 rows دايماً! UNKNOWN -- الصح: NOT IN مع قائمة بدون NULL، أو استخدم NOT EXISTS

🛍️ تمرين: تصميم Schema لمتجر إلكتروني

السيناريو

عايزين نبني Schema لمتجر إلكتروني بسيط: Customers, Products, Categories, Orders, Order_Items

  • Aحدّد الـ PK لكل جدول مع تبرير اختيارك.
  • Bحدّد الـ FKs وما هي الـ ON DELETE المناسبة لكل FK.
  • Cهل هناك Composite Keys؟ أين ومتى؟
  • Dأين يجب وضع NOT NULL وأين ممكن يكون NULL؟
E-Commerce Schema with All Constraints
CREATE TABLE category ( category_id SERIAL PRIMARY KEY, category_name VARCHAR(50) NOT NULL UNIQUE, parent_id INT REFERENCES category(category_id) ON DELETE SET NULL -- ↑ Recursive (Self-ref): فئة أب. لو الأب اتحذف → الأبناء parent_id = NULL ); CREATE TABLE customer ( customer_id SERIAL PRIMARY KEY, email VARCHAR(150) NOT NULL UNIQUE, -- Alternate Key full_name VARCHAR(100) NOT NULL, phone VARCHAR(20), -- Nullable: ممكن مش معاه created_at TIMESTAMPTZ NOT NULL DEFAULT NOW() ); CREATE TABLE product ( product_id SERIAL PRIMARY KEY, sku VARCHAR(50) NOT NULL UNIQUE, -- Alternate Key product_name VARCHAR(200) NOT NULL, price DECIMAL(10,2) NOT NULL CHECK (price >= 0), stock_qty INT NOT NULL DEFAULT 0 CHECK (stock_qty >= 0), category_id INT REFERENCES category(category_id) ON DELETE RESTRICT -- ↑ RESTRICT: امنع حذف فئة فيها منتجات ); CREATE TABLE orders ( order_id SERIAL PRIMARY KEY, customer_id INT NOT NULL REFERENCES customer(customer_id) ON DELETE RESTRICT, -- لا تحذف عميل عنده أوردرات order_date TIMESTAMPTZ NOT NULL DEFAULT NOW(), status VARCHAR(20) NOT NULL DEFAULT 'pending' CHECK (status IN ('pending','confirmed','shipped','delivered','cancelled')), total_amount DECIMAL(12,2) NOT NULL DEFAULT 0 CHECK (total_amount >= 0) ); CREATE TABLE order_item ( -- Composite PK: الأوردر + المنتج (مش ينفع نكرر نفس المنتج في نفس الأوردر) order_id INT NOT NULL REFERENCES orders(order_id) ON DELETE CASCADE, -- الأوردر اتحذف؟ احذف بنوده product_id INT NOT NULL REFERENCES product(product_id) ON DELETE RESTRICT, -- منع حذف منتج له orders quantity INT NOT NULL CHECK (quantity > 0), unit_price DECIMAL(10,2) NOT NULL CHECK (unit_price >= 0), -- ↑ متلازم: unit_price في وقت الشراء (مش السعر الحالي) PRIMARY KEY (order_id, product_id) -- Composite PK );
B — تبرير ON DELETE لكل FK

PRODUCT.category_id → RESTRICT: منع حذف فئة فيها منتجات (business rule)
ORDER.customer_id → RESTRICT: منع حذف عميل عنده أوردرات (للأرشيف والمحاسبة)
ORDER_ITEM.order_id → CASCADE: لو الأوردر اتحذف، البنود لا معنى لها
ORDER_ITEM.product_id → RESTRICT: لا تحذف منتج له بنود في أوردرات قديمة
