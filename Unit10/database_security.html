💉 SQL Injection — أكبر خطر

SQL Injection — #1 Web Security Risk, How It Works & How to Stop It

SQL Injection = الهجوم الأكثر شيوعاً على قواعد البيانات منذ 20 سنة. المهاجم يُدخل SQL Code في خانة Input العادية ليُنفَّذ كـ Query.

❌ VULNERABLE — String Concatenation
// User Input:
email = "' OR 1=1 --"

// Generated Query:
SELECT * FROM users
WHERE email = '' OR 1=1 --'

// 1=1 دائماً true → كل الـ rows
// -- يُعلّق باقي الـ Query
✅ SAFE — Parameterized Query
// User Input:
email = "' OR 1=1 --"

// Query معزولة عن الـ Input:
await pool.query(
'SELECT * FROM users WHERE email=$1',
[email] // treated as data, NOT code
);
// يبحث عن email literal
Injection Prevention — Node.js مع pg وTypeORM
// ✅ pg: Parameterized Query ($1, $2, ...) const result = await pool.query( 'SELECT id, name FROM users WHERE email = $1 AND role = $2', [email, role] ); // ✅ TypeORM: Query Builder const user = await userRepo .createQueryBuilder('u') .where('u.email = :email', { email }) // Parameterized .andWhere('u.is_active = :active', { active: true }) .getOne(); // ✅ Drizzle ORM: Type-safe const user = await db.select() .from(users) .where(eq(users.email, email)); // Never raw string // ⚠️ إذا حتجت Dynamic Table/Column name: validate whitelist const ALLOWED_COLS = ['name', 'email', 'created_at']; if (!ALLOWED_COLS.includes(sortCol)) throw new Error('Invalid column'); const query = `SELECT * FROM users ORDER BY "${sortCol}" DESC`;
🔐 Row-Level Security — Multi-Tenant Protection

PostgreSQL RLS — Each User Sees Only Their Own Data, Enforced at DB Level

RLS يُضيف Policy على مستوى الـ Database — حتى لو الكود بُه Bug يُرجع بيانات User آخر، الـ Database تمنعها تلقائياً.

Row-Level Security — E-Commerce Multi-Tenant
-- 1: Enable RLS ALTER TABLE orders ENABLE ROW LEVEL SECURITY; ALTER TABLE order_items ENABLE ROW LEVEL SECURITY; -- 2: Create Policies CREATE POLICY orders_isolation ON orders FOR ALL USING (user_id = current_setting('app.current_user_id')::BIGINT); CREATE POLICY orders_admin ON orders FOR ALL USING (current_setting('app.current_role') = 'admin'); -- 3: في الـ App — set User Context per Request // Node.js middleware لضبط context async function withUserContext<T>( userId: number, role: string, fn: (client: PoolClient) => Promise<T> ): Promise<T> { const client = await pool.connect(); try { await client.query(` SET LOCAL app.current_user_id = ${userId}; SET LOCAL app.current_role = '${role}'; `); return await fn(client); } finally { client.release(); } } -- الآن: SELECT * FROM orders يُرجع أوامر هذا الـ User فقط -- حتى لو قال الكود SELECT * FROM orders بدون WHERE
👥 Roles — Least Privilege Principle

PostgreSQL Roles — App User, Read-Only, Admin, Least Privilege

Database Roles Setup — Least Privilege
-- مبدأ أساسي: كل Role يحصل على الحد الأدنى من الـ Permissions -- App Role: INSERT/SELECT/UPDATE (لا DELETE، لا DDL) CREATE ROLE app_user LOGIN PASSWORD 'strong_app_pass'; GRANT CONNECT ON DATABASE ecommerce TO app_user; GRANT USAGE ON SCHEMA public TO app_user; GRANT SELECT, INSERT, UPDATE ON ALL TABLES IN SCHEMA public TO app_user; GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO app_user; -- لا DELETE! حماية من bugs والـ App لا تحذف سجلات مباشرة -- Analytics / BI Role: Read-Only CREATE ROLE analytics_user LOGIN PASSWORD 'analytics_pass'; GRANT CONNECT ON DATABASE ecommerce TO analytics_user; GRANT SELECT ON ALL TABLES IN SCHEMA public TO analytics_user; -- Read-Only: يرى كل شيء لكن لا يُعدّل شيئاً -- Migration Role: DDL فقط CREATE ROLE migrator LOGIN PASSWORD 'migrator_pass'; GRANT ALL ON SCHEMA public TO migrator; -- يُستخدم فقط وقت الـ deployments، لا في الـ Runtime -- لا تشغّل الـ App بـ superuser! -- postgresql.conf: -- ssl = on -- password_encryption = scram-sha-256

📋 Audit Logging وBackup Strategy

Who Did What When — Audit Trail & 3-2-1 Backup Rule

Audit Trigger — تتبّع كل التعديلات
-- Audit Log Table CREATE TABLE audit_log ( id BIGSERIAL PRIMARY KEY, table_name TEXT NOT NULL, operation TEXT NOT NULL, -- INSERT/UPDATE/DELETE old_data JSONB, new_data JSONB, changed_by TEXT DEFAULT current_user, app_user_id BIGINT, changed_at TIMESTAMPTZ DEFAULT NOW() ); -- Audit Trigger Function CREATE OR REPLACE FUNCTION audit_trigger() RETURNS TRIGGER LANGUAGE plpgsql AS $$ BEGIN INSERT INTO audit_log(table_name, operation, old_data, new_data, app_user_id) VALUES( TG_TABLE_NAME, TG_OP, CASE WHEN TG_OP = 'INSERT' THEN NULL ELSE row_to_json(OLD) END, CASE WHEN TG_OP = 'DELETE' THEN NULL ELSE row_to_json(NEW) END, current_setting('app.current_user_id', true)::BIGINT ); RETURN NEW; END; $$; -- Attach to sensitive tables CREATE TRIGGER audit_orders AFTER INSERT OR UPDATE OR DELETE ON orders FOR EACH ROW EXECUTE FUNCTION audit_trigger(); -- Backup: 3-2-1 Rule -- 3 نسخ، 2 وسائط مختلفة، 1 خارج الموقع -- pg_dump --format=custom --compress=9 -d ecommerce > backup.dump -- pg_basebackup للـ Physical Backup -- WAL Archiving للـ Point-in-Time Recovery
  • استخدم Parameterized Queries دائماً — لا String Concatenation أبداً
  • فعّل Row-Level Security للـ Multi-Tenant Apps
  • أنشئ Roles منفصلة: App، Analytics، Migration — Least Privilege
  • شغّل SSL/TLS دائماً بين الـ App وقاعدة البيانات
  • Hash الـ Passwords بـ bcrypt (cost ≥ 12) — لا تُخزّن Plain Text أبداً
  • فعّل Audit Logging على الجداول الحساسة
  • لا تشغّل الـ App بـ PostgreSQL superuser
  • لا تضع كلمات المرور في الكود — استخدم Environment Variables
