🎤 Interview Questions الحقيقية

Most Asked Database Interview Questions at Top Egyptian Tech Companies

١. ما الفرق بين INNER JOIN وLEFT JOIN؟ ومتى تستخدم كل منهم؟
INNER JOIN: يُرجع الصفوف الموجودة في كلا الجدولين فقط. LEFT JOIN: يُرجع كل صفوف اليسار + المطابق من اليمين (NULL إذا لا يوجد). استخدم INNER عندما تريد البيانات المكتملة فقط (Orders مع Products موجودة). استخدم LEFT عندما تريد كل الكيانات بغض النظر (كل Users حتى اللي لم يطلبوا بعد).
٢. كيف تُحسّن Query بطيئة؟
الخطوات: (1) EXPLAIN ANALYZE لرؤية خطة التنفيذ. (2) ابحث عن Seq Scan على جداول كبيرة — ضع Index. (3) تحقق من ESR Rule للـ Composite Index. (4) تجنّب Functions على الـ Columns في WHERE. (5) راجع الـ Joins — هل الـ Cardinality صحيح؟ (6) إذا الـ Query لإحصاءات → Materialized View.
٣. ما ACID وما أهميته العملية؟
Atomicity: كل الـ Transaction ينجح أو يُلغى. Consistency: قاعدة البيانات تبقى في حالة صالحة. Isolation: Transactions المتزامنة لا تؤثر على بعض. Durability: بعد COMMIT، البيانات محفوظة حتى عند انقطاع الكهرباء. أهمية عملية: في E-Commerce، تحويل مبلغ بين حسابين يجب أن يكون Atomic — إذا فشل الجانب الثاني، يُلغى الأول أيضاً.
٤. متى تستخدم NoSQL بدلاً من Relational Database؟
NoSQL عندما: (1) Schema غير منتظم يتغيّر كثيراً (Redis للـ Cache، MongoDB للـ Product Catalog). (2) Write Throughput ضخم جداً يتجاوز Vertical Scale (Cassandra للـ IoT). (3) Latency أقل من 1ms ضروري (Redis). (4) البيانات طبيعتها Graph (Neo4j). أما Banking وE-Commerce Orders و ERP → PostgreSQL دائماً.
٥. ما الفرق بين Clustered Index وNon-Clustered Index؟
Clustered Index (في PostgreSQL يُسمى Heap أو Clustered بالـ command): يُرتّب البيانات فيزيائياً على الـ disk بنفس ترتيب الـ Index — لكل جدول واحد فقط (PK عادةً). Non-Clustered (Secondary Index): يحتفظ بـ Pointer للـ Row بدون إعادة ترتيب البيانات. في PostgreSQL: الـ Heap غير مُرتّب، كل الـ Indexes غير Clustered بشكل افتراضي — يمكن `CLUSTER table USING index` لكن تحتاج إعادة دورية.
٦. دِيزَاين Database لـ Twitter-like System
users(id, username, email) | tweets(id, user_id, content, created_at) | follows(follower_id, following_id, created_at). للـ Timeline: PostgreSQL مع Partitioning على created_at + GIN Index على content للـ Search. Feed Generation: Pre-compute (Fan-out on Write) للـ Users الصغار, Pull-based للـ Celebrities (+10M followers). Redis Sorted Set للـ Timeline Cache بـ tweet_id كـ Score. Cassandra لـ Activity Logs التاريخية.
