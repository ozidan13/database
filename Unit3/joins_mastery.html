الـ JOINs — إتقان الربط

SQL JOINs Mastery — Every Type, Every Edge Case

البيانات الحقيقية مش في جدول واحد. الـ JOINs هي اللي بتخليك تجمّع القصة الكاملة من الأجزاء المختلفة.

🤔 لماذا نحتاج الـ JOINs؟

Why JOINs Exist — The Distributed Data Problem

في قاعدة البيانات المنظّمة، البيانات المتعلقة محفوظة في جداول منفصلة. لماذا؟ عشان نتجنب التكرار (Redundancy) ونطبّق مبادئ الـ Normalization. لكن ده معناه إننا لما نعرض بيانات كاملة نحتاج نجمّع من جداول متعددة.

📊 مثال عملي — لماذا الفصل ضروري؟

لو خزّنا اسم القسم في كل صف موظف:
◆ موظفين في IT → "Information Technology" متكررة 50 مرة
◆ لو اتغيّر اسم القسم → لازم تعدّل 50 صف
◆ لو نسيت صف → بيانات متناقضة!

الحل: اسم القسم مرة واحدة في DEPARTMENT، والموظف عنده FK → dept_id. الـ JOIN بيجيب الاتنين مع بعض عند الاستعلام.

INNER JOIN — Result
SELECT e.name, d.dept_name FROM employee e INNER JOIN department d ON e.dept_id = d.dept_id; -- نتيجة: -- أحمد | IT -- فاطمة | HR -- محمد | IT -- نور → مش موجود! (dept_id = NULL → مفيش تطابق) -- Finance → مش موجود! (مفيش موظفين فيه) -- JOIN vs INNER JOIN: نفس الشيء. JOIN = INNER JOIN بالافتراضي SELECT * FROM a JOIN b ON a.id = b.id; -- نفس INNER JOIN -- USING بدل ON لو اسم العمود نفسه في الجدولين SELECT * FROM employee JOIN department USING (dept_id);
⬅️ LEFT OUTER JOIN
LEFT JOIN
كل الجدول الأيسر + المتطابق من الأيمن
All rows from LEFT table, NULLs where no match in RIGHT

الأكثر استخداماً — بيحتفظ بكل صفوف الجدول الأيسر (المرجع الأساسي). لو ما فيش تطابق في الجدول الأيمن، الأعمدة بتجي NULL.

LEFT JOIN — With Results
-- كل الأقسام، حتى اللي مفيهاش موظفين SELECT d.dept_name, COUNT(e.emp_id) AS headcount -- COUNT على FK مش * → يتجاهل NULLs FROM department d LEFT JOIN employee e ON d.dept_id = e.dept_id GROUP BY d.dept_id, d.dept_name ORDER BY headcount DESC; -- IT | 2 -- HR | 1 -- Finance | 0 ← LEFT JOIN يحتفظ بيه مع COUNT=0 -- ══ LEFT JOIN للإيجاد اللي مالوش مقابل (Anti-Join) ══ -- الأقسام اللي مفيهاش موظفين على الإطلاق SELECT d.dept_name FROM department d LEFT JOIN employee e ON d.dept_id = e.dept_id WHERE e.emp_id IS NULL; -- شرط Anti-Join -- Finance -- ══ تأثير WHERE على LEFT JOIN ══ SELECT d.dept_name, e.name FROM department d LEFT JOIN employee e ON d.dept_id = e.dept_id WHERE e.salary > 10000; -- ⚠️ بيحوّل LEFT JOIN لـ INNER JOIN فعلياً! -- لأن salary=NULL للصفوف اللي مالهاش match → WHERE يستبعدهم! -- الحل: ضع الشرط في ON بدل WHERE SELECT d.dept_name, e.name FROM department d LEFT JOIN employee e ON d.dept_id = e.dept_id AND e.salary > 10000; -- ✅ الشرط في ON يُطبَّق قبل الـ join
➡️ RIGHT OUTER JOIN
RIGHT JOIN
كل الجدول الأيمن + المتطابق من الأيسر
Mirror of LEFT JOIN — rare in practice

مرآة الـ LEFT JOIN. نادراً ما تُستخدم في الواقع لأن ممكن دايماً تعيد ترتيب الجداول وتعمل LEFT JOIN بدلاً منها. لكن بيفيد لما تكون بتضيف جدول لـ query موجودة.

RIGHT JOIN — vs Equivalent LEFT JOIN
-- RIGHT JOIN (أقل شيوعاً) SELECT e.name, d.dept_name FROM employee e RIGHT JOIN department d ON e.dept_id = d.dept_id; -- مكافئ لـ LEFT JOIN مع عكس الترتيب SELECT e.name, d.dept_name FROM department d LEFT JOIN employee e ON e.dept_id = d.dept_id;
🔄 FULL OUTER JOIN
FULL OUTER JOIN
كل الصفوف من الجانبين — النقص يُملأ بـ NULL
All rows from both tables, NULLs on non-matching sides

مزيج LEFT + RIGHT. مفيد في مقارنة قوائم أو Reconciliation.

FULL OUTER JOIN — Reconciliation Use Case
-- مقارنة: الموظفين اللي مالهمش قسم، والأقسام اللي مالهاش موظفين SELECT COALESCE(e.name, 'بدون موظف') AS employee, COALESCE(d.dept_name, 'موظف بلا قسم') AS department FROM employee e FULL JOIN department d ON e.dept_id = d.dept_id; -- FULL JOIN Anti-Join: الصفوف اللي مالهاش مقابل في أي جانب SELECT e.name, d.dept_name FROM employee e FULL JOIN department d ON e.dept_id = d.dept_id WHERE e.emp_id IS NULL OR d.dept_id IS NULL; -- الأقسام الفارغة + الموظفين بلا قسم
🔁 SELF JOIN — الجدول مع نفسه
SELF JOIN
نفس الجدول يُدمج مع نفسه — للعلاقات الهرمية
Table joined with itself for hierarchical relationships

بيُستخدم لما في علاقة هرمية داخل نفس الجدول — مثل: موظف وله مشرف (وهو كمان موظف)، تصنيف فرعي ومن تصنيف أب.

SELF JOIN — Manager Hierarchy
-- EMPLOYEE(emp_id, name, salary, supervisor_id FK→emp_id) -- ══ Unary 1:N — موظف مع مشرفه ══ SELECT e.name AS employee_name, m.name AS manager_name, m.department AS manager_dept FROM employee e LEFT JOIN employee m ON e.supervisor_id = m.emp_id -- LEFT JOIN عشان نشوف الموظفين اللي مالهمش مشرف (المدير التنفيذي) ORDER BY m.name NULLS LAST; -- ══ مقارنة الرواتب بين الزملاء ══ SELECT a.name AS emp1, b.name AS emp2, a.salary - b.salary AS salary_diff FROM employee a JOIN employee b ON a.dept_id = b.dept_id -- نفس القسم AND a.emp_id < b.emp_id -- < بدل != عشان مانكررش الأزواج ORDER BY ABS(a.salary - b.salary) DESC;
✖️ CROSS JOIN — الضرب الديكارتي
CROSS JOIN
كل صف مع كل صف — Cartesian Product
n × m rows — use carefully!

كل صف في الجدول الأول بيُدمج مع كل صف في الجدول الثاني. 5 صفوف × 4 صفوف = 20 صف. مفيد في توليد تركيبات أو جداول مرجعية.

CROSS JOIN — Use Cases
-- ══ توليد جدول أحجام × ألوان ══ SELECT s.size_name, c.color_name FROM sizes s CROSS JOIN colors c; -- 3 أحجام × 4 ألوان = 12 تركيبة -- ══ مقارنة كل موظف بكل وظيفة (لبناء توصيات) ══ SELECT e.emp_id, j.job_title, e.skills FROM employee e CROSS JOIN job_openings j WHERE e.dept_id != j.dept_id; -- استثني نفس القسم -- ⚠️ تحذير: جدولين كبيرين × = CROSS JOIN بطيء جداً -- 10,000 × 10,000 = 100,000,000 صف!
🔀 LATERAL JOIN — الـ JOIN الديناميكي
LATERAL JOIN
Subquery بتستخدم أعمدة الجدول السابق
Correlated Subquery as a JOIN — PostgreSQL powerful feature

الـ LATERAL بيخلّي الـ Subquery في FROM تستخدم أعمدة من الجداول السابقة في نفس الـ FROM. مفيد جداً لجلب "آخر N عناصر" لكل صف.

LATERAL JOIN — Top N per Group
-- آخر 3 أوردرات لكل عميل SELECT c.full_name, latest.order_id, latest.total_amount, latest.order_date FROM customer c LEFT JOIN LATERAL ( SELECT order_id, total_amount, order_date FROM orders WHERE customer_id = c.customer_id -- يستخدم c من الـ outer! ORDER BY order_date DESC LIMIT 3 ) latest ON TRUE WHERE c.is_active = TRUE ORDER BY c.full_name, latest.order_date DESC;

Chaining Multiple JOINs — E-Commerce Complete Report

⚠️ الأخطاء الشائعة في الـ JOINs

❌ نسيان JOIN condition: ينتج Cartesian Product بدون تحذير في بعض قواعد البيانات القديمة.
❌ WHERE بدل ON للـ LEFT JOIN: يحوّله لـ INNER JOIN فعلياً (شرحنا قبل كده).
❌ Ambiguous column: لو عمودان بنفس الاسم في جدولين ← لازم تحدد tb.col.
❌ استخدام COUNT(*) مع LEFT JOIN: بيعدّ NULLs! استخدم COUNT(e.emp_id).
❌ أكتر من JOIN على نفس الجدول بدون alias مختلف ← Error.

Multi-Table JOIN — E-Commerce Full Report
-- تقرير: اسم العميل، اسم المنتج، الكمية، السعر الفعلي، الفئة، اسم المدينة -- الجداول: customer → orders → order_item → product → category, customer → address → city SELECT c.full_name AS customer, ct.city_name AS city, o.order_id, TO_CHAR(o.order_date, 'YYYY-MM-DD') AS order_date, p.product_name, cat.category_name, oi.quantity, oi.unit_price, oi.quantity * oi.unit_price AS line_total FROM customer c JOIN address a ON c.customer_id = a.customer_id AND a.is_default = TRUE JOIN city ct ON a.city_id = ct.city_id JOIN orders o ON c.customer_id = o.customer_id JOIN order_item oi ON o.order_id = oi.order_id JOIN product p ON oi.product_id = p.product_id JOIN category cat ON p.category_id = cat.category_id WHERE o.status = 'delivered' AND o.order_date >= '2024-01-01' ORDER BY o.order_date DESC, c.full_name;

  • Aاجيب اسم كل طبيب، تخصصه، اسم قسمه، وعدد مواعيده الـ completed — بما فيهم الأطباء اللي مالهمش مواعيد.
  • Bاجيب الأقسام اللي مالهاش أطباء (Anti-Join).
  • Cلكل طبيب، اجيب أحدث 2 تشخيص (diagnosis) قالهم — استخدم LATERAL.
Hospital Reports — Solutions
-- A: أطباء مع عدد مواعيدهم (LEFT لإشمال بدون مواعيد) SELECT d.full_name, d.specialty, dep.dept_name, COUNT(a.appt_id) AS completed_appts FROM doctor d JOIN department dep USING (dept_id) LEFT JOIN appointment a ON d.doctor_id = a.doctor_id AND a.status = 'completed' -- الشرط في ON لا WHERE GROUP BY d.doctor_id, d.full_name, d.specialty, dep.dept_name ORDER BY completed_appts DESC; -- B: أقسام بلا أطباء (Anti-Join) SELECT dep.dept_name FROM department dep LEFT JOIN doctor d USING (dept_id) WHERE d.doctor_id IS NULL; -- C: LATERAL — آخر 2 تشخيص لكل طبيب SELECT d.full_name, latest.diagnosis, latest.scheduled_at FROM doctor d LEFT JOIN LATERAL ( SELECT diagnosis, scheduled_at FROM appointment WHERE doctor_id = d.doctor_id AND diagnosis IS NOT NULL ORDER BY scheduled_at DESC LIMIT 2 ) latest ON TRUE ORDER BY d.full_name;
