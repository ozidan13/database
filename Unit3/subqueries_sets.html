🗂️ أنواع الـ Subqueries

Subquery Classification — By Return Type and Position

الـ Subquery هو SELECT داخل SELECT. بيتصنّف بطريقتين: حسب ما بيرجعه، وحسب مكانه.

حسب ما بيرجعه

Scalar

بيرجع قيمة واحدة (صف واحد، عمود واحد). بيُستخدم في SELECT وWHERE مع =، <، >

SELECT (SELECT MAX(salary) FROM employee)

Row

بيرجع صف كامل (عدة أعمدة، صف واحد). نادرة الاستخدام.

WHERE (id, type) = (SELECT ...)

Table

بيرجع عدة صفوف وأعمدة. بيُستخدم في FROM و IN و EXISTS.

FROM (SELECT ...) AS sub

Column

بيرجع عمود واحد، صفوف متعددة. بيُستخدم مع IN وNOT IN.

WHERE id IN (SELECT id FROM ...)

حسب مكانه في الاستعلام

WHERE Subquery

للفلترة بناءً على نتيجة استعلام آخر: أجيب الموظفين اللي راتبهم أعلى من متوسط شركتهم.

FROM Subquery (Derived Table)

بيستخدم نتيجة SELECT كجدول مؤقت. مش محتاج تعمل CTE أو View مستقل.

SELECT Subquery (Scalar)

بيضيف عمود محسوب في النتيجة. لكل صف بيُنفَّذ مرة.

HAVING Subquery

مقارنة نتيجة Aggregate بـ Aggregate من جدول آخر.

🎯 Subqueries في WHERE

WHERE Subqueries — Scalar, IN, NOT IN, Comparison

WHERE Subqueries — All Patterns
-- ══ Scalar Subquery مع مقارنة ══ -- الموظفين اللي راتبهم أعلى من المتوسط SELECT full_name, salary FROM employee WHERE salary > (SELECT AVG(salary) FROM employee); -- ══ IN — قائمة قيم من Subquery ══ -- المنتجات اللي اتطلبت في أوردر معين SELECT product_name FROM product WHERE product_id IN ( SELECT product_id FROM order_item WHERE order_id = 500 ); -- ══ NOT IN — الغائبين ══ -- العملاء اللي ما عملوش أي أوردر خالص SELECT full_name, email FROM customer WHERE customer_id NOT IN ( SELECT customer_id FROM orders ); -- ⚠️ خطر NOT IN مع NULLs! -- لو في NULL واحدة في الـ Subquery نتيجة NOT IN بتبقى فاضية! -- SELECT * FROM a WHERE id NOT IN (1, 2, NULL) → 0 results! -- الحل: إضافة WHERE customer_id IS NOT NULL في الـ Subquery SELECT full_name FROM customer WHERE customer_id NOT IN ( SELECT customer_id FROM orders WHERE customer_id IS NOT NULL -- ✅ آمن ); -- الأفضل: استخدم NOT EXISTS أو LEFT JOIN Anti-Join -- ══ HAVING Subquery ══ -- الأقسام اللي متوسط راتبها أعلى من متوسط قسم IT SELECT dept_id, AVG(salary) FROM employee GROUP BY dept_id HAVING AVG(salary) > ( SELECT AVG(salary) FROM employee WHERE dept_id = (SELECT dept_id FROM department WHERE dept_name = 'IT') );
📋 Subqueries في FROM — Derived Tables

FROM Subqueries — Inline Views and Multi-Level Aggregation

لما بتحط Subquery في FROM، بتتعامل معاها كجدول مؤقت اسمه Derived Table أو Inline View. مفيد للحسابات على مستويين.

FROM Subqueries — Derived Tables
-- ══ متوسط المتوسطات — مستويان من الـ Aggregation ══ -- ما متوسط متوسطات رواتب الأقسام؟ (مش نفس المتوسط الكلي) SELECT AVG(dept_avg_salary) AS avg_of_avgs FROM ( SELECT dept_id, AVG(salary) AS dept_avg_salary FROM employee GROUP BY dept_id ) AS dept_averages; -- alias إلزامي في PostgreSQL -- ══ Pagination مع بيانات مسبقة ══ SELECT r.*, total_count FROM ( SELECT *, COUNT(*) OVER () AS total_count FROM product WHERE is_active = TRUE ) r ORDER BY r.product_id LIMIT 20 OFFSET 0; -- ══ تصنيف الـ Ranking بعد حسابه ══ SELECT * FROM ( SELECT customer_id, full_name, total_spent, RANK() OVER (ORDER BY total_spent DESC) AS rank FROM ( SELECT c.customer_id, c.full_name, SUM(o.total_amount) AS total_spent FROM customer c JOIN orders o USING (customer_id) GROUP BY c.customer_id, c.full_name ) AS spending ) AS ranked WHERE rank <= 10; -- أعلى 10 عملاء
📊 Subqueries في SELECT — أعمدة محسوبة

SELECT Subqueries — Inline Computed Columns

SELECT Subqueries
-- لكل موظف: راتبه، متوسط راتب قسمه، الفرق SELECT full_name, salary, dept_id, (SELECT ROUND(AVG(e2.salary),2) FROM employee e2 WHERE e2.dept_id = e1.dept_id) AS dept_avg, salary - (SELECT AVG(e2.salary) FROM employee e2 WHERE e2.dept_id = e1.dept_id) AS diff_from_avg FROM employee e1 ORDER BY dept_id, diff_from_avg DESC; -- ⚠️ SELECT Subquery بتُنفَّذ لكل صف → بطيئة على بيانات كبيرة -- الأفضل: استخدم Window Function أو JOIN مع Derived Table
🔄 Correlated Subqueries — الاستعلام المترابط

Correlated Subqueries — When Inner Depends on Outer

في الـ Correlated Subquery، الاستعلام الداخلي يستخدم عموداً من الاستعلام الخارجي. يعني بيُنفَّذ مرة لكل صف في الـ Outer Query.

💡 متى تكون Correlated؟

لو قدرت تشغّل الـ Subquery منفردة بدون الـ Outer Query → مش Correlated.
لو ما قدرتش (بتحتاج قيمة من الصف الحالي في Outer) → Correlated.

Correlated Subqueries — Patterns
-- ══ الموظفين اللي راتبهم أعلى من متوسط قسمهم ══ SELECT full_name, salary, dept_id FROM employee e1 WHERE salary > ( SELECT AVG(salary) FROM employee e2 WHERE e2.dept_id = e1.dept_id -- ← يستخدم الصف الحالي من Outer ); -- ══ لكل منتج: هل سعره أعلى من متوسط فئته؟ ══ SELECT product_name, price, CASE WHEN price > ( SELECT AVG(p2.price) FROM product p2 WHERE p2.category_id = p1.category_id ) THEN 'أعلى من المتوسط' ELSE 'أقل من أو يساوي' END AS price_position FROM product p1; -- ⚠️ بطيئة على بيانات كبيرة (تُنفَّذ N مرة) -- الأفضل: Window Function: AVG(price) OVER (PARTITION BY category_id)
✅ EXISTS وNOT EXISTS

EXISTS — The Existence Check — Fastest Anti-Join

الـ EXISTS بيتحقق إذا كانت الـ Subquery ترجع أي صف — مش بتهمه القيمة. بمجرد ما يلاقي صف واحد يوقف البحث (Short-circuit). أسرع من IN في معظم الحالات.

EXISTS vs NOT EXISTS vs IN
-- ══ EXISTS — بديل أفضل من IN في كتير حالات ══ -- العملاء اللي عندهم أوردر على الأقل واحد SELECT full_name FROM customer c WHERE EXISTS ( SELECT 1 -- مش مهم اخترت إيه — EXISTS بيشوف الوجود بس FROM orders o WHERE o.customer_id = c.customer_id ); -- ══ NOT EXISTS — أفضل بديل لـ NOT IN (آمن مع NULLs) ══ -- العملاء اللي ما عملوش أي أوردر SELECT full_name FROM customer c WHERE NOT EXISTS ( SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id ); -- ══ مقارنة IN vs EXISTS ══ -- IN: يجيب كل القيمة ثم يقارن WHERE customer_id IN (SELECT customer_id FROM orders); -- EXISTS: بيوقف لما يلاقي أول match → أسرع في قوائم كبيرة WHERE EXISTS (SELECT 1 FROM orders WHERE customer_id = c.customer_id); -- ══ EXISTS في UPDATE ══ -- خلّي العملاء النشطين VIP لو عندهم أكتر من 5 أوردرات UPDATE customer c SET is_vip = TRUE WHERE EXISTS ( SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id HAVING COUNT(*) > 5 );

🔢 ALL و ANY و SOME

ALL, ANY, SOME — Quantified Comparisons

ALL / ANY / SOME
-- ══ ALL: يجب أن يكون الشرط صحيحاً مع كل القيم ══ -- الموظفين اللي راتبهم أعلى من كل موظفي قسم HR SELECT full_name, salary FROM employee WHERE salary > ALL ( SELECT salary FROM employee WHERE dept_id = (SELECT dept_id FROM department WHERE dept_name = 'HR') ); -- مكافئ لـ: salary > MAX(salary from HR) -- ══ ANY/SOME: يجب أن يكون الشرط صحيحاً مع أي قيمة واحدة ══ -- المنتجات اللي سعرها أقل من أي منتج في فئة Electronics SELECT product_name, price FROM product WHERE category_id != 1 -- مش Electronics AND price < ANY ( SELECT price FROM product WHERE category_id = 1 ); -- مكافئ لـ: price < MAX(price from Electronics) -- SOME = ANY — نفس تماماً (alias) WHERE salary > SOME (SELECT salary FROM employee WHERE dept_id = 2);

UNION, INTERSECT, EXCEPT — Combining Query Results

بيدمجوا نتائج عدة SELECT. الشرط: نفس عدد الأعمدة ونفس أنواعها (أو متوافقة).

UNION

دمج نتيجتين — إزالة التكرار
UNION ALL: يحتفظ بالتكرار (أسرع)

INTERSECT

الصفوف المشتركة بين النتيجتين فقط

EXCEPT

الصفوف في الأولى وغير موجودة في الثانية
مثل Anti-Join

Set Operations — Real Examples
-- ══ UNION — قائمة موحّدة من مصدرين ══ -- كل الـ emails في النظام (موظفين + عملاء) SELECT email, 'employee' AS source FROM employee WHERE email IS NOT NULL UNION SELECT email, 'customer' AS source FROM customer WHERE email IS NOT NULL ORDER BY email; -- UNION يحذف التكرار (نفس email في الجدولين يظهر مرة) -- UNION ALL (أسرع — لا يزيل التكرار) SELECT product_id, 'sale' AS event FROM order_item WHERE order_date = CURRENT_DATE UNION ALL SELECT product_id, 'return' FROM return_item WHERE return_date = CURRENT_DATE ORDER BY product_id; -- ══ INTERSECT — الموظفين اللي هم كمان عملاء ══ SELECT national_id FROM employee INTERSECT SELECT national_id FROM customer; -- ══ EXCEPT — العملاء اللي مش موظفين ══ SELECT national_id FROM customer EXCEPT SELECT national_id FROM employee; -- ══ Set Operations مع أعمدة مختلفة (لكن بنفس العدد والنوع) ══ SELECT doctor_id AS person_id, full_name, 'doctor' AS role FROM doctor UNION ALL SELECT patient_id, full_name, 'patient' FROM patient ORDER BY full_name;

🛒 تمرين: تحليل متقدم لـ E-Commerce

الجداول

customer, orders(customer_id, total_amount, status), product(product_id, price, category_id), order_item(order_id, product_id, quantity)

  • Aباستخدام NOT EXISTS، اجيب العملاء اللي ما اشتروش منتج من فئة "Electronics".
  • Bاجيب المنتجات اللي سعرها أعلى من كل منتجات فئة "Accessories" — استخدم ALL.
  • Cعمل query يجيب: اسم العميل + إجمالي مشترياته + رتبته بين كل العملاء — استخدم Derived Table.
E-Commerce Advanced Queries — Solutions
-- A: NOT EXISTS — لم يشتر من Electronics SELECT c.full_name, c.email FROM customer c WHERE NOT EXISTS ( SELECT 1 FROM orders o JOIN order_item oi USING (order_id) JOIN product p USING (product_id) JOIN category cat USING (category_id) WHERE o.customer_id = c.customer_id AND cat.category_name = 'Electronics' ); -- B: ALL — سعر أعلى من كل Accessories SELECT product_name, price FROM product p JOIN category c USING (category_id) WHERE c.category_name != 'Accessories' AND price > ALL ( SELECT p2.price FROM product p2 JOIN category c2 USING (category_id) WHERE c2.category_name = 'Accessories' ); -- C: Derived Table — ترتيب العملاء حسب الإنفاق SELECT ranked.full_name, ranked.total_spent, ranked.rank FROM ( SELECT c.full_name, COALESCE(SUM(o.total_amount), 0) AS total_spent, RANK() OVER ( ORDER BY COALESCE(SUM(o.total_amount), 0) DESC ) AS rank FROM customer c LEFT JOIN orders o USING (customer_id) WHERE (o.status = 'delivered' OR o.status IS NULL) GROUP BY c.customer_id, c.full_name ) AS ranked ORDER BY ranked.rank;

٢. الفرق بين UNION وUNION ALL؟
مفيش فرق
UNION يزيل التكرار (أبطأ). UNION ALL يحتفظ بالكل (أسرع).
UNION ALL بيدمج الجداول كلياً
٣. ليه EXISTS أسرع من IN في القوائم الكبيرة؟
لأن EXISTS بيستخدم Index دايماً
EXISTS بيوقف (Short-circuit) بمجرد ما يلاقي أول match. IN بيجيب كل القيمة ثم يقارن.
IN أسرع وليس EXISTS
٤. ما الفرق بين salary > ALL(...) وsalary > ANY(...)؟
مفيش فرق
ALL: يجب أن يكون الشرط صحيحاً مع كل القيم (مكافئ لـ > MAX). ANY: يكفي أن يكون صحيحاً مع قيمة واحدة (مكافئ لـ > MIN في هذه الحالة).
ALL للأرقام و ANY للنصوص
