📋 ما هو DDL؟

DDL — Data Definition Language in the SQL Family

الـ SQL مش جملة واحدة — هو عائلة من اللغات الفرعية، كل واحدة لها هدف مختلف:

DDL — Definition

بيعرّف هيكل البيانات. الأوامر: CREATE, ALTER, DROP, TRUNCATE, RENAME, COMMENT.

Auto-commits — مش ممكن ROLLBACK

DML — Manipulation

بيتعامل مع البيانات. الأوامر: SELECT, INSERT, UPDATE, DELETE, MERGE.

Transactional — ممكن ROLLBACK

DCL — Control

بيتحكم في الصلاحيات. الأوامر: GRANT, REVOKE.

Security Layer

TCL — Transaction

بيتحكم في الـ Transactions. الأوامر: COMMIT, ROLLBACK, SAVEPOINT.

ACID Compliance

🎯 لماذا DDL مهم للـ Database Developer؟

أي تغيير في الـ Schema في Production يحتاج تخطيط دقيق. DELETE بالغلط → ROLLBACK. DROP TABLE بالغلط → بيانات ضايعة للأبد. الـ DDL هو اللي بيحدّد كل قيود الـ Business Logic على مستوى الـ Database.

🔤 أنواع البيانات في PostgreSQL

PostgreSQL Data Types — Choosing the Right Type

اختيار نوع البيانات الغلط بيأثر على الـ Performance والـ Storage والـ Data Integrity. سيبنا نتعلم كل نوع ومتى نستخدمه:

النصوص والحروف

النوعالوصفمتى تستخدمه؟مثال
CHAR(n)نص بطول ثابت — بيُكمَّل بمسافاترموز ثابتة الطول: رقم قومي، VIN، كود ISOCHAR(14) للرقم القومي
VARCHAR(n)نص بطول متغيّر بحد أقصىأسماء، emails، عناوينVARCHAR(100) للاسم
TEXTنص بدون حد أقصىمحتوى طويل: مقالات، ملاحظات، أوصافTEXT للـ bio أو description
💡 CHAR vs VARCHAR — الفرق العملي

CHAR(5) + قيمة "Hi" → تُخزَّن كـ "Hi " (3 مسافات). لما بتعمل مقارنة، PostgreSQL بيتجاهل الـ trailing spaces.
VARCHAR(5) + قيمة "Hi" → تُخزَّن كـ "Hi" (2 bytes بس).
الخلاصة: استخدم CHAR فقط للبيانات اللي طولها ثابت دايماً (رقم قومي = 14 حرف دايماً). لكل حاجة تانية استخدم VARCHAR أو TEXT.

الأرقام

النوعالنطاقالحجممتى تستخدمه؟
SMALLINT-32,768 لـ 32,7672 bytesتقييمات (1-5)، عمر، أعداد صغيرة
INTEGER / INT-2.1B لـ 2.1B4 bytesمعظم الـ IDs والأرقام العادية
BIGINT-9.2Q لـ 9.2Q8 bytesIDs في أنظمة كبيرة، timestamps
SERIAL1 لـ ~2.1B أو ~9.2Q4/8 bytesAuto-increment PK (مش حقيقي في PG 10+)
DECIMAL(p,s)حسب precisionvariableمبالغ مالية — دقيق
NUMERIC(p,s)حسب precisionvariableنفس DECIMAL
REAL / FLOAT4~6 decimal digits4 bytesحسابات علمية — مش مالية!
DOUBLE PRECISION~15 decimal digits8 bytesحسابات علمية أكثر دقة
⚠️ لا تستخدم FLOAT/REAL للمبالغ المالية!

FLOAT(1.1) + FLOAT(2.2) قد تساوي 3.3000000000000003 بسبب IEEE 754 floating point.
استخدم DECIMAL(15,2) أو NUMERIC(15,2) للمبالغ المالية. دايماً.

التاريخ والوقت

النوعالوصفمتى تستخدمه؟
DATEتاريخ فقط (YYYY-MM-DD) — 4 bytesتواريخ الميلاد، تواريخ الأحداث بدون وقت
TIMEوقت فقط — 8 bytesأوقات الجداول، ساعات العمل
TIMESTAMPتاريخ + وقت (محلي) — 8 bytesوقت الإنشاء، الأحداث في نفس timezone
TIMESTAMPTZتاريخ + وقت + timezone — 8 bytesتطبيقات متعددة الـ timezones — الأفضل دايماً
INTERVALفترة زمنيةمدة عقد، فترة إجازة

🏗️ CREATE TABLE — بالتفصيل

CREATE TABLE — Full Syntax and Options

CREATE TABLE Complete Syntax
-- الشكل الكامل لـ CREATE TABLE CREATE TABLE [IF NOT EXISTS] table_name ( -- عمود بسيط column_name data_type [column_constraints], -- column constraints (بجانب التعريف) col1 INT PRIMARY KEY, col2 VARCHAR(100) NOT NULL UNIQUE, col3 DECIMAL(10,2) DEFAULT 0.00 CHECK (col3 >= 0), col4 TIMESTAMPTZ NOT NULL DEFAULT NOW(), -- table constraints (منفصلة في الأسفل) CONSTRAINT pk_name PRIMARY KEY (col1, col2), CONSTRAINT uq_email UNIQUE (email), CONSTRAINT fk_dept FOREIGN KEY (dept_id) REFERENCES department(dept_id) ON DELETE RESTRICT ON UPDATE CASCADE, CONSTRAINT chk_salary CHECK (salary > 0 AND salary <= 1000000) ); -- CREATE TABLE IF NOT EXISTS (مش بيدي error لو موجود) CREATE TABLE IF NOT EXISTS logs ( log_id BIGSERIAL PRIMARY KEY, created TIMESTAMPTZ NOT NULL DEFAULT NOW(), message TEXT NOT NULL ); -- CREATE TABLE AS (نسخة من query) CREATE TABLE high_gpa_students AS SELECT * FROM student WHERE gpa > 3.5; -- ⚠️ بينسخ البيانات بدون الـ constraints!
💡 CONSTRAINT بالاسم — ليه مهم؟

لما تعمل CONSTRAINT مسمّاه، لو في violation جاتك رسالة الخطأ بتقولك violates constraint "chk_salary" — بتعرف على طول المشكلة. لو مش مسمّية، PostgreSQL يوّلد اسم تلقائي زي employee_salary_check — أصعب تتعامل معاه في ALTER TABLE.

ALTER TABLE — Every Operation You Need

في Production، مش هتحذف الجدول وتعيده. هتستخدم ALTER TABLE عشان تعدّل الهيكل بأمان. فيه types كتيرة:

ALTER TABLE Reference Guide
-- ══ إضافة عمود ══ ALTER TABLE employee ADD COLUMN middle_name VARCHAR(50); -- nullable جديدة ALTER TABLE employee ADD COLUMN is_remote BOOLEAN NOT NULL DEFAULT FALSE; -- مع DEFAULT يشتغل في Prod -- ⚠️ ADD COLUMN NOT NULL بدون DEFAULT = ERROR في Production لو في data! -- ══ حذف عمود ══ ALTER TABLE employee DROP COLUMN middle_name; ALTER TABLE employee DROP COLUMN old_col CASCADE; -- بيحذف الـ views المعتمدة عليه -- ══ تعديل نوع عمود ══ ALTER TABLE employee ALTER COLUMN phone TYPE VARCHAR(25); -- ممكن لو الـ data قابلة للتحويل ALTER TABLE orders ALTER COLUMN total_amount TYPE DECIMAL(15,4); -- زيادة precision ← آمن -- ══ إضافة / حذف NOT NULL ══ ALTER TABLE student ALTER COLUMN email SET NOT NULL; -- يفشل لو فيه NULLs موجودة ALTER TABLE student ALTER COLUMN phone DROP NOT NULL; -- آمن دايماً -- ══ تعديل DEFAULT ══ ALTER TABLE orders ALTER COLUMN status SET DEFAULT 'pending'; ALTER TABLE orders ALTER COLUMN status DROP DEFAULT; -- ══ إضافة constraint ══ ALTER TABLE employee ADD CONSTRAINT chk_salary CHECK (salary > 0); ALTER TABLE employee ADD CONSTRAINT fk_dept FOREIGN KEY (dept_id) REFERENCES department(dept_id); ALTER TABLE customer ADD CONSTRAINT uq_email UNIQUE (email); -- ══ حذف constraint ══ ALTER TABLE employee DROP CONSTRAINT chk_salary; -- ══ إعادة تسمية ══ ALTER TABLE employee RENAME COLUMN emp_num TO employee_code; ALTER TABLE employee RENAME TO staff;
🏢 في Production — كيف تضيف عمود NOT NULL آمن؟

مش ممكن تضيف عمود NOT NULL مباشرة لو في جدول عنده بيانات:

✅ الطريقة الآمنة:
Step 1: ADD COLUMN new_col TEXT (nullable) ←آمن
Step 2: UPDATE table SET new_col = 'default_value' ← ملا القديم
Step 3: ALTER COLUMN new_col SET NOT NULL ← بعد ما كل حاجة اتملت

DROP vs TRUNCATE vs DELETE — Know the Difference

DROP vs TRUNCATE vs DELETE

Organizing Tables with Schemas

الـ Schema في PostgreSQL هو namespace — بيتيح لك تنظيم الجداول في مجموعات منطقية.

Schema Management
-- إنشاء schema CREATE SCHEMA hr; CREATE SCHEMA sales; CREATE SCHEMA audit; -- إنشاء جدول في schema معين CREATE TABLE hr.employee ( ... ); CREATE TABLE sales.order ( ... ); CREATE TABLE audit.change_log ( ... ); -- تعيين search path (الافتراضي = public) SET search_path = hr, sales, public; -- الاستعلام بدون ذكر الـ schema (يستخدم search_path) SELECT * FROM employee; -- بيدوّر في hr أولاً ثم sales ثم public

المتطلبات

  • Aاكتب كل الجداول بـ DDL مع كل الـ Constraints المناسبة.
  • Bبعد الإنشاء — اكتشفنا إن IBAN لازم يكون exactly 34 حرف. اكتب الـ ALTER TABLE المناسب.
  • Cعايزين نتتبّع تغييرات الرصيد — أضف عمود balance_updated_at.
