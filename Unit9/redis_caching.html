Redis vs Traditional Database — When Sub-Millisecond Latency Matters

PostgreSQL Query وقتها 2-50ms. Redis Operation وقتها <1ms (Microseconds). عندك API يُستدعى 1000 مرة/ثانية وكل استدعاء يتطلب Query ثقيلة — Redis هو الحل.

🟠 Session Storage: JWT Revocation، User Sessions المشتركة عبر Servers
🟡 Rate Limiting: "100 request/minute لكل IP" — بدون DB queries

Redis Data Structures — The Secret Power of Redis

String

أي نص أو رقم أو Binary. حتى 512MB. يدعم INCR/DECR الـ Atomic.

Cache، Counters، Flags، JSON strings

Hash

Map من Field→Value. مثالي للـ Objects — تحديث Field واحد فقط.

User Profile، Product، Session Data

List

LinkedList من Strings. يدعم Push/Pop من الطرفين. Blocking Commands.

Job Queues، Activity Feed، Recent Items

Set

مجموعة Unique Strings. عمليات Union/Intersection/Difference.

Tags، Online Users، Unique Visitors

Sorted Set (ZSet) ⭐

Set مع Score. مُرتّب دائماً. الأقوى في Redis — Range Queries بالـ Score.

Leaderboard، Priority Queue، Trending

Stream

Log-like Data Structure. كل Entry له ID وFields. يدعم Consumer Groups.

Event Sourcing، Message Brokers، Audit Logs

Cache-Aside Pattern — Node.js مع ioredis
import Redis from 'ioredis'; const redis = new Redis({ host: 'localhost', port: 6379 }); async function getProduct(productId: string) { const cacheKey = `product:${productId}`; // 1. Check Cache First const cached = await redis.get(cacheKey); if (cached) { return { source: 'cache', data: JSON.parse(cached) }; } // 2. Cache Miss → Hit DB const product = await db.query( 'SELECT * FROM products WHERE id = $1', [productId] ); if (product) { // 3. Store in Cache with 1hr TTL await redis.setex(cacheKey, 3600, JSON.stringify(product)); } return { source: 'database', data: product }; } // Cache Invalidation — عند UPDATE async function updateProduct(productId: string, updates: any) { await db.query('UPDATE products SET ... WHERE id=$1', [productId]); await redis.del(`product:${productId}`); // Invalidate Cache } // Bulk Invalidation بالـ Pattern async function invalidateCategory(category: string) { const keys = await redis.keys(`products:category:${category}:*`); if (keys.length) await redis.del(...keys); // ⚠️ KEYS في Production: استخدم SCAN بدلاً منه (لا يُحجب) }
🔐 Sessions وRate Limiting

Session Management, JWT Blocklist, Rate Limiting with Sliding Window

Rate Limiting — Sliding Window Algorithm
// Rate Limiting: 100 requests/minute per IP async function rateLimit(ip: string): Promise<{ allowed: boolean; remaining: number }> { const key = `rate:${ip}`; const limit = 100; const now = Date.now(); const window = 60000; // 60 seconds in ms const pipeline = redis.pipeline(); pipeline.zremrangebyscore(key, 0, now - window); // Remove old entries pipeline.zadd(key, now, `${now}-${Math.random()}`); // Add current request pipeline.zcard(key); // Count requests pipeline.expire(key, 60); // TTL const results = await pipeline.exec(); const count = results[2][1] as number; return { allowed: count <= limit, remaining: Math.max(0, limit - count) }; } // JWT Blocklist — Logout يُبطل الـ Token async function logout(token: string, expiresAt: number) { const ttl = expiresAt - Math.floor(Date.now() / 1000); await redis.setex(`blocklist:${token}`, ttl, '1'); } async function isTokenBlocked(token: string): Promise<boolean> { return (await redis.exists(`blocklist:${token}`)) === 1; }

⚠️ Redis Gotchas — تجنّبها

🔴 لا تُخزّن Secret Keys في Redis بدون Encryption
🔴 KEYS * في Production = يُحجب Server — استخدم SCAN بدلاً منه
🔴 لا تُخزّن بيانات لا يمكن إعادة بنائها في Redis فقط — هو Cache لا Database
🔴 maxmemory-policy مهم — اختر allkeys-lru للـ Cache

🧠 اختبار الفهم
١. تريد بناء Leaderboard لعبة يتحدث في Real-time. أي Redis Data Structure تستخدم؟
Hash — لأنه يُخزّن Player وScore
Sorted Set (ZSet) — مُرتّب تلقائياً بالـ Score. ZADD لإضافة/تحديث Score، ZREVRANGE للـ Top-N، ZRANK لإيجاد ترتيب لاعب محدد. كل هذا بـ O(log n).
String — لكل لاعب key منفصل
٢. ما الـ Cache-Aside Pattern وما نقطة ضعفه؟
Cache يُكتب فيه مع كل Database Write
Cache-Aside: اقرأ من Cache أولاً. إذا Cache Miss: اقرأ من DB وخزّن في Cache. نقطة الضعف: خلال Cache Miss Window قد يصل طلبات كثيرة للـ DB معاً (Cache Stampede/Thundering Herd). الحل: Mutex Lock للـ Key أثناء الجلب.
Cache يُحدَّث Asynchronously
٣. لماذا نستخدم Lua Script لتحرير الـ Distributed Lock؟
لأن Lua أسرع
لضمان الـ Atomicity — التحقق أن القيمة هي قيمتنا ثم الحذف يجب أن يحدث بشكل Atomic. بدون Lua: قد يتحقق Server A من الـ Lock ثم قبل الحذف يُنتهي الـ TTL ويأخذ Server B الـ Lock ثم يحذفه Server A خطأً!
Lua الوحيد الذي يدعم DEL
٤. لماذا يجب تجنّب KEYS * في Production؟
لأنه يُرجع نتائج غلط
Redis Single-Threaded — KEYS * يُحجب الـ Server أثناء البحث في كل الـ Keys. مع ملايين الـ Keys = Server يتوقف عن الاستجابة لثوانٍ. SCAN: يبحث بشكل Incremental بدون blocking — آمن للـ Production.
