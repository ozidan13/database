EXISTS vs IN vs JOIN — Performance Differences
-- ❌ IN مع Subquery كبيرة — يُنفّذ الـ Subquery لكل صف SELECT * FROM customers WHERE id IN (SELECT customer_id FROM orders WHERE total > 5000); -- ✅ EXISTS — يتوقف بمجرد إيجاد أول نتيجة (Short-Circuit) SELECT * FROM customers c WHERE EXISTS ( SELECT 1 FROM orders o WHERE o.customer_id = c.id AND o.total > 5000 ); -- PostgreSQL غالباً يُحوّل IN → EXISTS تلقائياً (check EXPLAIN) -- ❌ OR على أعمدة مختلفة — يمنع استخدام Index SELECT * FROM customers WHERE email = 'a@b.com' OR phone = '01000000'; -- ✅ UNION ALL — يُمكّن استخدام Index لكل عمود SELECT * FROM customers WHERE email = 'a@b.com' UNION ALL SELECT * FROM customers WHERE phone = '01000000'; -- ❌ Function على الـ Column يمنع Index SELECT * FROM orders WHERE DATE(order_date) = '2024-01-15'; -- → Seq Scan لأن DATE() تُحوّل الـ Column -- ✅ Range بدون Function SELECT * FROM orders WHERE order_date >= '2024-01-15' AND order_date < '2024-01-16'; -- → Index Scan على order_date ✅ -- ❌ Wildcard أول → Seq Scan WHERE name LIKE '%محمد' -- لا يستخدم Index -- ✅ Wildcard آخر → Index Scan WHERE name LIKE 'محمد%' -- يستخدم B-Tree Index
📄 Pagination — OFFSET مقابل Keyset

Offset Pagination vs Keyset (Cursor-based) Pagination

OFFSET Pagination Problem vs Keyset Solution
-- ❌ OFFSET Pagination — المشكلة SELECT * FROM orders ORDER BY created_at DESC LIMIT 20 OFFSET 100000; -- PostgreSQL يقرأ 100020 صف ثم يرمي أول 100000! -- الصفحة 5000 (OFFSET=100000) أبطأ بكثير من الصفحة 1 -- ✅ Keyset Pagination — الحل الاحترافي -- الصفحة الأولى: SELECT id, created_at, customer_id, total FROM orders ORDER BY created_at DESC, id DESC -- id لـ Tiebreaking LIMIT 20; -- آخر سجل: created_at='2024-03-15 10:00', id=5050 -- الصفحة التالية — استخدم آخر قيم من الصفحة السابقة SELECT id, created_at, customer_id, total FROM orders WHERE (created_at, id) < ('2024-03-15 10:00', 5050) ORDER BY created_at DESC, id DESC LIMIT 20; -- Index Scan — O(log n) بغض النظر عن رقم الصفحة! -- Index: CREATE INDEX ON orders(created_at DESC, id DESC); -- ✅ COUNT مع Pagination — تجنّب COUNT(*) الثقيل -- بدلاً من COUNT(*) الدقيق: SELECT reltuples::bigint AS approximate_count FROM pg_class WHERE relname = 'orders'; -- تقريب سريع جداً من pg_class stats
🪟 Window Functions — تحليل بدون GROUP BY

Window Functions — Running Totals, Rankings, Lead/Lag

Window Functions — E-Commerce Analytics
-- ══ Running Total للمبيعات الشهرية ══ SELECT order_date, daily_revenue, SUM(daily_revenue) OVER ( ORDER BY order_date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW ) AS cumulative_revenue FROM ( SELECT DATE(order_date) AS order_date, SUM(total) AS daily_revenue FROM orders WHERE status = 'delivered' GROUP BY 1 ) daily; -- ══ Customer Ranking بإجمالي مشترياتهم ══ SELECT c.full_name, SUM(o.total) AS total_spent, RANK() OVER (ORDER BY SUM(o.total) DESC) AS rank, DENSE_RANK() OVER (ORDER BY SUM(o.total) DESC) AS dense_rank, NTILE(10) OVER (ORDER BY SUM(o.total) DESC) AS decile FROM customers c JOIN orders o ON o.customer_id = c.id GROUP BY c.id, c.full_name; -- ══ LEAD/LAG — مقارنة بالفترة السابقة ══ SELECT month, revenue, LAG(revenue) OVER (ORDER BY month) AS prev_month, ROUND((revenue - LAG(revenue) OVER (ORDER BY month)) * 100.0 / NULLIF(LAG(revenue) OVER (ORDER BY month), 0), 1) AS growth_pct FROM monthly_revenue; -- ══ أفضل منتج لكل Category (PARTITION BY) ══ SELECT category_id, product_id, name, sales FROM ( SELECT category_id, product_id, name, sales, ROW_NUMBER() OVER (PARTITION BY category_id ORDER BY sales DESC) AS rn FROM products ) ranked WHERE rn = 1; -- أفضل منتج فقط لكل Category
🗄️ Materialized Views — تحسين التقارير الثقيلة

Materialized Views — Pre-compute Heavy Queries

Materialized View — Monthly Sales Dashboard
-- Query تستغرق 30 ثانية! → نحوّلها لـ Materialized View CREATE MATERIALIZED VIEW mv_monthly_sales AS SELECT DATE_TRUNC('month', o.order_date) AS month, c.country, cat.name AS category, COUNT(DISTINCT o.id) AS order_count, SUM(oi.quantity) AS units_sold, SUM(oi.unit_price * oi.quantity) AS revenue, AVG(o.total) AS avg_order_value FROM orders o JOIN customers c ON c.id = o.customer_id JOIN order_items oi ON oi.order_id = o.id JOIN products p ON p.id = oi.product_id JOIN categories cat ON cat.id = p.category_id WHERE o.status = 'delivered' GROUP BY 1, 2, 3; -- Index على الـ Materialized View CREATE INDEX idx_mv_month ON mv_monthly_sales(month); CREATE INDEX idx_mv_country ON mv_monthly_sales(country); -- التحديث (Refresh) REFRESH MATERIALIZED VIEW CONCURRENTLY mv_monthly_sales; -- CONCURRENTLY: لا يُقفّل عند القراءة — يحتاج UNIQUE index CREATE UNIQUE INDEX ON mv_monthly_sales(month, country, category); -- جدولة التحديث (باستخدام pg_cron extension) SELECT cron.schedule( 'refresh-sales-mv', '0 * * * *', -- كل ساعة 'REFRESH MATERIALIZED VIEW CONCURRENTLY mv_monthly_sales' ); -- القراءة من الـ View — كأنها جدول عادي SELECT * FROM mv_monthly_sales WHERE month = DATE_TRUNC('month', NOW()) ORDER BY revenue DESC; -- 30 ثانية → 2ms ✅
⚙️ PostgreSQL Config Tuning — الإعدادات الحيوية

postgresql.conf — Key Performance Parameters

postgresql.conf — أهم إعدادات الأداء
-- ══ Memory ══ shared_buffers = 4GB -- 25% من الـ RAM (الأهم) work_mem = 64MB -- لكل Sort/Hash operation — احتاط من Memory bloat maintenance_work_mem = 512MB -- للـ VACUUM، CREATE INDEX effective_cache_size = 12GB -- الـ RAM المتاحة للـ OS Cache (75% من RAM) -- ══ WAL ══ wal_buffers = 64MB -- Buffer للـ WAL write (-1 = auto) checkpoint_completion_target = 0.9 -- تمدّد الـ Checkpoint على 90% من الفترة max_wal_size = 4GB -- الحد الأقصى للـ WAL قبل Checkpoint -- ══ Planner Costs (تساعد الـ Planner على اختيار الخطة الصحيحة) ══ random_page_cost = 1.1 -- للـ SSD (الافتراضي 4.0 للـ HDD) effective_io_concurrency = 200 -- للـ SSD -- ══ Connections ══ max_connections = 100 -- استخدم PgBouncer لأكثر من هذا -- ══ Autovacuum ══ autovacuum_vacuum_scale_factor = 0.05 -- 5% (الافتراضي 20% — كثير للجداول الكبيرة) autovacuum_analyze_scale_factor = 0.02 -- 2% -- ══ Per-Session Tuning لـ Heavy Reports ══ SET work_mem = '256MB'; -- لـ Report Session فقط SET random_page_cost = 1.1;
🎯 أهم 5 تحسينات فورية

1️⃣ shared_buffers = 25% RAM — الأهم على الإطلاق
2️⃣ random_page_cost = 1.1 إذا كنت على SSD
3️⃣ أضف Indexes على Foreign Keys (PostgreSQL لا يفعل هذا تلقائياً!)
4️⃣ ANALYZE بعد Bulk Insert كبير
5️⃣ استبدل OFFSET بـ Keyset Pagination في الـ APIs

🧠 اختبار الفهم
١. لماذا WHERE DATE(order_date) = '2024-01-15' يُبطئ الـ Query؟
DATE() دالة بطيئة
DATE() تُطبَّق على كل صف — تمنع استخدام الـ B-Tree Index على order_date. الحل: WHERE order_date >= '2024-01-15' AND order_date < '2024-01-16' → Range Query يستخدم الـ Index.
لأن DATE لا تدعم Timestamp
٢. ما مشكلة OFFSET 100000 في Pagination؟
يُرجع بيانات غلط
PostgreSQL يقرأ 100020 صف ثم يرمي أول 100000. كلما زاد الـ OFFSET زاد الوقت خطياً. الحل: Keyset Pagination باستخدام WHERE (col, id) < (last_val, last_id) → O(log n).
لا توجد مشكلة
٣. ما الـ Materialized View ومتى تستخدمه؟
View عادية لكن محفوظة في الـ Cache
نتائج Query مُحسوبة مسبقاً ومُخزّنة على الـ Disk — القراءة فورية. للتحديث تحتاج REFRESH MATERIALIZED VIEW. مثالي للـ Reports الثقيلة التي لا تحتاج real-time data.
View مُقفّلة للكتابة
٤. ما قيمة random_page_cost الموصى بها للـ SSD ولماذا؟
4.0 (الافتراضي)
1.1 للـ SSD — الافتراضي 4.0 مصمّم للـ HDD (Random I/O ثقيل). SSD: Random I/O ~= Sequential. قيمة 1.1 تُخبر الـ Planner أن Index Scan أرخص → يختاره بدلاً من Seq Scan.
0.5 للـ SSD
