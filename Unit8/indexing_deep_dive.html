Index Types — Choose the Right One for Each Use Case

B-Tree ⭐

الافتراضي والأكثر استخداماً. يدعم =، <، >، BETWEEN، LIKE 'abc%'، ORDER BY.

= < > BETWEEN LIKE 'abc%'

Hash

أسرع من B-Tree للـ = فقط. لا يدعم Range Queries. يُستخدم نادراً (PostgreSQL 10+ reliable).

= فقط، لا < < LIKE

GIN (Generalized Inverted)

للبيانات المركّبة: JSONB، Arrays، tsvector (Full-Text Search). يُنشئ inverted index لكل عنصر داخل القيمة.

JSONB @> || array @> || to_tsvector @@

GiST (Generalized Search Tree)

للبيانات الهندسية والـ Ranges والـ Full-Text البسيط. يدعم العمليات الفضائية (PostGIS).

geometry &&, tsquery, ip4r

BRIN (Block Range Index)

صغير جداً (أصغر بـ 1000x من B-Tree). مثالي للبيانات المُرتّبة فيزيائياً مثل timestamps وserial IDs.

Timestamp، Serial — بيانات متسلسلة

SP-GiST

للبيانات ذات التوزيع غير المنتظم كالنقاط الهندسية والـ IP Addresses والـ Tries.

inet ops, text patterns, box ops
🌳 B-Tree — كيف يعمل من الداخل؟

B-Tree Internal Structure — Why It's O(log n)

B-Tree (Balanced Tree) = شجرة متوازنة. كل عمليات البحث O(log n). بيانات مُرتّبة في الـ Leaf Nodes مع pointers للـ Heap Pages.

Composite Index — الترتيب مهم جداً (ESR Rule)
-- ESR Rule: Equality → Sort → Range -- الحقول التي تُفلتر بـ = تأتي أولاً، ثم ORDER BY، ثم BETWEEN/</> -- Query: WHERE status = 'active' AND category_id = 3 ORDER BY price -- Index الصحيح: CREATE INDEX idx_products_composite ON products(status, category_id, price); -- ✅ status (Equality) → category_id (Equality) → price (Sort/Range) -- ❌ الترتيب الخاطئ: price أولاً CREATE INDEX idx_wrong ON products(price, status, category_id); -- لن يُستخدم بكفاءة! PostgreSQL لا يستطيع Skip الأعمدة الأولى -- ══ Column Prefix Rule: يستخدم الـ Index من اليسار ══ -- Index: (a, b, c) -- ✅ WHERE a = 1 → يستخدم (a) -- ✅ WHERE a = 1 AND b = 2 → يستخدم (a, b) -- ✅ WHERE a = 1 AND b = 2 AND c = 3 → يستخدم (a, b, c) -- ❌ WHERE b = 2 → لن يستخدم Index! (تجاوز a) -- ❌ WHERE c = 3 → لن يستخدم Index! -- ══ NULL values في الـ B-Tree ══ -- B-Tree يفهرس NULL — يدعم IS NULL وIS NOT NULL CREATE INDEX idx_null_safe ON orders(shipped_at NULLS FIRST);
🎯 Advanced Indexes — الأسلحة الخفية

Partial, Expression, Covering, and GIN Indexes

Partial Index — يفهرس جزءاً من الجدول فقط
-- مثال E-Commerce: 90% من الـ orders بحالة 'delivered' لا نبحث فيها -- نريد Index للـ active orders فقط -- ❌ Index عادي: يفهرس 1M record (معظمها غير مفيد) CREATE INDEX idx_orders_all ON orders(status); -- ✅ Partial Index: يفهرس 50K فقط (pending + shipped) CREATE INDEX idx_orders_active ON orders(customer_id, order_date) WHERE status IN ('pending', 'shipped'); -- أصغر بـ 20x، أسرع بـ 10x، يشغل ذاكرة أقل -- Partial Unique: Email ليس NULL فقط CREATE UNIQUE INDEX idx_user_email_unique ON users(email) WHERE email IS NOT NULL; -- يسمح بـ NULL متعددة لكن يمنع تكرار email
Expression Index و GIN Index
-- Expression Index: يفهرس نتيجة دالة -- Query: WHERE LOWER(email) = 'ahmed@test.com' -- ❌ بدون Expression Index → Seq Scan (LOWER() تمنع استخدام الـ Index) CREATE INDEX idx_email_lower ON customers(LOWER(email)); -- الآن: WHERE LOWER(email) = 'ahmed@...' → يستخدم Index -- Expression Index لـ Date Parts CREATE INDEX idx_orders_month ON orders(DATE_TRUNC('month', order_date)); -- ══ GIN Index للـ JSONB ══ -- Query: WHERE metadata @> '{"color": "red"}' CREATE INDEX idx_products_meta_gin ON products USING gin(metadata); -- يدعم: @> (contains), ? (key exists), ?| (any key), ?& (all keys) -- GIN للـ Full-Text Search ALTER TABLE products ADD COLUMN search_vector tsvector GENERATED ALWAYS AS (to_tsvector('arabic', name || ' ' || COALESCE(description, ''))) STORED; CREATE INDEX idx_products_fts ON products USING gin(search_vector); -- البحث: SELECT name, ts_rank(search_vector, query) AS rank FROM products, to_tsquery('arabic', 'لابتوب & ألعاب') query WHERE search_vector @@ query ORDER BY rank DESC LIMIT 20; -- ══ BRIN Index للـ Timestamp (أصغر بـ 1000x) ══ CREATE INDEX idx_orders_date_brin ON orders USING brin(order_date) WITH (pages_per_range = 64); -- مثالي إذا الـ orders مُدرجة بترتيب زمني (correlated)
🧠 استراتيجيات الفهرسة الصحيحة

Indexing Strategy — What to Index, What to Avoid

✅ متى تُضيف Index؟

• أعمدة في WHERE، JOIN ON، ORDER BY، GROUP BY
• Foreign Keys (PostgreSQL لا يُنشئها تلقائياً!)
• أعمدة ذات Cardinality عالية (email, phone → أفيد من gender)
• Queries تُرجع <10% من الصفوف (Selective)

❌ متى لا تُضيف Index؟

• جداول صغيرة (<1000 صف) — Seq Scan أسرع!
• أعمدة boolean أو low-cardinality (gender, is_active) — أضف Partial Index بدلاً
• Heavy-Write Tables — كل INSERT/UPDATE/DELETE يُحدّث الـ Index
• Indexes غير مُستخدمة → تثقل الـ Write بدون فائدة

فحص الـ Indexes غير المُستخدمة
-- Indexes لم تُستخدم منذ آخر reset SELECT schemaname, tablename, indexname, idx_scan AS times_used, idx_tup_read AS tuples_read, idx_tup_fetch AS tuples_fetched, pg_size_pretty(pg_relation_size(indexrelid)) AS index_size FROM pg_stat_user_indexes WHERE idx_scan = 0 -- لم يُستخدم أبداً AND relid NOT IN ( -- ليس PK أو Unique SELECT conindid FROM pg_constraint WHERE contype IN ('p', 'u') ) ORDER BY pg_relation_size(indexrelid) DESC; -- حجم كل Index: SELECT indexname, pg_size_pretty(pg_relation_size(indexrelid)) AS size FROM pg_stat_user_indexes ORDER BY pg_relation_size(indexrelid) DESC;
🔧 Index Maintenance — CONCURRENTLY وBloat

Index Creation, Rebuilding, and Bloat Management

CONCURRENTLY — إنشاء Index بدون Lock
-- ❌ بدون CONCURRENTLY: يُقفّل الجدول للكتابة! CREATE INDEX idx_orders_status ON orders(status); -- يمنع INSERT/UPDATE/DELETE أثناء البناء — مشكلة في Production -- ✅ CONCURRENTLY: لا lock على الجدول CREATE INDEX CONCURRENTLY idx_orders_status ON orders(status); -- يأخذ وقتاً أطول لكن لا يمنع العمليات الأخرى -- لا يعمل داخل Transaction Block -- إعادة بناء Index (لعلاج الـ Bloat) REINDEX INDEX CONCURRENTLY idx_orders_status; REINDEX TABLE CONCURRENTLY orders; -- كل Indexes الجدول -- حذف Index: DROP INDEX CONCURRENTLY idx_orders_status; -- ══ Index Bloat — تشخيص ══ SELECT indexname, pg_size_pretty(pg_relation_size(indexrelid)) AS current_size, ROUND((1 - avg_leaf_density/100) * 100, 1) AS bloat_pct FROM pg_stat_user_indexes JOIN pgstatindex(indexrelid::regclass::text) USING (...) -- (نستخدم pgstattuple extension للتفاصيل الدقيقة) -- علاج بسيط: REINDEX REINDEX INDEX CONCURRENTLY idx_heavy_bloat;
🏭 E-Commerce Index Strategy

orders(customer_id, status, order_date) — Composite للـ customer history
orders(status) WHERE status IN ('pending','shipped') — Partial للـ active orders
products USING gin(metadata) — GIN للـ JSONB attributes
products USING gin(search_vector) — Full-Text Search

🧠 اختبار الفهم
١. عندك Index على (a, b, c). هل الـ Query WHERE b=2 AND c=3 ستستخدمه؟
نعم دائماً
لا — الـ B-Tree Index يُستخدم من اليسار (Column Prefix Rule). يجب أن يبدأ بـ a. بدون a في الـ WHERE، لن يستخدم الـ Index. الحل: أنشئ Index منفصل على (b, c).
نعم لكن جزئياً فقط على c
٢. متى تستخدم GIN Index بدلاً من B-Tree؟
للـ Integer columns
للبيانات المركّبة: JSONB @> (contains)، Arrays، tsvector (Full-Text). GIN ينشئ inverted index لكل عنصر داخل الـ Value — B-Tree يعامل القيمة كوحدة واحدة.
للـ Range Queries فقط
٣. لماذا يجب استخدام CREATE INDEX CONCURRENTLY في Production؟
لأنه أسرع
CREATE INDEX بدون CONCURRENTLY يُقفّل الجدول للكتابة أثناء البناء. في Production هذا يمنع كل INSERT/UPDATE/DELETE ويُعطّل التطبيق. CONCURRENTLY: لا lock، يعمل بالتوازي.
CONCURRENTLY الوحيد الذي يعمل على PostgreSQL 15
٤. ما الـ Partial Index وما فائدته على جدول orders؟
Index على نصف الأعمدة
Partial Index يُفهرس صفوفاً محددة فقط (بـ WHERE clause). مثال: INDEX ON orders(customer_id) WHERE status='pending' — يُفهرس 50K بدلاً من 1M. أصغر وأسرع لأن معظم Queries تهتم بالـ active orders فقط.
