Partitioning & Scaling

🤔 لماذا نحتاج Partitioning؟

The Problem — When Tables Become Too Large

جدول orders بـ 500M صف = مشاكل:

❌ بدون Partitioning

• VACUUM يستغرق ساعات على جدول 500M صف
• Index Rebuild يُوقف الـ Production
• DELETE لبيانات قديمة بطيء جداً (يُولّد WAL ضخم)
• الـ Index لا يتناسب في الـ shared_buffers

✅ مع Partitioning

• VACUUM يعمل على partition صغير فقط
• حذف الـ Partition القديم = DROP TABLE فوري (بدون DELETE scanning!)
• كل partition له Index أصغر يتناسب في الـ Cache
• Partition Pruning: Query تقرأ partition واحد فقط بدلاً من 500M

🗂️ أنواع الـ Partitioning

Range, List, Hash — When to Use Each

Range Partitioning

على نطاق من القيم. الأشيع: التاريخ (شهر/سنة). مثالي للـ Time-Series Data.

PARTITION BY RANGE(order_date)

List Partitioning

على قائمة قيم محددة. مثالي للـ Geographic Data (دولة/منطقة).

PARTITION BY LIST(country_code)

Hash Partitioning

توزيع متساوٍ بالـ Hash. لا يوجد منطق زمني أو جغرافي. مثالي للتوزيع المتوازن عند Sharding.

PARTITION BY HASH(customer_id)
Range Partitioning — Orders by Month
-- الجدول الأساسي (Parent Table) CREATE TABLE orders ( id BIGSERIAL NOT NULL, customer_id BIGINT NOT NULL, order_date TIMESTAMPTZ NOT NULL, total NUMERIC(12,2), status TEXT ) PARTITION BY RANGE (order_date); -- Partitions فردية (Child Tables) CREATE TABLE orders_2024_q1 PARTITION OF orders FOR VALUES FROM ('2024-01-01') TO ('2024-04-01'); CREATE TABLE orders_2024_q2 PARTITION OF orders FOR VALUES FROM ('2024-04-01') TO ('2024-07-01'); CREATE TABLE orders_2024_q3 PARTITION OF orders FOR VALUES FROM ('2024-07-01') TO ('2024-10-01'); CREATE TABLE orders_2024_q4 PARTITION OF orders FOR VALUES FROM ('2024-10-01') TO ('2025-01-01'); -- DEFAULT Partition — يستقبل ما لا يناسب القيم المحددة CREATE TABLE orders_default PARTITION OF orders DEFAULT; -- Index على الـ Partitions — يُطبَّق على كل partition CREATE INDEX idx_orders_customer ON orders(customer_id); -- PostgreSQL ينشئ index تلقائياً على كل partition!
List و Hash Partitioning
-- ══ List Partitioning — بالدولة ══ CREATE TABLE customers ( id BIGSERIAL, email TEXT, country_code TEXT NOT NULL ) PARTITION BY LIST (country_code); CREATE TABLE customers_mena PARTITION OF customers FOR VALUES IN ('EG', 'SA', 'AE', 'JO', 'KW'); CREATE TABLE customers_europe PARTITION OF customers FOR VALUES IN ('UK', 'DE', 'FR', 'NL'); CREATE TABLE customers_other PARTITION OF customers DEFAULT; -- ══ Hash Partitioning — توزيع متساوٍ ══ CREATE TABLE audit_logs ( id BIGSERIAL, user_id BIGINT, action TEXT, created_at TIMESTAMPTZ ) PARTITION BY HASH (user_id); CREATE TABLE audit_logs_0 PARTITION OF audit_logs FOR VALUES WITH (MODULUS 4, REMAINDER 0); CREATE TABLE audit_logs_1 PARTITION OF audit_logs FOR VALUES WITH (MODULUS 4, REMAINDER 1); CREATE TABLE audit_logs_2 PARTITION OF audit_logs FOR VALUES WITH (MODULUS 4, REMAINDER 2); CREATE TABLE audit_logs_3 PARTITION OF audit_logs FOR VALUES WITH (MODULUS 4, REMAINDER 3);
✂️ Partition Pruning — السحر الحقيقي

Partition Pruning — The Planner Reads Only Relevant Partitions

Partition Pruning — EXPLAIN يُظهر أي Partitions تُقرأ
-- Query على orders مع فلتر التاريخ EXPLAIN SELECT * FROM orders WHERE order_date BETWEEN '2024-01-01' AND '2024-03-31'; -- الناتج: -- Append -- → Seq Scan on orders_2024_q1 ← فقط هذا الـ Partition! -- Partitions removed: 3 out of 4 ← يتجاهل Q2,Q3,Q4 -- ❌ لا Pruning: بدون فلتر على الـ Partition Key SELECT * FROM orders WHERE customer_id = 42; -- يقرأ كل الـ Partitions! (لأن customer_id ليس الـ Partition Key) -- ✅ حذف Partition قديم — فوري جداً ALTER TABLE orders DETACH PARTITION orders_2022_q1; -- أو DROP TABLE orders_2022_q1; -- فوري بغض النظر عن الحجم! -- إضافة Partition جديد CREATE TABLE orders_2025_q1 PARTITION OF orders FOR VALUES FROM ('2025-01-01') TO ('2025-04-01'); -- Indexes تُنشأ تلقائياً على الـ Partition الجديد
🔄 Connection Pooling — PgBouncer

PgBouncer — Handle Thousands of Connections Without Overloading PostgreSQL

كل Connection في PostgreSQL = Process كامل (~5MB RAM). 1000 Connection = 5GB RAM مستهلكة فقط للاتصالات. PgBouncer يُحلّ هذا.

PgBouncer — الإعداد والمفاهيم
-- pgbouncer.ini [databases] ecommerce_db = host=127.0.0.1 dbname=ecommerce_db [pgbouncer] listen_port = 6432 listen_addr = * auth_type = md5 auth_file = /etc/pgbouncer/users.txt pool_mode = transaction ← الأشيع: Connection يُؤجَّر فقط أثناء Transaction max_client_conn = 1000 ← 1000 client يتصلون بـ PgBouncer default_pool_size = 25 ← فقط 25 Connection حقيقية لـ PostgreSQL reserve_pool_size = 5 -- pool_mode options: -- session: Connection مُخصَّص لكل Client حتى ينتهي -- transaction: الأفضل — يُؤجَّر فقط أثناء Transaction (تُرجَّع بعدها) -- statement: كل statement (خطر مع Multi-statement Transactions) -- النتيجة: 1000 Client ← PgBouncer ← 25 PostgreSQL Connection -- تقليل استهلاك RAM: 5GB → 125MB!

Streaming Replication, Read Replicas, Vertical vs Horizontal Scaling

Streaming Replication Setup (PostgreSQL Primary)
-- Primary postgresql.conf wal_level = replica ← الحد الأدنى للـ Replication max_wal_senders = 5 ← عدد الـ Replicas المسموح بها wal_keep_size = 1GB ← احتفظ بالـ WAL لحين قراءته من الـ Replicas -- Primary pg_hba.conf host replication replica_user 10.0.0.2/32 scram-sha-256 -- إنشاء Replication User CREATE ROLE replica_user REPLICATION LOGIN PASSWORD 'strongpass'; -- Replica Setup: نسخ Base Backup $ pg_basebackup -h primary_ip -U replica_user -D /var/lib/postgresql/data -Fp -Xs -P -R -- -R: ينشئ standby.signal تلقائياً → يبدأ كـ Replica -- Replica postgresql.conf (v14+) primary_conninfo = 'host=primary_ip port=5432 user=replica_user password=pass' hot_standby = on ← يسمح بالقراءة من الـ Replica hot_standby_feedback = on ← يمنع VACUUM من حذف Rows محتاجة للـ Query -- Application Level: توزيع الـ Reads -- WRITE → Primary -- READ (Reports, Analytics) → Replica -- مثال Node.js بـ PgBouncer: -- writePool → primary:6432 -- readPool → replica:6432

١. ما الـ Partition Pruning ولماذا هو مهم؟
حذف Partitions القديمة
الـ Planner يُحدّد Partitions المطلوبة فقط ويتجاهل الباقي — مثلاً فلتر شهر يناير يقرأ orders_2024_q1 فقط من 4 Partitions. يُقلّل الـ I/O بشكل دراماتيكي.

٢. ما الفرق الجوهري بين DETACH PARTITION وDELETE لبيانات قديمة؟
لا فرق في الأداء
DETACH/DROP PARTITION: فوري (metadata operation فقط) بغض النظر عن الحجم. DELETE: يُنفَّذ صفاً بصف، يُولّد WAL ضخم، يستغرق ساعات على ملايين الصفوف. الـ Partitioning يجعل Retention Policy سهلة!
DELETE أفضل لأنه يُحدّث الإحصائيات
٣. ما الـ pool_mode المناسب في PgBouncer لمعظم التطبيقات؟
session pool
transaction pool — يُؤجَّر الـ Connection الحقيقي من Pool أثناء Transaction فقط، يُرجَع بعد COMMIT/ROLLBACK. يُمكّن 1000 client على 25 connection. statement pool خطير مع Multi-statement Transactions.
statement pool
