EXPLAIN ANALYZE

Reading Query Execution Plans — The Most Powerful Debugging Tool in PostgreSQL

EXPLAIN ANALYZE هو X-Ray لـ Query بتاعتك — يُريك بالضبط ماذا PostgreSQL يفعل، كم وقت تستهلك، وأين الـ Bottleneck. بدونه، تحسين الأداء مجرد تخمين.

EXPLAIN vs EXPLAIN ANALYZE Cost وRows وWidth Seq Scan vs Index Scan Hash Join vs Nested Loop Actual vs Estimated Rows Buffers Analysis pg_stat_statements
⚗️ EXPLAIN vs EXPLAIN ANALYZE

Estimation vs Reality — The Critical Difference

EXPLAIN

يُقدّر الخطة فقط بدون تنفيذ فعلي. Cost وRows مجرد تقديرات من الـ Planner. سريع لكن غير دقيق للتشخيص.

EXPLAIN ANALYZE

يُنفّذ الـ Query فعلاً ويُجمع الإحصائيات الحقيقية. تجد الفرق بين التقدير والواقع. استخدمه دائماً للتشخيص.

EXPLAIN ANALYZE — خيارات كاملة
-- الأساسي EXPLAIN SELECT * FROM orders WHERE customer_id = 1; -- الأقوى: تنفيذ حقيقي + كل التفاصيل EXPLAIN (ANALYZE, BUFFERS, VERBOSE, FORMAT JSON) SELECT * FROM orders WHERE customer_id = 1; -- الخيارات: -- ANALYZE: تنفيذ فعلي وقياس الوقت -- BUFFERS: عدد الـ Pages المقروءة من Cache والـ Disk -- VERBOSE: مزيد من التفاصيل (Output columns, Schema) -- COSTS: يُظهر أو يُخفي الـ Cost (مفعّل افتراضياً) -- FORMAT: TEXT (افتراضي)، JSON، XML، YAML -- للـ DML (لا يُطبّق التغييرات) BEGIN; EXPLAIN ANALYZE UPDATE orders SET status = 'shipped' WHERE id = 1; ROLLBACK; -- يُلغي التغيير لكن الـ Plan ظهر
📖 كيف تقرأ الـ Execution Plan؟

Anatomy of an Execution Plan — Cost, Rows, Width, Actual Time

الـ Plan يُقرأ من أسفل لأعلى — الـ Node الأعمق هو الأول في التنفيذ. كل سطر يحمل معلومات ثمينة.

Hash Join (cost=245.50..1823.40 rows=12000 width=64) (actual time=15.234..180.445 rows=11892 loops=1) Hash Cond: (o.customer_id = c.id) → Seq Scan on orders o (cost=0..890.00 rows=50000 width=48) (actual time=0.012..45.123 rows=50000 loops=1) → Hash (cost=150..150 rows=7640 width=16) Batches: 4 Memory Usage: 512kB ← تجاوز الـ work_mem! → Seq Scan on customers c (cost=0..150 rows=7640 width=16) Planning Time: 2.145 ms Execution Time: 195.234 ms
الحقل المعنى ملاحظة
cost=0.00..890.00تكلفة البدء..تكلفة الانتهاء (وحدات افتراضية)ليست ميلي ثانية
rows=50000تقدير عدد الصفوف الناتجةقد يختلف عن actual
width=48متوسط حجم الصف بالـ Bytesمفيد لحساب الذاكرة
actual time=0..45msالوقت الفعلي للتنفيذمن ANALYZE فقط
loops=5كم مرة نُفِّذ الـ Node (Nested Loop)actual يُضرب × loops
⚠️ المخيف: rows estimation خاطئ جداً

إذا وجدت rows=10 لكن actual rows=50000 — الـ Planner يعيش في وهم! سبب؟ الإحصائيات قديمة. الحل: ANALYZE table_name; لتحديث الإحصائيات.

🔍 أنواع الـ Scan Nodes

Scan Types — Sequential, Index, Bitmap, Index-Only

Seq Scan — المصيبة الكبيرة

يقرأ كل الجدول من أول لآخر. مقبول للجداول الصغيرة أو عند جلب >10% من الصفوف. مقبول للجداول الصغيرة فقط!

Index Scan — الشاطر

يستخدم الـ Index للعثور على الصفوف مباشرة. مثالي لـ Selective Queries (تُرجع صفوف قليلة).

Bitmap Index Scan

يجمع عدة Index Scans في Bitmap، ثم يقرأ الـ Pages بترتيب فيزيائي. مثالي للـ OR conditions وRange Queries.

Index-Only Scan ⭐

الـ Index يحمل كل البيانات المطلوبة — لا حاجة للرجوع للجدول الأصلي. الأسرع على الإطلاق. يحتاج Covering Index.

مثال: Seq Scan → Index Scan → Index-Only
-- ❌ Seq Scan على 1M record EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 42; -- Seq Scan on orders (cost=0..25432 rows=1 width=80) -- (actual time=12.3..180.4 rows=23 loops=1) -- ✅ إضافة Index CREATE INDEX idx_orders_customer ON orders(customer_id); EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 42; -- Index Scan using idx_orders_customer -- (cost=0.43..12.5 rows=23 width=80) -- (actual time=0.045..0.234 rows=23 loops=1) -- ⭐ Covering Index → Index-Only Scan CREATE INDEX idx_orders_cust_cover ON orders(customer_id) INCLUDE (status, total_amount, order_date); -- تضمين البيانات في الـ Index EXPLAIN ANALYZE SELECT status, total_amount, order_date FROM orders WHERE customer_id = 42; -- Index Only Scan (cost=0.43..8.1 rows=23 width=24) -- (actual time=0.023..0.089 rows=23 loops=1) -- Heap Fetches: 0 ← لا يقرأ الجدول الأصلي أبداً!
💾 Buffers Analysis — الـ Cache Performance

BUFFERS Option — Hit vs Read, Cache Effectiveness

EXPLAIN (ANALYZE, BUFFERS) — قراءة الـ Buffer Stats
EXPLAIN (ANALYZE, BUFFERS) SELECT o.id, c.email, o.total_amount FROM orders o JOIN customers c ON c.id = o.customer_id WHERE o.status = 'pending'; -- الناتج (مع تفسير كل سطر): -- Seq Scan on orders -- Buffers: shared hit=1240 read=380 ← 1240 من Cache، 380 من الـ Disk -- Cache Hit Ratio = 1240/(1240+380) = 76.5% ← جيد, الهدف >90% -- hash=X: صفحات مستخدمة لبناء الـ Hash Table في الذاكرة -- written=X: صفحات كتبها PostgreSQL (Dirty Pages) -- حساب Cache Hit Ratio للـ DB كاملة: SELECT sum(heap_blks_hit) AS cache_hits, sum(heap_blks_read) AS disk_reads, ROUND(sum(heap_blks_hit) * 100.0 / NULLIF(sum(heap_blks_hit + heap_blks_read), 0), 2) AS cache_hit_ratio FROM pg_statio_user_tables; -- هدفك: > 95% — لو أقل زِد shared_buffers في postgresql.conf
🔧 تشخيص المشاكل الشائعة

Common Query Plan Problems and Their Solutions

pg_stat_statements — إيجاد أبطأ الـ Queries
-- تفعيل pg_stat_statements (postgresql.conf) -- shared_preload_libraries = 'pg_stat_statements' CREATE EXTENSION IF NOT EXISTS pg_stat_statements; -- Top 10 أبطأ queries SELECT ROUND(total_exec_time/1000,2) AS total_sec, calls, ROUND(mean_exec_time,2) AS avg_ms, ROUND(stddev_exec_time,2) AS stddev_ms, rows, query FROM pg_stat_statements ORDER BY total_exec_time DESC LIMIT 10; -- Queries بأسوأ Cache Hit Ratio SELECT queryid, ROUND(shared_blks_hit * 100.0 / NULLIF(shared_blks_hit + shared_blks_read,0),1) AS hit_pct, shared_blks_read AS disk_reads, query FROM pg_stat_statements WHERE shared_blks_read > 1000 ORDER BY shared_blks_read DESC LIMIT 20; -- إعادة تعيين الإحصائيات: SELECT pg_stat_statements_reset();
🎯 علامات المشاكل في الـ Plan

🔴 Seq Scan على جدول كبير: أضف Index
🔴 rows=10 لكن actual=500000: شغّل ANALYZE لتحديث الإحصائيات
🔴 Hash Batches: 4+: زِد work_mem في الـ Session
🔴 Nested Loop مع rows كثيرة: ربما Hash Join أفضل
🔴 Execution Time عالي لكن Planning Time منخفض: الـ Query تحتاج Index
🟢 Index Only Scan: ممتاز — Covering Index يعمل!

🧠 اختبار الفهم
١. ما الفرق بين EXPLAIN وEXPLAIN ANALYZE؟
ANALYZE أبطأ بسبب طباعة التفاصيل
EXPLAIN يُقدّر الـ Plan بدون تنفيذ فعلي. EXPLAIN ANALYZE يُنفّذ الـ Query فعلاً ويُظهر الوقت الحقيقي وعدد الصفوف الفعلية — ضروري للتشخيص الدقيق.
EXPLAIN ANALYZE يُعدّل البيانات
٢. رأيت في الـ Plan: rows=5 لكن actual rows=80000. ما السبب والحل؟
الـ Index مفقود
الإحصائيات قديمة — الـ Planner لا يعرف التوزيع الحقيقي للبيانات. الحل: ANALYZE table_name؛ أو تأكّد أن autovacuum يعمل.
الـ Query مكتوبة غلط
٣. ما الـ Index-Only Scan ولماذا هو الأسرع؟
Scan يستخدم RAM فقط
الـ Index يحمل كل البيانات المطلوبة (Covering Index بـ INCLUDE) — لا يحتاج للرجوع للجدول الأصلي. Heap Fetches=0! أسرع لأنه يقرأ بيانات أقل.
يستخدم الـ OS Cache مباشرة
٤. في BUFFERS: hit=100, read=900. ما Cache Hit Ratio وماذا يعني؟
90% — ممتاز
10% فقط — معظم البيانات تُقرأ من الـ Disk. هذا بطيء جداً. الحل: زِد shared_buffers في postgresql.conf أو حسّن الـ Queries لتقليل Disk I/O.
