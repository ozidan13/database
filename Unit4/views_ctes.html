👁️ لماذا نستخدم Views؟

Views — The Abstraction Layer Over Your Tables

الـ View هو استعلام محفوظ بـ اسم — بيتعامل معاه كجدول. لكنه مش بيخزّن بيانات (بشكل افتراضي). في كل مرة تستعلم منه بيُنفَّذ الاستعلام الأصلي.

🔐 الأمان

تُعطي المستخدمين وصولاً لأعمدة محددة فقط. مثلاً: HR يشوف الرواتب، لكن باقي الإدارات مش عارفينها.

🧹 البساطة

تُخفي تعقيد الـ JOINs. المستخدم يكتب SELECT * FROM employee_full بدل 5 JOINs.

🏗️ الصيانة

لو تغيّرت هيكل الجدول، بتعدّل الـ View بس — التطبيق مش بيتأثر.

🏗️ CREATE VIEW — بالتفصيل

Full CREATE VIEW Syntax and Options

CREATE VIEW — Complete Reference
-- ══ View بسيطة ══ CREATE VIEW active_employees AS SELECT emp_id, full_name, job_title, dept_id, hire_date, EXTRACT(YEAR FROM AGE(hire_date))::INT AS years_service FROM employee WHERE is_active = TRUE; -- الاستعلام منها مثل جدول عادي SELECT * FROM active_employees WHERE dept_id = 3; -- ══ CREATE OR REPLACE VIEW — تعديل آمن ══ CREATE OR REPLACE VIEW active_employees AS SELECT emp_id, full_name, job_title, dept_id, salary, -- أضفنا salary hire_date, EXTRACT(YEAR FROM AGE(hire_date))::INT AS years_service FROM employee WHERE is_active = TRUE; -- ⚠️ لازم تحتفظ بنفس الأعمدة القديمة بنفس الترتيب + إضافة جديدة -- ══ View مع JOIN محفوظة ══ CREATE VIEW employee_full AS SELECT e.emp_id, e.full_name, e.salary, d.dept_name, m.full_name AS manager_name, ROUND(AVG(e2.salary) OVER (PARTITION BY e.dept_id), 2) AS dept_avg_salary FROM employee e JOIN department d ON e.dept_id = d.dept_id LEFT JOIN employee m ON e.supervisor_id = m.emp_id JOIN employee e2 ON e2.dept_id = e.dept_id WHERE e.is_active = TRUE; -- ══ إدارة الـ Views ══ DROP VIEW old_view; DROP VIEW IF EXISTS old_view; DROP VIEW old_view CASCADE; -- يحذف الـ views المعتمدة عليها -- عرض تعريف View موجودة SELECT pg_get_viewdef('employee_full', TRUE); -- قائمة كل الـ Views في Schema SELECT table_name FROM information_schema.views WHERE table_schema = 'public' ORDER BY table_name;
⚠️ View مش بتخزّن بيانات!

كل مرة بتعمل SELECT من View → بينفّذ الاستعلام من جديد. لو الاستعلام بطيء → الـ View بطيئة. الحل: Materialized View (لاحقاً).

كمان: الـ View بتعكس البيانات الحالية دايماً — مفيش Cache.

✏️ Updatable Views — Views قابلة للتعديل

When and How You Can INSERT/UPDATE/DELETE Through a View

في PostgreSQL، بعض الـ Views بتقدر تعمل عليها INSERT وUPDATE وDELETE مباشرة — كأنها جدول حقيقي. لكن في شروط صارمة.

📋 شروط الـ Updatable View

✅ الـ FROM بيحتوي جدول واحد بس (مش JOIN)
✅ مفيش DISTINCT، GROUP BY، HAVING، LIMIT، OFFSET
✅ مفيش Window Functions، Set Operations
✅ الأعمدة اللي في SELECT مش Expressions معقدة

لو أي شرط مش متحقق → Read-Only View

Updatable Views and WITH CHECK OPTION
-- ══ View بسيطة — Updatable تلقائياً ══ CREATE VIEW it_employees AS SELECT emp_id, full_name, salary, hire_date FROM employee WHERE dept_id = 1; -- IT Department -- بتقدر تعمل UPDATE عليها مباشرة UPDATE it_employees SET salary = salary * 1.10 WHERE emp_id = 42; -- ⚠️ مشكلة: بتقدر تضيف موظف لـ dept_id = 5 من غير ما تعرف! INSERT INTO it_employees (emp_id, full_name, dept_id) -- بنضاف لـ employee VALUES (99, 'أحمد', 5); -- dept_id=5 مش في الـ View لكن اتضاف! -- ══ WITH CHECK OPTION — يمنع إضافة بيانات خارج الـ View ══ CREATE OR REPLACE VIEW it_employees AS SELECT emp_id, full_name, salary, dept_id FROM employee WHERE dept_id = 1 WITH CHECK OPTION; -- يرفض أي INSERT/UPDATE تخالف الـ WHERE -- الآن هذا سيفشل: INSERT INTO it_employees (emp_id, full_name, dept_id) VALUES (99, 'أحمد', 5); -- ERROR: new row violates check option for view "it_employees" -- ══ INSTEAD OF Trigger — لـ Views غير Updatable ══ CREATE OR REPLACE FUNCTION fn_insert_employee_full() RETURNS TRIGGER AS $$ BEGIN INSERT INTO employee (full_name, salary, dept_id) VALUES (NEW.full_name, NEW.salary, NEW.dept_id); RETURN NEW; END; $$ LANGUAGE plpgsql; CREATE TRIGGER trig_insert_employee_full INSTEAD OF INSERT ON employee_full FOR EACH ROW EXECUTE FUNCTION fn_insert_employee_full();
💾 Materialized Views — Views بتخزّن البيانات

Materialized Views — Cached Results for Heavy Queries

الـ Materialized View بتخزّن نتيجة الاستعلام فعلاً في disk. بيرجع النتيجة بسرعة — لكن محتاج تعمله REFRESH لتحديثه.

Materialized Views — Full Reference
-- ══ CREATE MATERIALIZED VIEW ══ CREATE MATERIALIZED VIEW monthly_sales_summary AS SELECT DATE_TRUNC('month', order_date) AS month, COUNT(*) AS order_count, SUM(total_amount) AS revenue, AVG(total_amount) AS avg_order_value, COUNT(DISTINCT customer_id) AS unique_customers FROM orders WHERE status = 'delivered' GROUP BY DATE_TRUNC('month', order_date) ORDER BY month; -- ══ تحديث الـ Materialized View ══ REFRESH MATERIALIZED VIEW monthly_sales_summary; -- ⚠️ يُقفل الجدول أثناء التحديث (blocking) REFRESH MATERIALIZED VIEW CONCURRENTLY monthly_sales_summary; -- ✅ بدون قفل — يحتاج UNIQUE Index على الـ MV -- إضافة INDEX على MV للبحث السريع CREATE UNIQUE INDEX ON monthly_sales_summary (month); CREATE INDEX ON monthly_sales_summary (revenue); -- ══ Automatic Refresh بـ Cron Job أو Trigger ══ -- في PostgreSQL ما فيش automatic refresh — بتعمله manually أو بـ pg_cron -- مثال: CALL REFRESH MATERIALIZED VIEW CONCURRENTLY monthly_sales_summary; -- بتشغّله كل يوم في midnight -- ══ DROP MATERIALIZED VIEW ══ DROP MATERIALIZED VIEW monthly_sales_summary;

📝 CTE — Common Table Expressions

WITH Clause — Named Subqueries for Clarity and Reuse

الـ CTE هو طريقة لتسمية Subquery وإعادة استخدامها. بيحوّل استعلام معقد لخطوات واضحة. مش بيحسّن الأداء دايماً — لكن بيحسّن القراءة جداً.

CTE — From Basic to Multiple CTEs
-- ══ CTE بسيطة ══ WITH high_earners AS ( SELECT emp_id, full_name, salary, dept_id FROM employee WHERE salary > 20000 ) SELECT h.*, d.dept_name FROM high_earners h JOIN department d USING (dept_id) ORDER BY salary DESC; -- ══ CTEs متعددة — Pipeline Queries ══ WITH -- CTE 1: إجمالي مشتريات كل عميل customer_spending AS ( SELECT customer_id, COUNT(*) AS order_count, SUM(total_amount) AS total_spent FROM orders WHERE status = 'delivered' GROUP BY customer_id ), -- CTE 2: تصنيف العملاء حسب الإنفاق customer_segments AS ( SELECT cs.*, CASE WHEN total_spent > 100000 THEN 'platinum' WHEN total_spent > 50000 THEN 'gold' WHEN total_spent > 10000 THEN 'silver' ELSE 'bronze' END AS segment FROM customer_spending cs ), -- CTE 3: دمج مع بيانات العميل enriched AS ( SELECT c.full_name, c.email, seg.order_count, seg.total_spent, seg.segment FROM customer c JOIN customer_segments seg USING (customer_id) ) -- الاستعلام الأخير على CTEs: SELECT segment, COUNT(*) AS count, AVG(total_spent) AS avg_spent FROM enriched GROUP BY segment ORDER BY avg_spent DESC; -- ══ CTE في UPDATE ══ WITH vip_candidates AS ( SELECT customer_id FROM orders GROUP BY customer_id HAVING COUNT(*) > 10 AND SUM(total_amount) > 50000 ) UPDATE customer SET is_vip = TRUE WHERE customer_id IN (SELECT customer_id FROM vip_candidates);
🏢 CTE vs Subquery — متى تختار كل واحد؟

CTE أفضل عندما:
✅ عايز تستخدم نفس الـ Subquery أكتر من مرة
✅ الاستعلام معقد ومحتاج تقسيمه لخطوات
✅ Recursive queries (BST لازم CTE)

Subquery أفضل عندما:
→ استخدام بسيط لمرة واحدة
→ بعض query planners بيُحسّنوا Subqueries أحسن من CTEs

في PostgreSQL قبل v12: الـ CTE كانت "optimization fence" — مش هيتحسّنها مع الـ outer query.
في v12+: تم إضافة MATERIALIZED و NOT MATERIALIZED keywords للتحكم في السلوك.

🔄 Recursive CTE — الاستعلامات الشجرية

Recursive CTEs — Traversing Hierarchical Data

الـ Recursive CTE بتشتغل على البيانات الهرمية — شجرة الموظفين والمشرفين، تصنيفات متداخلة، قوائم أجداد. الاستعلام بيرجع ويستعلم من نفسه.

①

Anchor Member: الاستعلام الابتدائي — يُعطي نقطة البداية (مثل: المدير التنفيذي)

②

UNION ALL: يربط الاستعلام الابتدائي بالاستعلام التكراري

③

Recursive Member: الاستعلام اللي بيـ JOIN نفس الـ CTE — بيوسّع الشجرة مستوى بمستوى

④

Termination: التكرار بيوقف لما الـ Recursive Member لا يرجع صفوف جديدة

Recursive CTE — Employee Hierarchy
-- ══ شجرة الموظفين والمشرفين ══ -- EMPLOYEE(emp_id, full_name, supervisor_id) WITH RECURSIVE org_chart AS ( -- Anchor: المدير التنفيذي (مفيش مشرف) SELECT emp_id, full_name, supervisor_id, 0 AS depth, full_name AS path FROM employee WHERE supervisor_id IS NULL UNION ALL -- Recursive: أضف الموظفين المباشرين SELECT e.emp_id, e.full_name, e.supervisor_id, oc.depth + 1, oc.path || ' → ' || e.full_name FROM employee e JOIN org_chart oc ON e.supervisor_id = oc.emp_id ) SELECT REPEAT(' ', depth) || full_name AS hierarchy, depth, path FROM org_chart ORDER BY path; -- ══ Recursive CTE: تصنيفات متداخلة ══ WITH RECURSIVE category_tree AS ( SELECT category_id, name, parent_id, 1 AS level FROM category WHERE parent_id IS NULL -- الجذور UNION ALL SELECT c.category_id, c.name, c.parent_id, ct.level + 1 FROM category c JOIN category_tree ct ON c.parent_id = ct.category_id ) SELECT REPEAT('-- ', level-1) || name AS tree_view, level FROM category_tree ORDER BY name; -- ⚠️ تحذير: Infinite Loop إذا في دورة في البيانات -- الحل: إضافة LIMIT أو شرط depth WHERE depth < 10 -- في الـ Recursive Member → يمنع اللوب اللانهائي
✍️ تمرين عملي

🏪 تمرين: نظام متجر

الجداول

category(category_id, name, parent_id), product(product_id, name, price, category_id, is_active), orders(order_id, customer_id, total_amount, created_at)

  • Aأنشئ View اسمها active_products_with_category تجيب اسم المنتج والسعر واسم التصنيف — للمنتجات النشطة فقط.
  • Bأنشئ Materialized View للمبيعات الشهرية، وأضف لها UNIQUE INDEX على الشهر.
  • Cباستخدام Recursive CTE، اعرض شجرة التصنيفات بالمسافات البادئة.
Solution
-- A: View CREATE VIEW active_products_with_category AS SELECT p.product_id, p.name, p.price, c.name AS category_name FROM product p JOIN category c USING (category_id) WHERE p.is_active = TRUE; -- B: Materialized View CREATE MATERIALIZED VIEW mv_monthly_sales AS SELECT DATE_TRUNC('month', created_at) AS month, COUNT(*) AS orders, SUM(total_amount) AS revenue FROM orders GROUP BY DATE_TRUNC('month', created_at); CREATE UNIQUE INDEX idx_mv_month ON mv_monthly_sales (month); -- C: Recursive CTE WITH RECURSIVE cat_tree AS ( SELECT category_id, name, parent_id, 0 AS lvl FROM category WHERE parent_id IS NULL UNION ALL SELECT c.category_id, c.name, c.parent_id, ct.lvl+1 FROM category c JOIN cat_tree ct ON c.parent_id = ct.category_id ) SELECT REPEAT(' ', lvl) || name AS tree FROM cat_tree ORDER BY name;
🧠 اختبار الفهم
١. الفرق الجوهري بين Regular View وMaterialized View؟
الـ Materialized View أسرع دايماً بدون أي عيوب
Regular View لا تخزّن بيانات (دايماً محدّثة لكن أبطأ). Materialized View تخزّن (أسرع لكن محتاج REFRESH للتحديث).
Materialized View أقدر أكتب عليها INSERT/UPDATE
٢. ليه نستخدم WITH CHECK OPTION في الـ View؟
لتسريع الـ View
لمنع إدراج أو تعديل بيانات تخالف شرط الـ WHERE في الـ View — يضمن Consistency
لجعل الـ View Read-Only
٣. في الـ Recursive CTE، ما الـ Anchor Member؟
الاستعلام اللي بيرجع للـ CTE نفسها
الاستعلام الابتدائي (غير التكراري) اللي بيحدد نقطة البداية — مثل جذر الشجرة
الاستعلام النهائي بعد الـ WITH
٤. متى تُفضَّل CTE على Subquery؟
دايماً — الـ CTE أسرع
لما تحتاج نفس الـ Subquery أكتر من مرة، أو الاستعلام معقد ومحتاج تقسيمه لخطوات واضحة
الـ CTE مش بتشتغل مع UPDATE
